# /daily-plan — morning plan for the day

Write a durable dated **plan for today** so the day starts from a written intent rather than manually. The morning half of the V-263 scheduled-ritual pair (`/daily-summary` is the evening half). Reads yesterday's summary (the evening artifact) plus the local pipeline KB — no Linear MCP, so the unattended run needs no auth.

**Self-contained by design.** A scheduled run has no session context — every path below is absolute or repo-anchored; never reference "this conversation".

**Sequenced after the harvests + yesterday's summary.** The launchd schedule (09:27) fires after the 09:07 bug + 09:17 feedback harvests, so their output is already in place when the plan reads the KB.

## 0. Flags + setup
- `--yes` — non-interactive (the launchd runner passes this); no confirm gates here.
- `--dry-run` — print the assembled plan to stdout instead of writing the file.
- `ROOT=~/.claude`. `TODAY=$(date +%F)`; `YESTERDAY=$(date -v-1d +%F)` (BSD/macOS date).
- Artifact path: `$ROOT/pipeline/daily/$TODAY-plan.md`. Ensure the dir: `mkdir -p "$ROOT/pipeline/daily"`.

## 1. Gather context (all local, fail-open)
- **Yesterday's summary** — read `$ROOT/pipeline/daily/$YESTERDAY-summary.md` if present. Its `## Notes for tomorrow's plan` section is the primary seed (carryover, a forced-halt to retry, a feedback theme). Absent (first run, or a skipped day) → note "no prior summary" and lean on the KB alone.
- **Roadmap / open work** — read `$ROOT/pipeline/roadmap.md` (near-term priorities) and, if present, `$ROOT/pipeline/owed.md` (deferred-but-owed obligations) — surface anything due/near-term. Do NOT dump the whole file; pull the currently-active items only (targeted read/grep).
- **Parked ideas** — glance at `$ROOT/pipeline/parked.md` for anything that has become timely; usually skip.
- **In-flight** — optionally `find "$ROOT/docs/plans" -maxdepth 1 -name '*-build.md' -mtime -3 2>/dev/null` for build plans touched in the last few days as a hint at what's mid-flight; keep it light. (`find`, not a bare glob — shell-independent, fail-open.)

## 2. Assemble + write the artifact
Compose a short, honest, prioritized plan (not a wish-list — 3–6 concrete intents for today):

```markdown
# Daily plan — <TODAY>
> Generated by /daily-plan (morning ritual, V-263). From yesterday's summary + the pipeline KB.

## Carryover from yesterday
- <from $YESTERDAY-summary.md's "Notes for tomorrow" + any forced-halt to retry — or "No prior summary.">

## Today's focus (prioritized)
1. <the single most important thing to move today, with why>
2. <next>
3. <…>  (3–6 items; concrete and checkable, not vague)

## Owed / near-term (from roadmap + owed)
- <any due obligation or near-term priority worth naming today — or "nothing due">

## Watch
- <a risk, a blocked item, or a feedback theme to keep in mind — omit if none>
```

- `--dry-run` → print to stdout, do not write. Emit `result: /daily-plan dry-run — plan assembled from <yesterday-summary? / KB>; artifact NOT written.`
- Else → `Write` it to `$ROOT/pipeline/daily/$TODAY-plan.md` (overwrite on a same-day re-run). Emit `result: /daily-plan — wrote pipeline/daily/$TODAY-plan.md (<N> focus items).`

## Hard rules
- **Read-only on the pipeline** except the one artifact under `pipeline/daily/`. No Linear mutation, no commit/push (artifact is gitignored — durable on disk).
- **Fail-open, never crash** — a missing summary or KB file degrades to a note; the ritual always produces (or dry-runs) a plan.
- **No fabrication (convention 8)** — the plan is grounded in what the summary + KB actually say; don't invent priorities or claim a state you didn't read. Judgment about *what to prioritize* is fine and expected; asserting a *fact* that isn't in the sources is not.
- One run = one day's plan. Idempotent (a same-day re-run overwrites).
