# /go — chained build autopilot with instrumented gates

Run the execution pipeline end-to-end on one ticket without re-typing between phases:

```
/go <ticket-id>              ≡  /next-ticket <id> → [/research] → [/scope] → /build (owns the design thesis-check, §3.5) → /land-ticket
/go --architect <ticket-id>  ≡  … → /scope (forced on) → [architect stage: research → options → Q&A → human-directed design] → /build → /land-ticket
```

`--architect` (§0) inserts one **human-owned architecture stage** before `/build` — the full-interactive intensity of the pre-code architecture checkpoint `/build §3.5` renders at checkpoint intensity on a `deep` ticket (V-325). It reuses `/plan`'s research + option-presentation machinery (§5), and is human-present-only.

`/go` is an **orchestrator**, same shape as `commands/audit-cycle.md`: it runs each phase by reading `~/.claude/commands/<cmd>.md` and **executing it literally**, and it **honors every gate the underlying skill raises** — it removes the manual re-typing between phases, never the gates. On top of that it **instruments** every gate: each one is classified, its resolution recorded, and the run is appended to a persistent friction map so we learn over many runs which gates are ceremony and which are load-bearing.

Read `~/.claude/workflow-conventions.md` first **with the `Read` tool — never `cat`/Bash (V-299): `cat` overflows the ~40 KB file to a persisted result that is *not* shown inline, forcing a second `grep`/`sed` pass to see the conventions you meant to read; the `Read` tool puts them in context in one pass**, plus the **active model profile** — read it from the absolute `~/.claude/pipeline/profiles/<family>.md` (convention 12), **never** `$(git rev-parse --show-toplevel)/pipeline/profiles/…`: the profiles live in the V repo regardless of which repo `/go` is driving (exactly like the `errors.jsonl` logger path), so a `/go` run driving a non-V repo must not resolve the profile under its own repo root. The profile's `autonomy` posture governs how `/go` carries itself between gates, and its `design-check` knob determines whether the §3 design-stage verdict arrives subagent-checked or `(self-checked)`. `/go` owns no gates of its own — the sub-commands own theirs; `/go` classifies, drives, and logs them.

**Read these two once, here, and carry them forward for the whole invocation (V-293).** This single `Read` loads the whole conventions substrate that the chain collectively leans on (convs. 4/5 for `/next-ticket`, 1/2/3/8 for `/scope`, 8/9 for `/build`, 11 for `/land` — hence read once, whole); **do not then `grep`/`sed`/`cat` individual conventions (11/12, …) back out of it** — they are already in context from this read, so a second extraction pass to "read conventions 11/12" is pure duplicated cost (V-299, the exact waste that ticket reports). When you execute each chained sub-command (§5) by reading its file and following it literally, **do not re-read `workflow-conventions.md` or the active profile** — they are already in context from this top-of-run read (the `read-discipline` knob, `opus-4-8.md`, applied to the chain's own preambles). Each **inline** sub-command's read-first preamble carries an explicit "under `/go`, skip the re-read" clause for exactly this — `/next-ticket`, `/scope`, `/build`, `/fix-bug`, `/land-ticket`. The skip is scoped to **inline, same-context** phases only: a **fresh-context subagent** the chain dispatches — `/thesis-check` (the `design-check` subagent), and any subagent-run `/verify-tests` / `/research` — has its own empty context, so it genuinely reads conventions/profile itself; never suppress its read. A standalone, direct invocation of any sub-command also reads them per its own preamble (that read stays intact).

## 0. Args + setup

- `$ARGUMENTS` = `[--architect] <ticket-id>` (e.g. `ENG-54` or `--architect ENG-54`). The ticket id is required — `/go` always drives a specific ticket, so it never hits `/next-ticket`'s auto-pick preview gate. No ticket id → STOP, ask for one.
- **`--architect`** (optional) — run the full-interactive **architect stage** (§5) between `/scope` and `/build`: research → present options → human Q&A + direction → record the human-directed architecture decision, *before* any code is written. It **forces `/scope` and the architect stage on regardless of the ticket's depth-class** (raise-only — it never lowers a gate). It is **human-present-only** — the stage's option-forks block on `AskUserQuestion`, so an unattended/background `/go --architect` would stall (like `/plan`, which requires a live user). Absent → `/go` runs its normal chain with no architect stage.
- `root="$(git rev-parse --show-toplevel)"`; read `$root/.claude/ticket-flow.json` (missing → `/ticket-flow-init`, STOP).
- Initialize an empty **run ledger** (§3) in working memory. It accumulates one entry per gate encountered across all phases and is flushed to the audit file (§4) at run end — clean completion **or** forced halt.

## 1. Gate taxonomy (classify before you drive)

Every pause the pipeline can produce is exactly one of three kinds. `/go` pre-knows the inventory below (derived from the four commands) and classifies each pause as it fires.

**Hard stop** — something is genuinely off; **halt and surface, the human resolves.** `/go` does **not** auto-resolve and does **not** `p` past it. Crucially, **overridable stops are treated as hard**: scope's `needs-eyes` and build's `--force` gate *can* be forced with a flag, but `/go` **never injects `--force`** — they fire because validation found a real problem (design call #1). A hard stop **ends the `/go` run** (logged as `forced`); the human fixes the underlying cause and re-runs `/go` (or the next command directly).

**Confirm gate** — a deliberate human checkpoint on an irreversible/again-significant action. `/go` pauses and awaits one token (§2): `p` proceeds with the command's proposed default; anything else is obeyed as a literal instruction.

**Ack gate (`[HARD STOP]`, convention 11's third kind)** — a pause a bare `p` must **not** pass: the human must actually engage (answer a question, waive a named finding per-item, review a research output, or review a formed architecture decision). The §4.6 security-HIGH waiver and the manual-test residue are the existing instances; research-ticket review points (Initiative II) render this way too, and the **deep-ticket pre-code architecture review** (`/build` §3.5, V-325 — a `depth-class = deep` ticket's `sound`-verdict checkpoint before any code is edited) is one too. At an ack gate `/go` re-states the gate on a `p` and waits; only an explicit instruction clears it. Logged `ack'd` (engaged and cleared) or `intervened` (redirected).

### Inventory (per phase)

- **`/next-ticket <id>`** — *hard stops*: In-Progress guard hit, dirty worktree, leftover branch with the ticket's name, binding mismatch / standalone-redirect / no-target. *Confirm gate*: the §1.C **orphan-recovery offer** — when `bin/orphan-detect.sh` returns `orphan-recoverable` (a crashed-mid-setup worktree: binding-less, clean, on a non-base ticket branch), `/next-ticket` *offers* recovery instead of dead-stopping. `/go` drives it per §2: `p` = the reconstruct-binding default. This is deliberately **not** a hard stop; only the genuinely-ambiguous `dirty-stop`/`foreign-stop` binding-mismatch keeps hard-stop status. (Its auto-pick preview gate is also a confirm gate, but it fires only on auto-pick — `/go` always passes an explicit id, so it never fires here.)
- **`[/research]` (conditional)** — *no gates.* Read-only prior-art recon (`commands/research.md`); fires only for a net-new solution, skipped for routine pattern-following tickets. No hard stops, no confirm gates — it degrades gracefully to plain web search if `/deep-research` is unavailable, so there is nothing for `/go` to intercept. Logged in the ledger as a phase outcome (`run` / `skipped`), not a gate.
- **`/scope`** — *hard stops*: preflight (not In Progress, branch drift, dirty tree, pre-existing plan), §3 `needs-eyes` (**overridable → treated as hard**), §4 snippet compile-fail (**overridable → treated as hard**). No confirm gates on the default path.
- **design thesis-check (owned by `/build` §3.5, both paths)** — the rung (`docs/implementation-design-rung.md`, `commands/thesis-check.md`) is triggered by `/build`, not `/go`: `/build` §3.5 fires `/thesis-check` against the plan's `## Implementation design` whenever the plan lacks a `## Thesis-check` block, on **both** the scope-gate and skip-gate paths; `/go` never invokes it itself. So this is **not a separate `/go` phase**: a **non-`sound`** verdict surfaces as a `/build` **hard stop** (caught by the `/build` inventory bullet below — `/go` logs it `forced`, never `p`s past it, never injects the build-despite ack, that ack being the human's; amending or re-scoping the design is drivable in-session by the agent or the human), and a **`sound`** verdict is a recorded *design-stage outcome* (like `/research`'s `run`/`skipped`) that `/go` reads from the plan's `## Thesis-check` block after `/build` and logs to the gate-audit (§3/§4). **Additionally, on a `deep` ticket (V-325), a `sound` verdict surfaces `/build` §3.5's convention-11 `[HARD STOP]` pre-code architecture ack-gate** — the formed architecture decision presented for review before §4 edits. This is an **ack gate**, not a new phase: it reuses the same thesis-check surface (one mechanism, no second stop), fires only on `deep` (`light`/`standard` render nothing), and `/go` drives it **inline** per §2's ack-gate protocol (re-state on a bare `p`, clear only on explicit engagement), logging `ack'd`/`intervened`. A deep+sound run therefore records **two** ledger lines: the design-stage outcome (the `sound` verdict, no tally) and this ack-gate (`ack'd`/`intervened`, counted in the intervened tally).
- **`/build`** — *hard stops*: preflight failures, §4 structural-impossibility `needs input`, §5 verify retry-exhaustion / pre-existing-failure / infra, §6 push rejection. No confirm gates (autonomous by default). *Ack gate (deep tickets only):* on a `depth-class = deep` ticket, §3.5 renders the convention-11 `[HARD STOP]` **pre-code architecture ack-gate** on a `sound` verdict (V-325) — see the design thesis-check bullet above; `/go` drives it inline per §2's ack-gate protocol, logged `ack'd`/`intervened`.
- **architect stage (`--architect` only — the interactive intensity, §5)** — three distinct interaction surfaces, classified per convention 11's discrete-fork exception: (1) the **option-forks** (present 2–3 candidate approaches + tradeoffs) use **`AskUserQuestion`** — permitted because `--architect` guarantees the human is present, and picking among options is a discrete fork, not an approval of an irreversible action (`workflow-conventions.md §11`). These are **not `/go` gates** — no `p`, no ledger gate line (their outcome folds into the human-directed design the stage records). Open-ended Q&A/direction is freeform message turns, likewise not a gate. (2) The stage's **terminal proceed-to-build** is a convention-11 `needs input:` **confirm gate** on a `light`/`standard` ticket (where `/build §3.5` renders no gate) — driven per §2's one-token protocol, logged `p'd`/`intervened`. (3) On a **`deep`** ticket the terminal ack is **not** a second stop — it **defers to `/build §3.5`'s existing `[HARD STOP]` pre-code architecture ack-gate** (V-325 — one mechanism), which `/go` already drives inline per the `/build` bullet above (logged `ack'd`/`intervened`). So the architect stage adds **at most one** `/go`-visible gate: a light/standard terminal confirm, or (deep) nothing beyond the §3.5 ack-gate already inventoried.
- **`/fix-bug` (route `fix-bug` only — replaces `/scope`+`/build`)** — the routed repro-first path. *Confirm gate*: §2's **can't-reproduce / stale-but-real → close-or-re-scope** decision (cancelling/re-scoping a ticket is irreversible and rests on a judgment — driven per §2's one-token protocol, `p` = the proposed close/re-scope, anything else an instruction). It carries **no hard stops of its own**: §3 keeps `/build §4.6` external-mutation read-back + convention-8 no-fabrication (a *claim*-blocker, not a pause), and the path ends in the **full unchanged `/land-ticket`**, so a `/fix-bug` run also meets every `/land` gate below (§5, §6.7 confirms; §4.8/§4.6/permission hard stops). It runs no scope plan and no §3.5 thesis-check, so **no** design-stage thesis-check line is recorded for a `fix-bug` run.
- **research route (route `research` only — replaces `/scope`+thesis-check+`/build`, inline in `/go`)** — the knowledge-deliverable path: freeform inline investigation → land a **finding doc** in `docs/` → the **full unchanged `/land-ticket`** (`docs/research-route-path-design.md`). It is **inline** (no peer command, unlike `/fix-bug`) and carries **no gates of its own** — no scope plan, no §3.5 thesis-check (so **no** design-stage thesis-check line is recorded for a `research` run), no `/build`, no `/verify-tests`; the freeform investigation and the finding-doc write raise nothing for `/go` to intercept. `/build §4.6` read-back + convention-8 no-fabrication still bind *if* the investigation mutates live state (rare for a read-only finding) — a claim-blocker, not a pause. The path ends in `/land-ticket`, so it meets every `/land` gate below (§5, §6.7 confirms; §4.8/§4.6/permission hard stops) — and `/land`'s diff-driven gates are route-invariant on the doc-only diff (§4.8 keys on `migration`-kind, §6.8 on `cfg.docs` ripple — neither fires on a `code`-kind finding doc).
- **`/land-ticket`** — *confirm gates*: §5 "approve proceeding" gate, §6.7 manual-test pause (the actual merge approval), and the **live-system-mutation `ask`**: the `guard-sensitive-access.py` hook raises a confirm on any `crontab`/`launchctl`/`systemctl` write or `settings.json` permissions edit performed during the flow — drive it per §2, never auto-dismiss. *Hard stops*: §4.8 `migration`-with-no-migration, any infra failure, **any harness permission prompt** (e.g. `sb-push --apply`, deny-by-default — `/go` lets the human resolve it, never auto-dismisses), and the **permission-bypass-flag DENY**: the same hook hard-blocks `--dangerously-skip-permissions` / `--permission-mode bypassPermissions` anywhere in a command — `/go` never injects a bypass flag and never works around the block (log `forced`, surface to the human). **Two things are excluded from the §6.7 `p`:** an unresolved §4.6 security-review **HIGH** and a non-empty §4.9 **HUMAN TESTING REQUIRED** residue — `p` advances only the residue-free, non-security merge default; full rules in §2. (§6.7 also auto-files two **non-blocking** post-merge spin-outs into the deferred-checks bucket — POST-DEPLOY MONITORING → `verify-the-fix:` (V-257) and DEFERRED VERIFICATION → `check-later:` (V-319); `p` passes both and they add **no** gate — recorded post-merge outcomes, not gates.)

**Route-aware gate layout (V-233, `land-ticket.md` §4.10):** §5/§6.7 render **droid-details-on-top, decision-payload-at-the-bottom** on every route — the mechanical receipts up top (skimmable), and a **Built / How & why / To check** brief just above the `needs input:` stop (`To check` = the §4.9 human-only residue alone). The route sets the bottom's depth + whether §5 collapses: **`build`/`fix-bug`** → standard two gates in that layout; **`research`** → the bottom *leads with* the finding's decision substance, read from the finding doc's structured Decision block (`## Question` · `## Candidate approaches` · `## Relevant standards` · `## Recommendation` w/ pros-cons — the block the §5 research path is required to write, V-301), still two gates, same one-token `p`; **`fast`** with no migration/findings/empty §4.9 residue → §5+§6.7 **collapse into one** merge confirm, so `/go` meets a **single** confirm gate here and logs **one** gate-audit line for it (acceptance-faithful: the gate is still recorded — the shape changed, not the instrumentation). The collapse is **diff-driven**: a `fast` land that unexpectedly carries a migration/finding keeps the two-gate `standard` shape (route-invariant, like §4.8). `/go` reads the route it already captured in §5 step 1 and drives whichever shape `land-ticket` renders — it never re-shapes the gate itself (`land-ticket` owns its gate rendering, convention 11).

Net: on the happy path the confirm gates `/go` meets are **`/land-ticket` §5 and §6.7** — **collapsed to a single §6.7 merge confirm on a `fast` land with no migration/findings/empty residue** (V-233, `land-ticket.md` §4.10), and on a `research` land both gates carry the finding's decision substance foregrounded (still two gates, same `p`); everything else is a hard stop. The design + thesis-check rung adds **no confirm gate** — a `sound` verdict is a recorded design-stage outcome and a non-`sound` one is a hard stop — but on a **`deep`** ticket it adds one **ack gate**: `/build` §3.5's pre-code architecture review on a `sound` verdict (V-325), driven per §2's ack-gate protocol and logged `ack'd`/`intervened` (`light`/`standard` add nothing). So the happy-path *confirm*-gate set is unchanged in kind (route only reshapes/collapses `/land-ticket`'s two gates, never adds one); the only rung-added gate is the deep-ticket architecture ack gate.

If a future gate appears that isn't in this inventory: classify it by its nature — reversible deliberate checkpoint → confirm; "something is wrong" / irreversible-without-eyes → hard — log it, and it gets captured in the audit map like any other.

## 2. Confirm protocol (the one-token pause)

This is convention 11's **orchestration affordance** over a sub-skill's `needs input:` confirm gate — not a separate gate mechanism. The sub-skill's `needs input:` prose stays the source of truth; `/go` reads it and offers the human one token. (So a sub-skill must render its gate as a `needs input:` stop, never an `AskUserQuestion` modal, or `/go` can't drive it.)

At a **confirm gate**, `/go`:
1. **Leads with a one-line restatement of the ticket's goal** (`<TICKET-ID>: <goal>`, per convention 11 — the human often launched `/go` without reading the ticket, so the gate must say what it is about before asking for a yes), then prints the gate: which command + section, what it's authorizing, and the command's **proposed default** (e.g. land-ticket §5: the DB-change summary + PR/order/strategy; §6.7: the manual-test verdict + "merge?").
2. **STOPs the turn and waits.** The harness has no literal single-keypress capture — the "keystroke" is simply the user's **next message**.
3. Reads the reply:
   - Exactly **`p`** (case-insensitive, trimmed) → proceed with the proposed default. Log resolution `p'd`.
   - **Anything else** → treat the message as a literal instruction from the human and obey it (they may redirect, amend, deny, or add a constraint). Log resolution `intervened`. After acting on the instruction, continue the pipeline from where it makes sense.

**A `/land-ticket` §4.6 security-review HIGH waiver is never carried by `p`.** `p` proceeds only with the gate's *stated non-security default*. An unresolved §4.6 security **HIGH** finding is a merge hard-block at the §6.7 pause, cleared **only** by a fix or the human's **explicit, finding-specific waive** — the distinct **`waive HIGH: <finding>`** token defined at `land-ticket.md` §6.7; a §5-style proceed never clears it (`land-ticket.md` §6.7 + §5). So `/go` must never *offer* to fold that waiver into the proceed token (e.g. "I'll take `p` as also waiving the marker-word HIGH unless you say otherwise"), never default it, and never pre-bundle it: the proceed token and the security-gate waiver stay strictly distinct. At a §6.7 pause with a HIGH still standing, `p` does **not** authorize the merge — surface the HIGH as its own decision and require a separate, non-`p` human instruction — the `waive HIGH: <finding>` token naming the finding — to waive it (logged `intervened`).

**A genuinely-required manual-test residue is likewise never skipped by `p`.** `land-ticket.md` §4.9 partitions manual-test needs by who-can-do-it: the session runs every item observable from the diff/tools/repo itself and reports the result, and §6.7 hands the human only the residue it cannot observe (taste/wording, forward-looking outcomes, real-world/UI/runtime). When that HUMAN TESTING REQUIRED residue is non-empty, `p` proceeds only with the gate's *residue-free merge default* — it does **not** authorize skipping the residual testing, and `/go` must never *suggest* or accept `p` as a way past it. The residue is cleared only by the human **reporting each item's observed result** or an **explicit, per-item waive** (logged `intervened`), exactly as the §4.6 HIGH is — the proceed token and the manual-test residue stay strictly distinct, the testing being the human's act. (When the residue is empty — `No human testing required.` — `p` is the normal one-token merge approval.)

**A genuinely-owed acceptance-item verification that *cannot* run at land is not this residue — it is auto-filed, and `p` correctly passes it (V-319).** `land-ticket.md` §4.9 partitions such an item (an end-to-end runtime read-back needing a throwaway resource — V-309's case) into the **DEFERRED VERIFICATION** block, a sibling of POST-DEPLOY MONITORING: it is **not** check-now (a human can't run it at the pause), so it never blocks and adds **no** `/go` confirm gate — `p` passes it exactly as it passes a POST-DEPLOY MONITORING block. §6.7 then **auto-files it as a `check-later:` follow-up ticket** into the deferred-checks bucket after the merge, so the owed verification is *tracked as a ticket*, never carried on `p` into nothing and never lost to a Done-comment prose note. `/go` drives this identically to the monitoring auto-file: it is a recorded post-merge outcome (surfaced in the land summary as `deferred verification: <ID>`), not a gate. A *merely-nice-to-have* deferred check is dropped as today — only a genuinely-owed one is minted.

**Visual/eyeball residue — localhost by default + a distinct discharge token (V-318).** When the residue is *visual/UI-eyeball*, `land-ticket.md` §4.9 Step 3b runs the session's half **before** the pause: it starts the repo's localhost dev server **by default** (no separate human request) and drives a browser tool to assert/screenshot each visual item — every item it can positively assert is **promoted to SESSION-VERIFIED** with that observed evidence and is **no longer residue** (so it needs **no** gate line — it was discharged by proof, not a keystroke). Whatever genuinely stays human-eyeball is presented at §6.7 with the **already-running localhost URL** and a **distinct per-item token** — `checked <item>: <result>` (report) or `waive check <item>` (per-item waive) — that is **not** `p`: a bare `p` advances only the residue-free merge default and re-surfaces any standing visual item, exactly as it never carries the §4.6 HIGH. `/go` drives these per-item tokens as ack-gate engagement (logged `ack'd`/`intervened`), and must never *offer* `p` as a way to self-certify an eyeball item (the block-vs-self-certify overload V-318 retires). On an app-less repo (this one) Step 3b no-ops with a note and no localhost starts — the token mechanism is unchanged.

`p` never applies to a hard stop — there is nothing to rubber-stamp; a hard stop is logged `forced` and ends the run (§1).

## 3. Run ledger + end-of-run buckets

Maintain an ordered ledger; one entry per gate encountered:

```
{ phase, gate, classification: hard-stop|confirm, resolution: p'd|intervened|forced, note }
```

**A `collapsed` `/land-ticket` land (V-233 — `land-ticket.md` §4.10: `fast` + every gate input empty) contributes a single gate entry**, not two — `land-ticket §6.7 · merge confirm (collapsed §5+§6.7) · confirm · <resolution>` — because the §5 + §6.7 gates merged into the one merge confirm the human actually saw. The ledger records one line for the one gate (acceptance-faithful: the gate is still recorded, just once). A `research` land still logs two entries (§5 + §6.7), each noting the decision-substance foreground; a `build` land is unchanged.

Bucket definitions (used in both the on-screen summary and the audit file):
- **p'd** — a confirm gate the human proceeded through with `p` (rubber-stamped). Perennially-p'd gates are prune candidates.
- **intervened** — a confirm gate where the human gave a non-`p` instruction (redirected/amended/denied). An intervened gate earned its keep.
- **ack'd** — an ack gate (`[HARD STOP]`, convention 11) the human cleared with the explicit engagement it demanded (a per-item waive, an answer, a named-token ack) — never with a bare `p`.
- **forced** — a hard stop that fired and halted the run (the human had to resolve something genuinely off).

For the §4 flush, an `ack'd` entry counts in the **intervened** tally (both mean the human engaged; `bin/log-gate-audit.mjs`'s tally flags are unchanged) — the per-gate line carries `ack'd` verbatim so the friction map still distinguishes them.

**Design-stage outcome (the B3 rung; trigger owned by `/build` §3.5).** `/go` records one ledger entry per run for the design thesis-check, sourced from the plan's `## Thesis-check` block that `/build` §3.5 writes — not from a `/go`-driven invocation. A **`sound`** verdict is a *recorded outcome* — it adds a line but lands in **no** tally bucket; a **non-`sound`** verdict surfaces as a `/build` hard stop logged `forced` (counted). Represent it as a ledger entry with `phase: design-thesis-check`, `gate: /thesis-check verdict <verdict>`, `classification: hard-stop|outcome`, `resolution: forced|sound`. A `sound (self-checked)` verdict (the profile's `design-check` self-check path, convention 12) counts as `sound` — log the annotation verbatim in the gate line so the friction map shows which route ran.

**Deep-ticket architecture ack-gate (V-325).** On a `deep` ticket, a `sound` verdict additionally produces `/build` §3.5's `[HARD STOP]` pre-code architecture ack-gate — a **separate** ledger entry from the design-stage outcome above: `phase: build-arch-ack`, `gate: /build §3.5 pre-code architecture ack-gate`, `classification: ack`, `resolution: ack'd|intervened`. Per the ack-gate rule above it counts in the **intervened** tally (both mean the human engaged), with the per-gate line carrying `ack'd`/`intervened` verbatim so the friction map still distinguishes them. A deep+sound run therefore logs two lines — the design-stage `sound` outcome (no tally) and this ack-gate (`ack'd`/`intervened`, counted). `light`/`standard` tickets log only the design-stage outcome line (no ack-gate).

**Architect stage (`--architect`, §5).** The stage's **option-forks are not ledgered as gates** — an `AskUserQuestion` discrete fork is not a `/go` confirm/ack gate (§1), so it produces no ledger line; its outcome is captured in the `## Architect session` record, not the friction map. What the stage *does* log depends on depth-class: a **`light`/`standard`** ticket's terminal proceed-to-build is a **confirm gate** entry — `phase: architect-stage`, `gate: architect terminal proceed-to-build`, `classification: confirm`, `resolution: p'd|intervened` (counted normally); a **`deep`** ticket adds **no** architect-stage line — its terminal ack **is** the `build-arch-ack` line above (one mechanism, so it is not double-counted). The design-stage `## Thesis-check` outcome line is recorded as usual (the architect stage changes *what* design is checked, not the instrumentation of the check).

## 4. Audit persistence (cross-run friction map)

At run end (clean completion **or** forced halt), append this run to `pipeline/audit/gate-audit.md` as a dated block in this exact format so cross-run greps work:

```markdown
## <YYYY-MM-DD HH:MM> — <TICKET-ID> (<outcome: completed | forced-halt at <phase>>)
Tallies: p'd <n> · intervened <n> · forced <n>
- <phase> · <gate> · <classification> · <resolution> — <one-line note>
- ...
```

The design thesis-check contributes one such line per run (B3; trigger owned by `/build` §3.5), so design-stage outcomes are recorded alongside the gates: `design-thesis-check · /thesis-check verdict <verdict> · <hard-stop|outcome> · <forced|sound> — <note>`. A `sound` line is `outcome` classification and adds 0 to the tallies; a non-`sound` line is `hard-stop`/`forced` and increments `--forced`. The verdict is read from the plan's `## Thesis-check` block (written by `/build` §3.5), then passed to the helper as another `--gate` (the helper accepts arbitrary verbatim lines — no helper change).

The **deep-ticket architecture ack-gate** (V-325) contributes its own line when it fires: `build-arch-ack · /build §3.5 pre-code architecture ack-gate · ack · <ack'd|intervened> — <note>`, passed as another `--gate` and counted in `--intervened` (the `ack'd`/`intervened` tally rule, §3). It appears only on a `deep`+`sound` run — never on `light`/`standard`, and never on a non-`sound` verdict (which halted earlier as a `forced` hard stop).

**Append via the sanctioned helper — never a heredoc, never a bare `mkdir`.** Use `bin/log-gate-audit.mjs`, a single allowlisted call (covered by the blanket `Bash(node ~/.claude/bin/*.mjs)` rule):

```bash
node ~/.claude/bin/log-gate-audit.mjs --ticket <TICKET-ID> --outcome "<completed | forced-halt at <phase>>" --pd <n> --intervened <n> --forced <n> --gate "<phase> · <gate> · <classification> · <resolution> — <note>" --gate "<...>"
```

`--gate` is repeatable, one per ledger line, verbatim. The helper stamps `<YYYY-MM-DD HH:MM>`, assembles the block, creates `pipeline/audit/` inline (`mkdirSync` recursive — idempotent, **no `mkdir` prompt**), and appends to the **canonical** checkout's ledger (the path is resolved relative to the helper's own `bin/` location, so the friction map accumulates in the one source-of-truth repo even though this flush runs at §6 — after `/land-ticket` has, in standalone mode, torn the worktree down and dropped the session back into the shared checkout where the bg-isolation guard blocks `Write`/`Edit`). The helper is the only sanctioned path.

One ticket can't prove a gate is dead weight; twenty can. Greppable lines (`grep 'land-ticket §5' pipeline/audit/gate-audit.md`) let a later reader see whether a gate is perennially p'd (prune candidate) or regularly intervened (load-bearing). Never rewrite prior blocks — append only (the helper is append-only by construction).

## 5. Drive the pipeline

Run the phases in order. For each: announce it, read `~/.claude/commands/<cmd>.md`, execute it literally on `$ARGUMENTS`, and intercept its gates per §1–§2, logging each into the ledger. Between phases there is **no** extra `/go` prompt (unlike `audit-cycle`'s per-phase "proceed?") — `/go`'s whole point is to flow straight through; the only pauses are the sub-commands' own gates.

**Announce a phase banner for every step — including one that executes no command — so the on-screen sequence is contiguous (1→2→3→4→5→6) and never reads as a silently-dropped step.** A step that runs nothing still prints a one-line `Phase N (<name>) — <why nothing ran>` banner: a *skipped* conditional phase (Phase 2 `/research` not net-new; Phase 3 `/scope` when `/next-ticket` decided *skip*) prints `Phase N (<cmd>) — skipped: <one-line reason>`, and the *folded* Phase 4 prints its banner per step 4 below. The numbers stay fixed (they map to the §1 taxonomy) — the banner explains the gap rather than renumbering it away.

1. **`/next-ticket <id>`** — sets up the worktree/branch, moves the ticket In Progress, and emits its **route** (`build | fast | fix-bug | research`, the chain-entry router) plus — on route `build` — the scope-necessity decision (`scope` or `skip`) in the hand-off (`fast` carries a guaranteed `skip`; `fix-bug` and `research` carry none), plus the **`Depth-class:`** line (`light|standard|deep`, V-323; no line ⇒ `standard`). Capture all three — downstream `/scope` and `/build` §3.5 read depth-class as a raise-only conditioner on their profile knobs. Any hard stop here → log `forced`, go to §6.
2. **`[/research]`** *(conditional — net-new only)* — if the ticket introduces a **net-new solution** (no existing in-repo pattern; a new dependency/library/service; a novel capability/protocol/algorithm/security-crypto surface — the same trigger `/scope` §5 and `commands/research.md` key off), run `/research <the solution>` to produce the `## Prior art & standards` brief, and thread it into the `/scope` phase so `/scope` writes it into the build plan rather than re-invoking the recon. Routine pattern-following ticket → **skip this phase** (the existing code is the prior art). `/research` is read-only and raises no gates, so there is nothing to intercept; log the outcome (`run` / `skipped`) in the ledger. (When `/scope` is skipped too — `/next-ticket` said `skip` — a net-new `/research` brief still surfaces in the run summary for the human; `/build`'s inline path has no plan to carry it.)
3. **`/scope`** *(conditional)* — **honor `/next-ticket`'s route + gate, do not re-decide** — *except under `--architect`, which forces `/scope` on*: route `fast` → skip this phase unconditionally (the router guaranteed the `/scope` skip — banner: `Phase 3 (/scope) — skipped: route fast, /scope skip guaranteed by the router`); otherwise run `/scope` only if step 1's decision was `scope`; if it was `skip`, skip this phase (`/build` runs its inline path). **When `--architect` is set** (§0), run `/scope` **regardless of the skip decision** (raise-only, like a `deep` depth-class — banner when it would otherwise skip: `Phase 3 (/scope) — forced on by --architect`), so there is a durable `## Implementation design` for the architect stage to steer and for `/build`/`/land` to read; a `fast`/`fix-bug`/`research` route still owns its own path and the architect stage does not insert there (`--architect` on those routes is a no-op — note it and continue). This keeps `/go` consistent with the pipeline's own conditional-scope design rather than forcing scope unconditionally. If step 2 produced a prior-art brief, pass it in so `/scope` lands it in `## Prior art & standards` instead of re-running the recon. Any hard stop (incl. `needs-eyes`, treated as hard) → log `forced`, go to §6.

**Architect stage — `--architect` only, inserted here (after Phase 3 `/scope`, before Phase 4).** When `--architect` is set (§0) and the route is `build`, run one interactive **architect stage** so the architecture decision is demonstrably the human's *before* any code is written. It is **not** a renumbered phase — announce it with its own banner (`Architect stage (--architect) — human-owned architecture decision`) and keep the fixed phase numbers intact. It **orchestrates existing `/plan` machinery by reference — it does not duplicate it**:

  a. **Research** — run `/plan §3`'s research machinery over the ticket's Acceptance + the `## Implementation design` `/scope` just wrote: the parallel `Agent`-per-plan-point findings sweep (observed `file:line`, per convention 8) + a spike-per-genuine-unknown. This is the "conducts extensive research" half; invoke it **by reference** (read `~/.claude/commands/plan.md §3` and run it), never a second copy. Surface the findings to the human in the same format `/plan §3` uses — a short "what I learned" brief, `file:line`-cited, two paragraphs max.
  b. **Present options + take direction** — run `/plan §4/§5`'s option-presentation machinery: present **2–3 candidate approaches** with tradeoffs + relevant standards via **`AskUserQuestion`** (the discrete approach-fork — permitted here because `--architect` makes the human present, convention-11's discrete-fork exception), and open freeform message turns for the human's open-ended questions and direction. For a net-new architectural choice, run `/research <the solution>` for a cited prior-art brief and let it inform the options (the `/plan §5` reuse). Iterate until the human signals the decision is settled — this is a conversation, not a form; do **not** pre-commit an approach.
  c. **Record the human-directed decision** — rewrite the plan's `## Implementation design` section (`docs/plans/<ticket-id-lowercased>-build.md`, lowercased) to the architecture the human directed, and **append a `## Architect session` block** capturing the draft→final change: the `/scope` autonomous draft, the options presented, the human's answers/direction, and the resulting decision. This makes **Acceptance 4** observable — the recorded design demonstrably changed in response to the human's answers, before §4/`/build` writes any code. Commit the plan edit (`docs(plan): architect-directed design for <ID>`) so the human-owned design is durable in the PR.
  d. **Terminal pre-code ack — one mechanism, depth-class-branched:**
     - **`deep`** → render **no** terminal confirm here. `/build §3.5` will thesis-check the human-directed design and, on a `sound` verdict, render its existing convention-11 `[HARD STOP]` pre-code architecture ack-gate (V-325) — that **is** the terminal ack (one mechanism, no second stop). `/go` drives it inline at Phase 5 per §2's ack-gate protocol.
     - **`light`/`standard`** → `/build §3.5` renders no ack-gate on these, so the architect stage renders its **own** terminal convention-11 `needs input:` **confirm gate** before handing to Phase 5 — the goal-restatement line then the stop, never an `AskUserQuestion` modal:
       ```
       <ID>: <one-line goal>
       needs input: proceed to build the architect-directed design above — reply to build it, or amend/redirect the design. Reply to proceed, or state the change.
       ```
       Drive it per §2's one-token protocol (`p` = build as designed; anything else = a literal instruction), logged `p'd`/`intervened`.

  The convention-11 line is held explicitly: **`AskUserQuestion` is confined to the option-forks** (b); **every proceed/irreversible gate stays `needs input:` prose** (d) — `/go`'s gate path forbids modals, and the `needs input:` stop is the only surface the bg-state classifier and `/go`'s `p` path can see. Then proceed to Phase 4 and Phase 5, where `/build §3.5` thesis-checks the *human-directed* design (no `## Thesis-check` block exists yet, so the check fires fresh on the human's decision).

  **Worked example (encoded proof, test-less repo — Acceptance 4).**
  - *`standard` ticket, `/go --architect ENG-77`:* `/scope` drafts `## Implementation design` = "approach A: extend module X in place". Architect stage (a) research surfaces that X is called from three seams; (b) present via `AskUserQuestion`: **A** extend-in-place · **B** new adapter seam · **C** refactor X first — the human asks about blast radius, then picks **B**. (c) `## Implementation design` is rewritten to approach **B**; `## Architect session` records `draft: A (extend in place) → final: B (new adapter seam) — changed by: human chose B after blast-radius Q&A`. (d) standard → terminal `needs input:` confirm → human replies to proceed → Phase 5 `/build` thesis-checks design **B**, builds it. The recorded design changed from A→B in response to the human's answers before any code — **Acceptance 4 observable**.
  - *`deep` ticket:* same stage; at (d) no terminal confirm is rendered — Phase 5 `/build §3.5` returns `sound` on design **B** and renders the V-325 `[HARD STOP]` architecture ack-gate, which `/go` drives inline (the one terminal ack, no second stop).

4. **Design thesis-check — owned by `/build`, not invoked here.** The check fires inside `/build` §3.5 on both paths, so there is nothing to run at this position — but **announce it anyway** (per §5's announce-every-step rule above): print `Phase 4 (design thesis-check) — no separate invocation; folded into /build §3.5, verdict recorded after Phase 5`. Then proceed straight to step 5. **Its design-stage outcome is recorded *after* step 5, not here** — a post-`/build` instrumentation read, never an invocation: once `/build` has run, read the verdict from the plan's `## Thesis-check` block (`docs/plans/<ticket-id-lowercased>-build.md`, lowercased) and log one ledger line per §3 — a `sound` verdict is a no-tally recorded outcome; a non-`sound` verdict will already have surfaced as a `/build` hard stop in step 5 (logged `forced`). **If that block carries a `Product:` line (a design-touching ticket — V-281), surface it in the run summary** (`design-touching: P1/P2/P3 <result>`) so the human sees the product sub-bar ran; an `n/a` Product line (or none) means the ticket was engineering-only and nothing extra is surfaced.
5. **`/build`** — implement, verify, push. Autonomous; any hard stop (verify exhaustion, push rejection, structural impossibility) → log `forced`, go to §6. `/build` §3.5 also runs the design thesis-check — a **non-`sound`** verdict is one of its hard stops (log `forced`, never `p`/`--force` past it). **On a `deep` ticket, a `sound` verdict then renders `/build` §3.5's convention-11 `[HARD STOP]` pre-code architecture ack-gate before §4 (V-325) — intercept it inline while executing `/build` (the gate fires mid-run, before any code is edited): drive it per §2's ack-gate protocol (re-state the gate on a bare `p`, clear only on explicit engagement — never inject `--force`), and log the ack-gate ledger entry `ack'd`/`intervened` (§3). `light`/`standard` render no such gate.** After `/build` completes (or stops), read the verdict from the plan's `## Thesis-check` block and log the design-stage outcome line per step 4 / §3.
6. **`/land-ticket`** — publish + merge + close. Intercept its confirm gates (§5, §6.7) per §2; intercept its hard stops (§4.8 migration-missing, infra, permission prompts) per §1. On a confirm gate, pause for the one-token reply and log `p'd`/`intervened`.

Announce `Phase 3–5 (/fix-bug) — routed repro-first path (route fix-bug)`, read `~/.claude/commands/fix-bug.md`, and execute it literally on `$ARGUMENTS`. Intercept its one gate — §2's close-or-re-scope **confirm gate** — per §2's one-token protocol (log `p'd`/`intervened`); its §3 inherits `/build §4.6` read-back + no-fabrication (claim-blockers, not pauses). If `/fix-bug` closes/cancels the ticket at §2 (can't-reproduce / stale-but-real), the run ends there — there is no code to land; record it and go to §6. Otherwise `/fix-bug` ends by handing to Phase 6 (`/land-ticket`). **No design-stage thesis-check line is recorded for a `fix-bug` run** (the check is dropped by design — note its absence in the summary rather than logging a verdict).

**Route `research` — run the inline investigate→finding-doc path in place of Phases 3–5.** When step 1 captured route `research`, skip Phases 3 (`/scope`), 4 (thesis-check), and 5 (`/build` + `/verify-tests`) — the deliverable is knowledge, so there is nothing to scope, red-team, build, or verify (`docs/research-route-path-design.md`). Unlike `fix-bug` there is **no peer command** — the path is driven inline by `/go`. Print one `skipped` banner + log one `skipped` ledger line per bypassed rung, then run the path:
  - `Phase 3 (/scope) — skipped: research route (deliverable is a finding, nothing to scope)`
  - `Phase 4 (design thesis-check) — skipped: research route (no build design to red-team)`
  - `Phase 5 (/build + /verify-tests) — skipped: research route (no code to build or verify)`
  Then **investigate freeform inline** against the ticket's Acceptance (*not* `/research` — that is external-prior-art recon, the wrong tool for an internal analysis). **Dispatch any research sub-work via the right surface, never an unregistered `subagent_type`:** when the investigation needs prior-art/web depth or fans out into a spike, invoke `/deep-research` (or `/research`) through the **`Skill` tool**, or spawn `Agent` only with a **registered agent type** (`general-purpose`, or `Explore` for read-only search). **Never pass `subagent_type: 'deep-research'` or `'research'`** — those are *skills*, not agent types, so the dispatch errors `Agent type '…' not found` and the spike silently never runs (V-242). **When a spawned `Agent` may write scratch, embed the scratch-path directive in its prompt per convention 5** — hand it the absolute worktree `tmp/` (`$WT_ABS/tmp/`, resolved to its literal value) or `$CLAUDE_JOB_DIR/tmp`, never `/tmp` via a deep relative path — since a subagent's empty context inherits neither this convention nor the harness bg-session scratch preamble (V-322). Then **write the finding doc** to `docs/<finding-slug>.md`, **opening with the structured Decision block `/land-ticket`'s research foreground reads** — `## Question` (the problem being decided) · `## Candidate approaches` · `## Relevant standards` · `## Recommendation` (per-approach pros/cons + the pick) — so the §4.10/§6.7 foreground has a deterministic source and never falls back to dumping merge receipts (V-301); deeper investigation notes follow that block. **Post the finding as a ticket comment carrying at least the question + the Recommendation.** Commit + push the doc-only diff (as `/build` would), and hand to **Phase 6 (`/land-ticket`)**. The investigation raises no gate; `/build §4.6` read-back + convention-8 no-fabrication still bind if it mutates live state (a claim-blocker, not a pause). **No design-stage thesis-check line is recorded for a `research` run** (the check is dropped by design — note its absence in the summary). The ledger for a research run carries the routing line + the three `skipped` lines (none counted as `forced`/`intervened`) and **no** `design-thesis-check` verdict line — the encoded proof (test-less repo) that the mis-route halt is gone (Acceptance 2). The `/land` gates below still apply (the doc-only diff is route-invariant per the §1 inventory bullet).

Also heed any `⚠️ STALE-SKILL RISK` warning the staleness hook (`bin/check-skill-staleness.mjs`) injected. The hook is the real safety net; this re-read is the cheap complement. See `docs/stale-skill-execution.md`.

## Hard rules

- **Never skip an underlying skill's gate.** `/go` removes the manual re-typing between phases, not the gates. Each sub-command owns its own gates; `/go` classifies, drives, and logs them.
- **Never `p` past a hard stop, and never inject `--force`.** Overridable stops are treated as hard (design call #1) — they fire because something is genuinely off.
- **Never `p` or `--force` past a non-`sound` thesis-check.** The verdict is a hard stop reaching `/go` as a `/build` hard stop (`/thesis-check` §4 + design call #1); the **build-despite ack** is the **human's**, never `/go`'s — surface it (log `forced`). The design is then amended or re-scoped in-session — drivable by the agent or the human, producing a fresh design the check re-runs against — or the human gives the deliberate build-despite ack; `/go` never injects that ack itself.
- **Never `p` past the deep-ticket pre-code architecture ack-gate.** On a `deep` ticket a `sound` thesis-check verdict renders `/build` §3.5's convention-11 `[HARD STOP]` architecture ack-gate (V-325); a bare `p` does not pass it — clear it only with explicit human engagement (ack / amend / redirect), logged `ack'd`/`intervened`. Never inject `--force` to pass it (the ack is the human's). It fires only on `deep`; `light`/`standard` render no such gate.
- **`--architect` forces the architect stage + `/scope` on regardless of depth-class (raise-only), and is human-present-only.** The flag never *lowers* a gate — it adds the interactive stage and forces `/scope` even where the router said `skip`, but a `deep` ticket still gets `/build §3.5`'s `[HARD STOP]` as the terminal ack (deferred to, never suppressed). Because the stage blocks on `AskUserQuestion`, an unattended/background `/go --architect` stalls — the flag is meaningful only with a live human (§0). On a `fast`/`fix-bug`/`research` route (which owns its own non-scope path) the flag is a no-op — note it and continue.
- **`AskUserQuestion` never leaks into a proceed/irreversible gate.** In the architect stage it is confined to the **option-forks** (present candidate approaches — a discrete fork, human present, convention-11's exception); every **proceed/irreversible** gate — the stage's terminal light/standard confirm, and every `/build`/`/land` gate — stays `needs input:` prose, the only surface the bg-state classifier and `/go`'s `p` path can drive. A modal on a proceed/irreversible gate is the convention-11 violation this rule forbids.
- **Never carry a `/land-ticket` §4.6 security-review HIGH waiver in `p`, and never offer / default / pre-bundle one.** An unresolved §4.6 HIGH is cleared only by a fix or the human's explicit `waive HIGH: <finding>` token (`land-ticket.md` §6.7) — full rule in §2.
- **Never carry a genuinely-required manual-test residue past the §6.7 pause on `p`, and never suggest `p` as a way to skip it.** A non-empty HUMAN TESTING REQUIRED residue is cleared only by the human's per-item result or an explicit, item-specific waive — full rule in §2.
- `p` = proceed-as-default; **anything else = a literal human instruction, obeyed** (design call #3).
- **Honor `/next-ticket`'s route, scope-necessity decision, and depth-class** — `/go` reads the route (`build | fast | fix-bug | research`), the scope-vs-skip decision, and the `Depth-class:` line (V-323) from the hand-off; it never re-decides any of them.
- **Append-only** to `pipeline/audit/gate-audit.md`; never rewrite prior runs.
- One ticket per invocation.
- Convention 4: name the next step at the end (§6).
