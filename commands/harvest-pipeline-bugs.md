# /harvest-pipeline-bugs — turn the error log into patch-tickets

The **one** consumer of the `errors.jsonl` problem-sink. Captured pipeline errors rot if nothing reads them; this command is the standing reader that clusters the log into **deduped** patch-tickets and files them into the shared `bugs` standalone bucket. The whole sink is **pipeline-subject** — every entry (hook-caught tool failure or `/report-bug`) is a pipeline bug, so it all flows to the one `bugs` bucket regardless of which repo it fired in; the per-entry `origin` is triage context, not a routing key. Runs on demand and via a daily local **launchd LaunchAgent** (a `claude -p "/harvest-pipeline-bugs --yes"` trigger installed by `bin/install-harvest-launchd.sh` — it invokes this command, not a second implementation; launchd, not cron, so a slot missed while the Mac was asleep runs on the next wake). It must run **locally**, since the log is a per-machine file.

Read `~/.claude/workflow-conventions.md` first (esp. conventions 4 + 8). Reuses the `/sweep` → `/triage-findings` → `/bulk-fix` routing/clustering/dedupe patterns — it does **not** re-invoke them (the input is a JSONL log, not a findings doc); it files into the bucket directly, the way `/triage-findings` step 4b does.

## Load config
- `root="$(git rev-parse --show-toplevel)"`; read `$root/.claude/ticket-flow.json`. Missing → `/ticket-flow-init`, STOP.
- Required: `scopeLabel`, `bugBucket` (`{ name, team }`, plus the optional-but-preferred `id`). Resolve the harvest **bucket** name + team from `cfg.bugBucket` — `bucketName = cfg.bugBucket.name`, `bucketTeam = cfg.bugBucket.team`. If `cfg.bugBucket` is unset → STOP (`no bugBucket configured in ticket-flow.json — add { "bugBucket": { "name": "bugs", "team": "<owning team>", "id": "<project-uuid>" } }; see /ticket-flow-init`). **`bucketTeam`, not `cfg.linearTeam`, is the filing team** — so this runs correctly from any repo (e.g. invoked from a product repo on its own team, it still files into the `bugs` bucket's owning team — the pipeline's). The bucket must already exist in Linear under `bucketTeam` with the `parallel` label; this command does not create it (a one-time setup step).
- **Resolve `bucketId` once and use the ID everywhere a Linear call references the bucket.** A name is not a stable handle — a display-name rename resolves to *no project* without erroring, silently orphaning filings and the §4c dedupe:
  - `cfg.bugBucket.id` set → `bucketId = cfg.bugBucket.id` (the pinned UUID; preferred — no lookup, immune to name drift).
  - else → `mcp__linear get_project query: <bucketName>` → `bucketId = .id`. No project resolved → STOP (`cannot resolve bugBucket "<bucketName>" to a project ID — pin "id": "<uuid>" in cfg.bugBucket or fix the name; see /ticket-flow-init`).
  - From here, **`bucketId` is the dedupe + filing key** (§4c, §6); `bucketName` is retained only as the human-readable label for the §5 preview and §7 report. Never pass `bucketName` to a `list_issues`/`save_issue` `project:` field again.

## The sink (single source — read only this)
- **Log path is absolute, main-checkout.** The log lives beside the logger: `~/.claude/pipeline/audit/errors.jsonl`. It is **global, gitignored, per-machine** — every session writes there regardless of cwd (`pipeline/audit/README.md`). **Never** read a worktree-relative `pipeline/audit/errors.jsonl` — a worktree copy is gitignored-empty. Resolve the real path the same way `/report-bug` resolves the logger: the absolute `~/.claude` path, not `$root/...` (in a worktree `$root` is the wrong checkout).
- **Entry shape is fixed (verified on `main`):** each line is `{ ts, session, conversation, activeCommand, origin, tool, input?, error }`. `origin` is the repo the entry fired in (nearest git-root basename, best-effort, may be `null`) — **triage context only, never a routing key** (all entries file into the one `bugs` bucket). `conversation` is the **trace handle**: the resume-chain root of the session that filed the entry — i.e. the *thread* it was spotted in, stable across follow-ups/compaction, so several entries from one conversation share it. May equal `session` (a length-1 / untraceable chain) or be `null` (older entries predating the field). **There is NO `source` field — do not read one** (the `--source` contract was reverted; `bin/log-pipeline-error.mjs`'s flag parser handles only `--command`/`--error`/`--tool`/`--session`). `input` is optional and already `redact()`-ed.
- This log is the **single** problem source. Do not separately ingest session-review reports (they arrive *through* this log via `/review-session`). Do not treat the usage census as a problem source (corroboration only — below).

## 1. Read the new entries
- **Stream the new entries via the sanctioned reader — never a whole-file `Read`.** The log is append-only and unbounded; it now exceeds the 256 KB Read ceiling, so a whole-file `Read pipeline/audit/errors.jsonl` fails outright (`File content (NNN KB) exceeds maximum allowed size (256KB)`). Use `node ~/.claude/bin/read-errors-since.mjs` (allow-listed via `Bash(node ~/.claude/bin/*.mjs)`), which streams the log line-by-line and emits only entries with `ts >` the cutoff — so it never trips the ceiling regardless of log size. Pick the cutoff by flag:
  - `--since <ISO8601>` → `node ~/.claude/bin/read-errors-since.mjs --since <ISO8601>`.
  - `--all` → `node ~/.claude/bin/read-errors-since.mjs --since 1970-01-01T00:00:00.000Z` (epoch — reconsiders the whole log; safe, §4 dedupe prevents refiling).
  - else → `node ~/.claude/bin/read-errors-since.mjs` (no flag — the helper reads the watermark `~/.claude/pipeline/audit/.harvest-watermark` itself, a single ISO timestamp of the last harvested entry; absent → epoch).
- Capture the helper's stdout as the new-entry set. 0 new entries → report "nothing new since `<watermark>`; re-run with `--all` to reconsider the full log." STOP (still update nothing). The watermark is an **optimization**, not the correctness guard — §4's dedupe-against-open-tickets is.

## 2. Route + weight each entry by the `(tool, activeCommand)` pair
The writer discriminator is the `(tool, activeCommand)` pair — there is no `source` field. First decide **manual vs hook**, then sub-route manual entries by `activeCommand`.

**Manual-shaped detector (which writer wrote it).** An entry is **manual-mode** (the logger's behavioral backstop — `/report-bug`, future `/review-session`) iff it is `tool === "manual"` **OR it carries no `input` field**. This is structural, not a new field: the logger's `runManual()` writes `{ts, session, activeCommand, tool, error}` and **never** an `input`, whereas hook mode (`buildRecord`) always attaches `input` for an input-bearing tool. The `input`-absence test is what catches a `/report-bug --tool <subsystem>` entry, whose `--tool` overrides the default `"manual"` (so `tool` is e.g. `"go"`, not `"manual"`) while it is still a human report. Conversely a *hook* failure that merely happened **during** a `/report-bug` session has a real `tool` (e.g. `Bash`) **and** an `input` → it stays **hook**, even though its transcript-scanned `activeCommand` is `report-bug`.

| Condition | Route | Weight | Meaning |
|---|---|---|---|
| **not** manual-shaped (real `tool` + has `input`) | **hook** | by corroborated frequency (§3) | A hook-caught tool error (Bash non-zero, MCP/API failure, `PermissionDenied`, file/command not found). The transcript-scanned `activeCommand` names the command that was active — corroboration, not the route signal. |
| manual-shaped + `activeCommand === "report-bug"` | **human** | **high** | A human front-door report via `/report-bug` — a person took the time; weight it up. (Holds whether or not `--tool` tagged a subsystem.) |
| manual-shaped + `activeCommand === "review-session"` + a **`[lens-a/…]`** payload | **review** (Lens A) | **high** | A **Lens A** finding (`Allow` / `Deny/Ask`) emitted by `/review-session` — a `settings.json` permission patch-target keyed on a stable `<shape>` + `<reason>`; **maps near-directly to a patch** (the honest case). |
| manual-shaped + `activeCommand === "review-session"` + a **`[lens-b/…]`** payload | **review** (Lens B) | **low** | A **Lens B** finding — a correctness *candidate* (`failed-then-claimed` / `fabricated-id?`) or a conv-8 *pattern* flag (`error-swallow` / `doc-asserts-state`). It maps to **"go read this session," not a committed-artifact fix** — **never draft a `fix-the-command` for it** (an unbuildable `## Acceptance` burns a downstream `/go`/`/bulk-fix` run). Surface it for a human read, or leave it unfiled per the `--yes` quality bar; never a `high`/patch ticket. Lens B is now suppressed at `--emit`, but historical rows already in the sink still stream into this route on `--all` (§1) — cluster by `[lens-b/<type>]` per §4a and route away from patch. |
| manual-shaped + any other `activeCommand` | **self-report** | low | In-session self-report. **Dead branch** — canceled; no writer produces these today. Keep the route stub so a future writer drops in, but expect zero. |

Weighting feeds ticket **priority** (human → High; review/**Lens A** → High, review/**Lens B** → Low — Lens B is a candidate/pattern-flag, not a patch-target; hook → by frequency: recurring/critical → High, occasional → Medium, one-off → Low) and ordering in the preview.

## 3. Corroborate against the usage census (counts only — never a second source)
- The census (`bin/usage-stats.mjs`) supplies raw failed/repetitive-call **counts** to confirm a hook cluster is a real recurring problem vs a one-off. Run it read-only if useful; treat its numbers as corroboration of clusters already found in the log. **Never** mine the census for problems the log doesn't contain — the log is the only problem source.

## 4. Cluster + dedupe (one ticket per distinct problem, never per occurrence)
### 4a. Cluster by ROOT CAUSE (a problem, not a command-occurrence)
Group entries by their **normalized error/note shape** — the *problem*, independent of which command or tool hit it. Normalize first, to kill volatile fragments before comparing/hashing:
- strip the leading `Exit code N\n` preamble,
- replace absolute paths and `«redacted»` tokens with placeholders,
- drop branch names / timestamps / line-and-column numbers,
- **review-payload volatiles** — review-route entries carry per-occurrence fragments the way hook errors carry timestamps / line-cols; strip them so review findings cluster on their **stable anchor** before the §4b hash (emit-format authority: `bin/session-review.mjs:529/534`, read-only):
  - **Lens A** (`[lens-a/<bucket>] <shape> ×<count> — <reason>`): strip the `×<count>` repetition count (`×\d+` — the `×` is the U+00D7 multiplication sign emitted at `:529`, not an ASCII `x`). **Keep** `<bucket>`/`<shape>`/`<reason>` — they describe the allowlist candidate and are stable, so the same candidate dedups across sessions instead of splitting one key per count.
  - **Lens B** (`[lens-b/<type>] msg#<N> — <detail>`): strip **both** the `msg#<N>` message ordinal (`msg#\d+`) **and** the trailing per-turn excerpt (everything after the `— ` separator — a redacted snippet that varies every occurrence). Key on `[lens-b/<type>]` alone, so Lens B clusters by failure type instead of producing one near-unique key per occurrence.
  - The Lens-A-vs-Lens-B asymmetry is deliberate: Lens A keeps its `<reason>` (a stable description), Lens B drops its `<detail>` (a volatile excerpt) — stripping Lens A's reason would over-merge distinct allow candidates sharing a `<shape>`; keeping Lens B's detail would under-dedup it.
- collapse whitespace; keep a stable leading slice of the message.

**`tool` and `activeCommand` are EVIDENCE, not cluster dividers.** The same root cause routinely fires from several commands (e.g. `docs/` being gitignored breaks `git add docs/plans/<id>-build.md` in `/scope`, `/next-ticket`, **and** `/report-bug`); that is **one** problem → one ticket, citing all affected commands in its evidence — not three tickets keyed by `activeCommand`. Keying on `activeCommand` is the over-narrow-clustering trap: it refiles the same root cause per-command and per-run. Include `tool`/`activeCommand` in the key **only** when two genuinely-distinct problems happen to share a normalized error (rare — split by judgment and salt the key).

This normalization + root-cause grouping is the **highest-risk correctness detail** — under-normalizing (or keying on `activeCommand`) refiles the same problem every run; over-normalizing merges distinct bugs. When two clusters are plausibly the same root cause, **merge** and let the evidence list carry the per-command detail.
### 4b. Compute a stable `harvest-key`
`harvest-key = <route>:<short hash of the normalized error/note>` — deterministic, `tool`/`activeCommand`-independent, stable across runs and across the commands a root cause fires from. This key is the dedupe handle. (When 4a merged clusters that a future run might re-derive as separate keys — e.g. because the normalization slice differs — list **all** constituent keys on the ticket's `harvest-key:` line, space-separated, so any one re-derived key matches.)
### 4c. Dedupe against already-open bucket tickets
- `mcp__linear list_issues` `project: <bucketId>` (the resolved §0 project **ID**, never the name), `state` in `[Todo, Backlog, In Progress]` (open work — Done/Canceled excluded so a genuinely-recurring-after-fix problem *can* refile).
- Each harvested ticket body carries a `harvest-key: <key…>` line (§5, possibly several keys). **Drop any cluster whose re-derived key matches any key on any open ticket** — a recurring problem is never refiled while its ticket is open. Title + evidence corroborate; the key is primary. Count drops; report them.
### 4d. Stale-prod guard — before refiling a recurrence of a *merged-fix* ticket
§4c deliberately lets a recurrence of an already-fixed problem refile (Done/Canceled are excluded from the dedupe). But a recurrence is a **code gap** only if prod is actually running the fixed code — otherwise it is **deploy lag**, and refiling it as "fix incomplete / broken code" is the V-256 misdiagnosis: V-225's `save_issue` state-name fix (#176) was merged and correct, but the Hetzner wrapper ran pre-fix code from a stale deploy, so the same `ambiguous … matched 7 — pass an id` error kept firing and the ticket was wrongly reopened as incomplete. So for any surviving cluster that **matches a recently-closed bug ticket whose fix merged** — re-derive the cluster key against **recently Done/Canceled** bucket tickets (one extra `list_issues` `project: <bucketId>`, `state` in `[Done, Canceled]`, recent window) that carry a linked merged PR — run this guard **before** drafting it as a recurrence:
1. **Compare the live error to the merged code's wording.** Take the error's stable, code-emitted fragment (the un-normalized `error`, minus §4a volatiles) and `git grep` / `git log -S` it against the merged fix on `origin/main`. If the **post-fix** code cannot emit this exact wording — the fragment survives only in pre-fix history, or the fix changed the message — the recurrence is running **old code**. (V-225 tell: live `matched 7 — pass an id` vs the post-fix message `matched N in the target team`, which the new code emits instead.)
2. **Check the deployed artifact's freshness.** When the fix ships to a **deployed/built artifact** (a service bundle, not this source repo itself), inspect what prod actually runs: does the deployed bundle contain the fix's **new symbols**, and is its **mtime** at/after the fix's merge? (V-225: `resolveStateIdForTeam` ABSENT from `/opt/linear-mcp/dist`, bundle mtime `2026-06-10` ≪ #176's `06-18` merge.) Missing symbols / a pre-fix mtime ⇒ stale deploy. If prod is not independently reachable, the string tell in (1) alone is sufficient.
3. **Verdict.** An **old-format error** or a **stale deployed bundle** ⇒ flag **`stale prod` (deploy lag), not broken code**: do **not** refile it as a code-gap recurrence. Draft it, if anything, as a **redeploy** need — fix-class `fix-upstream`, naming the deployed target and the merged PR that isn't live — or drop it with a note that the fix is merged and only needs deploying. Only a recurrence carrying the **post-fix** wording against an **up-to-date** deploy is a genuine code gap that earns a code-fix refile.

This guard governs the **reopen decision** whether the harvester runs it in `--yes` mode or a human reads the drafted ticket at the §5 gate — the encoded proof (this test-less repo) that the V-225 misdiagnosis cannot silently recur: a recurrence whose error string the merged code cannot emit is routed to `stale prod`, never to "fix incomplete."

## 5. Outline the patch-tickets + confirm gate
For each surviving (deduped) cluster, draft a patch-ticket:
- **Title** (≤80 chars, imperative): the problem, e.g. `Fix: git add docs/plans/<id>-build.md fails — docs/ is gitignored`.
- **Body:**
  - **Problem** — one paragraph: what goes wrong.
  - **Evidence** — which sessions / how many occurrences / the writer (hook vs human vs review) / the `origin` repo(s) it fired in. Cite entry `ts`s and `session`s. (The cluster-level tally — the *verbatim* failing context goes in **Occurrence** below, not here.)
  - **Occurrence** — a short, coherent paragraph reconstructing *what was happening and what broke*, built from the cluster's **richest single representative entry** (the one carrying an `input` and the most specific verbatim `error` — pick by content, not recency). Draw on that entry's **verbatim (un-normalized) `error`** — the stored value is already `redact()`-ed but **not** §4a-normalized, so it still carries the absolute path / failing `input.command` + args / branch / line-and-column that the dedupe key strips — plus its `tool`, `activeCommand`, and `origin` repo. This is the reproduction context a fixer acts on; the §4a-normalized form is **not** reproduced here — it survives only as the `harvest-key`. Write it as prose a human can act on, not a field dump (e.g. *"Running `/go ENG-136` in `myapp`, the `/impeccable` readiness poll `… [ \"$code\" != 200 ] && echo …` exited 1 on the success path, tripping the §6 premise check."*). Never fabricate a path/command the entry didn't carry (convention 8); when the richest entry has no `input`, say so and reconstruct from the verbatim `error` alone.
  - **Fix sketch** — one of: **script-it** / **allow-it** (a settings allow-rule) / **fix-the-command** / **fix-upstream** / **doc-it**. For `review`-route **Lens A** entries the fix-class is often already supplied by the Lens A finding — carry it through (§2). A cluster the §4d stale-prod guard flagged `stale prod` is **`fix-upstream` (redeploy)** — naming the deployed target + the merged PR that isn't live — never `fix-the-command` (the code is already correct on `main`). A `review`-route **Lens B** entry has **no patch fix-class** — it is a candidate/pattern-flag, not a committed-artifact bug; do not force one of the five onto it (see §2's Lens B row: surface for human read or leave unfiled, never a `fix-the-command`).
    - **Artifact-locality check before `fix-the-command`.** `fix-the-command` means *edit a committed command/script in **this** (the harvest) repo* — so it's only honest when such a target exists here. Before drafting it, confirm the failing call maps to a committed artifact in the harvest repo. Two tells it does **not**: the entry's `origin` ≠ the harvest repo, or its `activeCommand` names no command/skill that exists here. A third: the failing `input.command` is an **ad-hoc inline command** with no committed home (`git grep` / `git log -S` a stable fragment finds nothing — the harvested-bug-premise disproof, run now at harvest instead of later at build). This uses `origin` only to pick a truthful fix-class — **not** to re-route the bucket (origin stays out of routing; every entry still files into the one `bugs` bucket). When any tell holds, do **not** draft a this-repo `fix-the-command`: its `## Acceptance` would be unbuildable here and burns a downstream `/go`/`/bulk-fix` run. Instead: **`fix-upstream`** when a committed artifact exists in another repo (name the `origin` repo in the body so a reader fixes it there); **`doc-it`** when the lesson generalizes (a footgun worth documenting here even though the occurrence was ephemeral). Either keeps the bug tracked under an Acceptance that's actually verifiable.
  - **Origin handle (always present when the source carried one)** — on its own line(s), so the build-time fallback (`next-ticket.md`'s bugs-bucket transcript check) and a human reviewer can pull the source with `/ingest-convo <id>`:
    - `conversations: <id…>` — the **distinct** non-null `conversation` ids across the cluster's entries that differ from their `session` (de-duplicated resume-chain roots — a genuine thread; one thread that reported the bug twice appears once), space- or comma-separated.
    - `sessions: <id…>` — for every entry whose `conversation` is `null` **or equal to its `session`** (the degenerate length-1 chain — the common case today), the distinct `session` id(s) instead. `conversation` and `session` ids share the id-space, so `/ingest-convo` resolves either.
    A cluster may emit both lines (mixed entries) or just one. **Omit the handle entirely only when no clustered entry carried `conversation` *or* `session`** (truly handle-less older entries) — never drop it by omission, and never invent one. This is what guarantees Part B always has something to open.
  - `harvest-key: <key>` — on its own line (the dedupe handle; do not omit).
  - `## Acceptance` — one verifiable item derived from the fix sketch (convention 3). For `script-it`/`allow-it`/`fix-the-command`/`doc-it` it's a this-repo build target. For `fix-upstream` it is **not buildable here** — phrase it as the upstream change plus the observable this-repo signal (e.g. "`errors.jsonl` shows no recurrence of `<pattern>` from `<origin>` after the upstream fix"), never a this-repo edit that doesn't exist.
- **Priority** from §2 weighting.

**Confirm gate (default, interactive):** print the drafted tickets (title · route · priority · evidence one-liner · fix sketch) + the dedupe drops + the route tally (hook N, human N, review N, self-report N). **No Linear writes until explicit go.** With `--dry-run`: print this preview and STOP (write nothing, do not advance the watermark) — `result: dry-run — <K> tickets would file into <bucketName> (<D> deduped against open), watermark unchanged. Re-run /harvest-pipeline-bugs to file.`

**Unattended mode (`--yes` — for the daily cron / headless `claude -p`):** skip the confirm gate and file the deduped clusters directly. This is safe because the **dedupe** in §4c (not the gate) is the correctness guard against refiling — the gate is human curation, absent in a cron. Still print the same preview to stdout (so the cron's log shows what was filed). `--yes` and `--dry-run` are mutually exclusive (`--dry-run` wins — never writes). In `--yes` mode, hold a quality bar on its own: file only clusters that clear the genuine-problem bar (recurring hook clusters, human/review reports); leave bare one-offs and infra/flaky blips (e.g. a single MCP timeout) unfiled, same judgment a human would apply at the gate.

## 6. File into the bugs bucket (on go)
For each drafted ticket: `mcp__linear save_issue` `team: <bucketTeam>`, `project: <bucketId>` (the resolved §0 project **ID**, never `bucketName` — passing a stale name creates the issue with **no project association at all, without erroring**, the orphan case), `labels: [<scopeLabel>]`, `priority` (per §2), `title`, `description` (the §5 body, `harvest-key:` line included verbatim so the next run's §4c dedupe matches). **`team` is `cfg.bugBucket.team` (the bucket's owning team), not the repo's `linearTeam`** — cross-team-correct from any repo. After each create, **read back the response and assert BOTH** (convention 8 — never claim a file landed in the bucket without confirming it did): the returned `id` is a real returned identifier (never fabricated), **and** the returned `projectId` **equals `bucketId`**. If `projectId` ≠ `bucketId` (or is absent) → the ticket orphaned despite the ID → **STOP and report loudly** (`save_issue for "<title>" landed in projectId <got> ≠ bucket <bucketId> — orphaned filing, aborting harvest`); do not advance the watermark, do not silently skip. Any create fails → STOP and report.

## 7. Advance the watermark + report
- Advance the watermark to the max `ts` of the harvested entries (or now, if all were deduped) — **only on a real (non-dry-run) pass** — via the sanctioned writer, NOT a `Write` tool or `>` redirect: `node ~/.claude/bin/advance-harvest-watermark.mjs --ts <max-ts-or-omit-for-now>`. The helper resolves `pipeline/audit/.harvest-watermark` from its own `bin/` location and writes the single ISO line itself. **Why the helper:** a raw `Write`/`>` to that dotfile trips the harness sensitive-file prompt, so in the unattended `--yes` cron the watermark never advances; the `node ~/.claude/bin/*.mjs` call is allow-listed and prompt-free. **Note the asymmetry:** a *deduped* problem can refile once its ticket closes (its entries stay re-derivable), but an entry the §5/`--yes` quality bar left **unfiled** is dropped permanently — the watermark advances past it. Re-examine the full log with `--all` if you suspect a skipped one-off has become a real pattern.
- **Rotate the live log if it has crossed the size threshold — only on a real (non-dry-run) pass, AFTER the watermark advance above**: `node ~/.claude/bin/rotate-errors-log.mjs` (allow-listed via `Bash(node ~/.claude/bin/*.mjs)`). It rolls the entries the just-advanced watermark now marks harvested (`ts <= watermark`) into a dated, gitignored archive `errors-<stamp>.jsonl` and rewrites the live `errors.jsonl` with only the un-harvested remainder (`ts > watermark`) — so the live sink stays under the 256 KB Read ceiling **without ever dropping an un-harvested entry** (the watermark is the safety key; an entry is archived only when provably harvested). It is a **no-op** under the size threshold, or when no watermark exists yet (nothing provably harvested), so it is always safe to call unconditionally here. **Order matters:** run it *after* the advance, never before — rotating against a stale watermark would under-archive. Skip it on `--dry-run`, like the watermark advance.
- Report: tickets filed (count + IDs + bucket URL), deduped-against-open (count), route tally, watermark advanced to `<ts>`.
- **Next step (convention 4):** `/bulk-fix --project "<bucketName>"` (run from the V repo, where `<bucketName>` is in `standaloneProjects`) to batch-execute the atomic ones; `/next-ticket <ID>` for any that need judgment.

Emit `result:` on its own line: `result: /harvest-pipeline-bugs filed <K> patch-ticket(s) into <bucketName> (<D> deduped against open), watermark → <ts>. Next: /bulk-fix --project "<bucketName>".`

## Hard rules
- **Single source.** Reads only `errors.jsonl`; routes by the `(tool, activeCommand)` pair; **no `source` field** is read. Never separately parses session-review reports; never treats the usage census as a problem source (corroboration only).
- **Always the main-checkout log** (`~/.claude/pipeline/audit/errors.jsonl`), never a worktree copy.
- **Deduped, idempotent.** One ticket per distinct problem (clustered), never per occurrence; re-running never refiles a problem whose bucket ticket is still open (`harvest-key` match against Todo/Backlog/In-Progress). Re-run after a fix lands (ticket Done) is allowed to refile if the problem genuinely recurs.
- **Stale-prod guard before refiling a merged-fix recurrence (§4d).** Never refile a recurrence of an already-merged-fix ticket as "broken code" without first checking the live error string against the merged code's wording **and** the deployed artifact's mtime/symbols — an old-format error (one the post-fix code cannot emit) or a stale bundle is **deploy lag**, flagged `stale prod` and routed to redeploy (`fix-upstream`), never reopened as "fix incomplete."
