# /land-ticket — publish + intelligent merge + close

Run this when the build phase is done. It absorbs publishing: if the ticket has no PR yet, it pushes and opens the Linear-linked PR, then runs the full land flow. **Human-invoked**; you do the analysis/resolution/merge/cleanup but **STOP for an explicit confirm before any irreversible action (merge, migration, branch/worktree deletion).** Never auto-merge past the gate. Never force-push. Never `reset --hard` / discard a side to "resolve" a conflict.

Read `~/.claude/workflow-conventions.md` first and follow it. (**Under `/go`**, `workflow-conventions.md` + the active profile are already in context from the chain's top-of-run read — skip the re-read per the `read-discipline` knob (V-293); a standalone `/land-ticket` reads it here.)

## Linear MCP call discipline (token-critical)
`list_issues` defaults to `limit: 50` and ~1.2KB per issue (null SLA fields, IDs, timestamps included) — payloads stay in context. **Always pass `state` + a tight `limit`.** In this skill the only list calls are existence checks (§7 completion check): use `state: <X>`, `limit: 1`, short-circuit on first hit. Use `get_issue` for the one ticket being landed — that's the one full payload we need.

**Schema-valid args — never pass `query` to these:** `get_issue` takes `id` (the issue identifier), **not** a `query` key — a `query` key is rejected with `MCP error -32602 … unrecognized_keys ["query"]`. `list_project_labels` likewise takes no `query` (its args are `name` / `limit` / `cursor` / `orderBy`).

## Bash discipline (read before composing shell)

Claude Code's allowlist matches **full command strings**. A compound like `sleep 8; until gh ...; do sleep 15; done; gh ... | python3 -c '...'` does NOT match `Bash(gh *)` — any pipe, chain, or control structure breaks the prefix match and triggers a permission prompt every iteration. Two correct patterns:

1. **Stateful loops or pipeline-heavy work** → call a helper in `~/.claude/bin/`. One Bash invocation, allowlisted via `Bash(node ~/.claude/bin/*.mjs)` / `Bash(bash ~/.claude/bin/*.sh)`. Used by §6.7 (`wait-for-check.mjs`) and §8.5 (`usage-stats.mjs`).
2. **Single-shot binaries** → discrete Bash calls. Each call matches its own allowlist pattern (`gh *`, `git *`, `npx tsc *`, etc.). **Claude parses outputs in-context** — never pipe `gh --json` to `python3 -c` or `jq` from a skill; the skill's reasoning is the parser.

## Load config (required)
- `cfg="$(git rev-parse --show-toplevel)/.claude/ticket-flow.json"`. Read it. If missing → tell the user to run `/ticket-flow-init`, STOP.
- **Resolve the main checkout once:** `MAIN_WT="$(git worktree list --porcelain | sed -n '1s/^worktree //p')"` — the path every `git -C "<main worktree>"` op in §7–§8 (and the `<main worktree>/.claude/usage-stats/…` writes) targets. The **first porcelain entry is always the main worktree** (git lists it first even when invoked from a linked worktree), so this is repo-agnostic and always absolute. **Never** derive the main checkout from the repo slug (`~/<repo>`) or a `…/projects/<owner>-<repo>` path, and **never** use `git rev-parse --show-toplevel` for it — inside a ticket worktree that returns the *worktree*, not the main checkout. Everywhere below, `<main worktree>` = `$MAIN_WT`.
- Fields: `linearTeam`, `scopeLabel`, `repo`, `baseBranch`, `requiredCheck`. Use these everywhere — never hardcode.
- Default merge strategy: **squash** (`gh pr merge --squash`) — collapses each PR to a single commit on `<baseBranch>`, so history stays near-linear and merge commits stop accumulating (the V-244 hygiene fix — the human asked for *fewer merge commits*). The §7 teardown already assumes squash: it force-deletes the local branch (`branch -D`) precisely because a squash leaves the local tip a non-ancestor of `<baseBranch>`. Use a **merge commit** (`gh pr merge --merge`) only when asked, or when a multi-commit PR's individual-commit structure is genuinely worth preserving.

## 0. Publish if needed (absorbed ship step)
- **Strip review flags first.** From `$ARGUMENTS`, capture `--skip-review` or `--full-review` if present and remove them — store as `<review_override>` for §4.5. Also capture `--security-review` (force the §4.6 gate ON) or `--no-security-review` (force it OFF) if present and remove them — store as `<security_override>` (`force-on` / `force-off` / unset) for §4.6, and surface which override was applied in the §9 report. Remaining argument(s) are the ticket/PR identifier.
- Resolve the ticket: `$ARGUMENTS` (issue id or PR#), else infer from the current worktree branch.
- `mcp__linear get_issue`; collect linked PRs (Linear `attachments` + `gh pr list --search "<id>"`).
- If the ticket has **no open PR**: confirm you're in its worktree, working tree committed (commit per repo conventions if not; never `--no-verify`); ensure the worktree has deps installed (a fresh project worktree may not — bootstrap first) and verify the required check passes via the **repo-local toolchain** (`npx --no-install tsc --noEmit` or `node_modules/.bin/tsc`), NOT a bare `npx tsc` — a bare one in a deps-less worktree installs the unrelated `tsc` package and exits 0 (false green; the "This is not the tsc command you are looking for" output = failure); `git push -u origin <branch>`; then compose the PR body into a temp file — `body="$(mktemp)"` — that links the ticket with a **non-closing** reference (`Part of <ID>`, `Relates to <ID>`, or a bare issue link) — **never a closing keyword** (`Closes/Fixes/Resolves/Completes/Implements <ID>`). A closing keyword auto-transitions the ticket (and Linear scans the whole body, so *any sibling it names*) to Done on merge, before acceptance is checked. **Distinct hazard — closed-ticket reopen:** even a *non-closing* ref emits a bare `V-<n>` token, Linear auto-attaches the PR to that issue, and its "PR opened → In Progress" automation drags an already-Done/Canceled issue back open. So in the PR body, **do not emit a bare `V-<n>` identifier for an already-closed cross-referenced ticket** — name it in prose / without the identifier; reserve bare `V-<n>` tokens for this PR's own (open) ticket and other still-open links (mitigation only — *any* identifier mention can re-link). Run `pr-close-guard --body-file "$body"` and abort on non-zero **before** `gh pr create --base <baseBranch> --body-file "$body"`. **Then remove the temp body as its own discrete Bash call — `rm -f "$body"` — never stapled onto another command** (conv. 7). The ticket is closed explicitly + acceptance-gated in §8, not by a magic-word. Then `mcp__linear save_issue` → In Review.
- If PR(s) already exist: continue.

## 1. Target list
List the PR(s) to land + order. Multiple → migrations first, smallest blast radius first. Every PR body references the ticket with a non-closing ref (no closing keywords — and no bare `V-<n>` for an already-closed sibling, which would reopen it; see §0); run `pr-close-guard <n>` per PR before merge and abort on non-zero (catches a body edited after creation).

## 2. Per-PR health read + classify
For each PR: `pr-health <n>` (the verb — wraps the gh pr-view rollup: state, mergeable, mergeStateStatus, checks, base/head, ±size) + `gh pr diff <n>`. Record head branch + worktree (`git worktree list`).
**Classify any failing/absent CI before reacting:**
- **Code failure** (real tsc error, real conflict, a test asserting wrong behavior) → genuine; resolve or STOP.
- **Staleness** (no CI ran because the workflow file didn't exist on a branch cut before CI was added; lint/build exits **127 = command not found** → missing dep from an old lockfile) → step 2.5.
- **Infra** (DB unreachable / project paused / `tenant not found`; network) → STOP and report. Do **not** blind-retry; only retry once the user confirms the root cause is fixed.
- **Pending** (no conclusion yet — `<requiredCheck>` is `IN_PROGRESS`/queued; a freshly pushed PR shows `mergeStateStatus: UNSTABLE` purely because checks haven't finished, or because non-blocking jobs are red by design). This is **not** a failure — do not classify it as one. Poll the required check to completion **via the helper** — `node ~/.claude/bin/wait-for-check.mjs --pr <n> --check <requiredCheck>` (see §6.7 for the canonical pattern; never compose `until`/`while`/`sleep` loops in shell) — before the step-5 gate; UNSTABLE caused only by non-blocking jobs while `<requiredCheck>` is green is acceptable.

## 2.5 Branch-freshness remediation (pre-baseline branches)
A branch cut before the current `<baseBranch>` baseline lacks the CI workflow / good lockfile / recent fixes; its CI can't pass until updated. This fix is **not** gated (feature branch, reversible):
- In the head worktree: `git fetch origin <baseBranch>` → `git merge origin/<baseBranch>` (this *is* step 3's conflict work — do it for real).
- Resolve per step 3, verify (step 5), commit, **push the branch** (pushing a feature branch to re-trigger CI is pre-approved). Re-poll CI.
- Required gate is `<requiredCheck>` — it must go green before the confirm gate. Non-blocking jobs may stay red by design.
- If `<baseBranch>` advances again before merge (e.g. a sibling PR landed), repeat the merge — GitHub runs the workflow from the **head** branch.

## 3. Conflict analysis by reading code (the core)
Don't trust the diff alone. In the head worktree, with `origin/<baseBranch>` merged in:
- For every conflicted file: **read the full file and both sides** (`git show :2:<f>` ours, `:3:<f>` theirs, + context). Determine each side's *intent*:
  - **Orthogonal** (each edits different logic/regions for unrelated reasons) → integrate both sides.
  - **True semantic conflict** (both change the same behavior incompatibly) → don't silently pick; draft the most plausible reconciliation AND flag it `needs-eyes`.
- **Freshness-header-only conflict (auto-resolve, never gated).** If a conflicted file's *sole* conflicting hunk is the System-doc freshness header (`> Last verified against code: YYYY-MM-DD (PR #N)` — both sides changed only that line), resolve it automatically: keep the **newer date** (tie → higher PR number); no `needs-eyes`, no prompt. This is the every-parallel-land collision class. Log the resolution in the §9 report (`freshness-header auto-resolved: <doc> → <date> (PR #N)`). **Scope strictly:** if the file has *any other* conflicting hunk, it is a normal conflict — fall through to the intent analysis above; do not auto-resolve.
- **Cross-sibling 3-way merge scan (`git merge-tree` — V-330).** Per-PR `MERGEABLE` compares each branch against pristine `<baseBranch>` only, so a real conflict between **two open PRs** — e.g. EDIT-1 ↔ EDIT-25 both rewriting `src/components/blocks/usePageEditor.ts` — stays invisible until one lands and the other rebases. **Enumerate + fetch the sibling heads here** (this scan owns its inputs — do not depend on §4, which runs later): `gh pr list --state open --json number,headRefName --jq '.[] | select(.number != <n>) | .headRefName'` → `git fetch origin <each>`. For each *other* open PR head, compute a proactive 3-way merge against this branch and scan for conflicts. On git ≥ 2.38: `git merge-tree --write-tree <this-branch> origin/<sibling>` — **exactly two branch args; `--write-tree` auto-computes the merge base, so passing a third (base) arg errors and must not be done** — a **non-zero exit** = they conflict (and the trailing "Conflicting files" section names them). On older git: `git merge-tree $(git merge-base <this-branch> origin/<sibling>) <this-branch> origin/<sibling>` and scan the output for `<<<<<<<` conflict markers. A conflict → surface at §5 as an advisory **`sibling-conflict-risk: <this-branch> ↔ <sibling> on <files>`** (named files) — **not** a hard block: the sibling PR may land first, or never, so it is a heads-up the human weighs, not a build defect of this branch. Parse defensively; if `merge-tree` can't be computed (version skew, a head that won't fetch), note `sibling-conflict scan unavailable — check manually` and continue (never error the land).
- Produce a conflict report: file → what each side did → resolution → confidence.

## 4. Migration & schema-ripple scan
- **DB-change summary (always emit — this is the surface the gates read).** `gh pr diff <n> --name-only` → filter `supabase/migrations/*`. For every migration added/changed in the PR, read it and print a loud one-line *what it does* (table/column/enum/policy/index touched). If any migration is present, lead with **`⚠️ DB CHANGES — landing pushes these to prod:`** then the list. If none, state plainly **`No DB/migration changes in this PR.`** This summary is surfaced at the §5 gate (before db push) **and again** at the §6.7 manual-test pause (before merge) — never silently skipped.
- Detect `supabase/migrations/*` in the PR → must be **applied as part of landing** (`sb-push --apply` — wraps `supabase db push --linked`); production action, **always-confirmed**: `sb-push --apply` / `supabase db push` must **never** be allowlisted (added to `allow`) — they stay prompt-on-every-run. Today this holds by default-deny (no `allow` rule for `sb-push`) plus the `ask Bash(supabase db push*)` rule; keep it that way.
- **Cross-branch migration-prefix collision check (hard gate — V-330).** `sb-push`'s duplicate/ordering guard fires only at the prod-push boundary, against the *remote applied history* — so a duplicate `YYYYMMDDHHMMSS` prefix among **open sibling PRs**, or one sorting before `<baseBranch>`'s latest, stays latent: git merges the distinct filenames cleanly, every PR shows `MERGEABLE`, and the clash only detonates at `db push`/`db reset` (the prefix is the PK in `schema_migrations`). When this PR touches `supabase/migrations/*`, run `migration-collision-check` (the `allow`-listed, read-only V-330 guard — a sibling of `sb-push`'s check, keyed on git refs not remote history) over the **merged set** *before* the §5 gate:
  - `git fetch origin <baseBranch>`, then enumerate + fetch every *other* open PR's head: `gh pr list --state open --json number,headRefName --jq '.[] | select(.number != <n>) | .headRefName'` → `git fetch origin <each>` (a head that can't be fetched is skipped by the guard's unresolvable-ref degrade — never an error).
  - `migration-collision-check --base origin/<baseBranch> --head <this-branch> --head origin/<sibling1> [--head origin/<sibling2> …]`. Exit 1 → **hard STOP, do not proceed to §5**: `needs input: cross-branch migration collision — <the guard's named offenders>. Two branches carry the same YYYYMMDDHHMMSS prefix (or one sorts out-of-order); re-mint the later migration via `sb-new` and repush before landing.` **Diff-driven and route-invariant** (like §4.8): it fires only when a migration is in the diff, regardless of the ticket's predicted route. (Minimal-core equivalent per the acceptance: `ls supabase/migrations | cut -c1-14 | sort | uniq -d` must be empty on the merged set — the guard computes exactly this across the refs.)
- **Content-safety lint.** Run `bin/migration-lint supabase/migrations/*.sql` over the PR's migrations and print its findings **inside the DB-change summary above**, so squawk's verdict on dangerous ops (DROP COLUMN, `ADD COLUMN ... NOT NULL` rewrite, type change, non-`CONCURRENTLY` index, rename, DROP) is in front of the human at the §5 gate. Advisory here — the hard stop already fired in `sb-push --apply` (fail-closed on any finding, exit 1 before the lock/push; squawk absent fails closed too). §5 just surfaces the same finding for the confirm.
- **Dependency pre-check (before approving):** for `ALTER TABLE ... ALTER COLUMN ... TYPE` / `ALTER TYPE` / `DROP TYPE`, grep migrations for views/rules selecting that column. PG refuses to alter a column a view depends on. A correct migration DROPs dependents first and recreates them after using the **latest** definition; restore any GRANTs.
- Avoid the **add-enum-value-then-use-in-same-transaction** trap; prefer mapping via `::text` into the new enum.
- Note cross-PR ordering (migration PR before dependents).

### If a migration fails on apply
- Classify (step 2): infra → STOP, no blind-retry; SQL defect → fix.
- For an SQL defect: confirm clean rollback — `supabase migration list --linked` should still show it *pending* (Supabase wraps each migration in a transaction). If partially applied → STOP and report.
- **Fixing it in place is correct** even though the standing rule (§"Hard rules") is "never modify an applied migration": that rule protects *applied* migrations. A never-applied migration that exists only on the unmerged PR branch must be fixed in place — commit the fix to the PR branch, re-apply, re-verify.

## 4.5. Structured PR review (size-tiered)

Classify the PR's review tier from `gh pr view --json additions,deletions,files` (already fetched in step 2). The tier decides spawn-vs-skip and which model runs it. **`<review_override>` from §0 wins over the heuristic:** `--full-review` forces Tier C; `--skip-review` forces Tier A. **Route `fast` also forces Tier A:** when the ticket's route is `fast` (read from the in-session hand-off, else the `Route:` line in the ticket's comments — one scoped `list_comments`; no record → default `build`, no force), classify Tier A regardless of the size heuristic — the same mechanism as `--skip-review`, with `<review_override>` still winning over the route. This touches only this size-tiered general review; §4.6's security gate and its `sensitive-diff-scan` floor are diff-driven and route-invariant — a fast ticket that unexpectedly touches a sensitive surface still escalates there.

**Tier A — trivial (no subagent).** ALL of these hold:
- `additions + deletions ≤ 30`
- `files.length ≤ 3`
- No changed file path matches the sensitive list under Tier C.

Action: print one line — `Auto-skipped review: trivial PR (N lines, M files). Pass --full-review to force.` Parent eyeballs `gh pr diff` (already in hand from step 2) and notes any obvious red flag inline (one or two sentences max). No subagent spawn. Skips straight to step 5.

**Tier C — sensitive (full Opus subagent).** ANY of these hold:
- Changed file path matches: `supabase/migrations/*`, `**/auth/**`, `**/middleware.*`, `**/api/**`, `**/{security,crypto,authz,acl,rbac}*`, or any file listed under `cfg.docs.systemDocs[].covers` for schema/migration docs.
- `additions + deletions > 200`.
- `files.length > 10` (broad blast radius even if line count is low).
- `<review_override> = --full-review`.

Action: spawn `Agent` with `subagent_type: general-purpose` (Opus default). Use the prompt template below.

**Tier B — medium (Haiku-routed subagent).** Everything not Tier A or Tier C.

Action: spawn `Agent` with `subagent_type: general-purpose` **and `model: "haiku"`**. Same prompt template. Haiku is fine for read-and-summarize review work; reserve Opus for the sensitive/large tier. (Per `~/.claude/memory/feedback_subagent_haiku_routing.md`.)

Subagent prompt template (Tier B and Tier C only — substitute `<PR#>`):

> Execute `~/.claude/commands/review-pr.md` literally on PR `<PR#>`. You are being invoked from `/land-ticket`, so:
> - Skip the `--comment` flow entirely — print-only.
> - Skip the "no PR found → suggest /land-ticket" branch — the PR exists.
> - Return ONLY the structured review sections from §2 of `review-pr.md` (Summary, Acceptance vs diff, Migration & schema safety, Correctness, Boundary/convention violations, Security, Style & clarity, Coverage). Omit sections that have nothing to say.
> - Do NOT return raw diff, file contents, exploration narration, or "I read X then Y" commentary — only the final review.
> - Every finding must cite real `file:line` per the skill's hard rules.

Parent-side handling (Tier B and Tier C):
- Print the review verbatim to the user (it's already terse and structured).
- Any `critical` or `high` finding → carries into step 5 as a **`needs-eyes`** item requiring specific user acknowledgment at the gate (same treatment as a `needs-eyes` conflict resolution).
- If the review surfaces a new issue prior steps missed (logic bug, missing acceptance item, boundary violation) — treat as `needs-eyes`. The user can deny the gate, fix the issue, push, and resume `/land-ticket`.
- Never auto-post `--comment` to Linear — that's the user's call (`/review-pr <PR#> --comment` separately).
- Multiple PRs in step 1 → spawn the review subagents in parallel (single message, multiple Agent tool calls) unless one PR's review needs to inform another's. Tier each PR independently.

## 4.6 Security review gate (judgment over a deterministic floor)
A focused security pass that gates the merge on a diff's security risk and costs ~0 on ordinary diffs. Distinct from §4.5's general review: §4.5 is advisory and size-tiered; this gate is security-specialized and **merge-blocking**. Conceptually it sits "after tests, before merge" — tests (`/verify-tests`) ran back in `/build`; this gate's findings feed the §5 confirm gate (MEDIUM/LOW) and the §6.7 merge pause (HIGH). It **reuses** `/security-review` — it never re-implements security analysis here.

**The gate is a judgment, not a keyword match.** It asks *"should this change be reviewed, and if so how deeply"* — reasoned against explicit, craft-referenced criteria — built on top of a deterministic detection **floor**, not in place of it. Load `~/.claude/craft/gating.md` (the gate-as-craft rail) and bring its stance: `bin/sensitive-diff-scan` is the floor — it catches sensitive surfaces by keyword and rates the sandbox/permission-surface families `HIGH`; that floor is **load-bearing and non-negotiable**, and the judgment layer on top is strictly **escalate-only** (it may raise scrutiny the floor didn't ask for; it never clears or downgrades a floor `HIGH`).

**Resolve whether the gate runs (cheap, before any spawn).** `<security_override>` from §0 wins over detection:
- `force-off` (`--no-security-review`) → **skip**, print `🔒 Security review: skipped (forced off via --no-security-review)`, carry no findings. Done.
- `force-on` (`--security-review`) → run the review regardless of detection (skip to "Run the review" below).
- unset → **auto-detect** from the diff via `bin/sensitive-diff-scan` (the single source of the path + content patterns — **never re-inline the regex here**; `bin/sensitive-diff-scan.test.sh` is the probe that proves them):
  - **Cheap test-only skip first** (keeps non-sensitive lands at ~0 cost, no subagent): `gh pr diff <n> --name-only`. If the changed set is **only** tests (`*.test.*`, `*.spec.*`, anything under `__tests__/`) → **skip**, print `🔒 Security review: skipped (test-only diff)`, carry no findings. Done. **Markdown is no longer auto-skipped here:** in this repo a skill / `CLAUDE.md` / any-`.md` edit is executable behavior, so a markdown-only diff routes to the scan below — `bin/sensitive-diff-scan` scans its content for the HIGH families only. Because the scanner is ~0 cost and the review subagent spawns only on a hit, a genuine doc carrying none of the HIGH markers still incurs no spawn; only markdown that actually adds a sandbox-disabling marker escalates.
  - **Scan otherwise:** `gh pr diff <n> | bin/sensitive-diff-scan`. It runs both passes — PATH (changed-file globs: the legacy §4.5-Tier-C set — `**/auth/**`, `**/middleware.*`, `**/api/**`, `supabase/migrations/*`, `**/{security,crypto,authz,acl,rbac}*`, `**/*{secret,token,credential,password,session,login}*`, `**/.env*`, `*serial*`/`*deserialize*`/`*.sql`/`*query*`) and **file-aware CONTENT** (added/removed lines; test files never a content hit, and markdown is scanned for the HIGH families only) — over the legacy sensitive tokens **plus** three sandbox-disabling families it rates **HIGH**: permission-bypass markers (`dangerously-skip-permissions`, `bypassPermissions`, `skip-permissions`), host-scheduler / persistence installs (`crontab`, `launchctl`, `launchd`, `systemctl`, `*.plist` under `LaunchAgents`/`LaunchDaemons`), and `settings*.json` permission-surface edits (`permissions`/`allow`/`deny`/`bgIsolation`). Exit 0 = sensitive (verdict `HIGH …` or `SENSITIVE …` on stdout); exit 1 = `non-sensitive`.
  - **Verdict `HIGH …`** → the diff disables or persists past the permission sandbox itself. **Record a structural HIGH finding** (`HIGH · <scanner reason> · sandbox/permission-surface change`) that feeds the §6.7 merge hard-block exactly like a review-found HIGH. Then still **run the review** (below) for full analysis. To locate **which** added line tripped the floor, run `gh pr diff <n> | bin/sensitive-diff-scan --locate` — it emits one `file:line: <SEVERITY> · <reason>` per hit off the **same** single-source marker patterns; **never re-inline the marker regex in an ad-hoc shell grep** (V-345 — that duplicates the patterns outside their one source and is an unallowlisted, ugrep-prone call).
  - **Verdict `SENSITIVE …`** → sensitive legacy surface; **run the review** (below) — severity of any finding comes from the subagent.
  - **Exit 1 (`non-sensitive`)** → the floor found no sensitive surface, but do **not** skip yet — carry the verdict to the gate's judgment step below, which may escalate it on a criterion the keyword scan can't see. (The cheap test-only skip above already terminated the ~0-cost path with no spawn for a test-only diff — a markdown-only diff now reaches this content scan; this content-scan `non-sensitive` is the verdict judgment reasons over.)

**Pose the gate's judgment — should we look harder, and how deeply (the craft rail).** The scanner answered *does this touch a sensitive surface by keyword* — the floor, not the whole question. Bring `craft/gating.md`'s stance: the gate is a judgment, reasoned against named criteria, **escalate-only above the floor**:
- **The floor is non-negotiable.** A scanner `HIGH` already recorded its structural hard-block (above) and reaches §6.7 regardless; a `SENSITIVE`/`force-on` verdict already routes to the review. Judgment **adds** to the floor — it never clears, downgrades, or skips a floor `HIGH`. (`craft/gating.md` "the asymmetry that makes this safe": the most dangerous diffs argue best for their own safety, so no judgment verdict may lower the floor.)
- **Escalate a `non-sensitive` scan when a criterion trips.** Ask `craft/gating.md`'s criteria of the diff even at exit 1: does it alter a **trust/permission boundary** (an authz check, role, RLS policy, capability grant)? wire a **new external input or sink** (untrusted input into a query, a new live-system write, a new deserialization)? widen **blast radius past the ticket's acceptance**? lean on or change a **security baseline** another ticket trusts (probe it, don't assume)? If one trips → **escalate to the review** ("Run the review" below) and name the criterion that drove it; the subagent assigns severity as usual.
- **Shape the depth.** When the review runs — escalated, or on a `SENSITIVE`/`force-on` hit — point it at the seam that moved: a boundary or input change earns a focused pass on that surface, not a generic full-diff sweep (`craft/gating.md` "how deep, once you're looking").
- **A clean skip is a stated decision, not a silent fall-through.** Nothing tripped → print `🔒 Security review: skipped (non-sensitive diff; judged against gating criteria, none tripped)`, carry no findings. Done — a considered pass.

**Run the review (on a sensitive hit, `force-on`, or a judgment escalation).** Spawn an `Agent` subagent (`subagent_type: general-purpose`, Opus — this is the sensitive tier, mirroring §4.5 Tier C) with the prompt below. It runs **in the PR's head worktree** (recorded in §2) so `/security-review`'s "pending changes on the current branch" scope resolves to the real `origin/<baseBranch>...<head>` diff, and it **reuses** the skill rather than duplicating its logic. **First, guard `origin/HEAD`:** before spawning, run `git -C "<head-worktree-path>" remote set-head origin <baseBranch>` (idempotent, no network — points `refs/remotes/origin/HEAD` at the already-fetched `origin/<baseBranch>`). The built-in `/security-review` computes its changed-file scope with `git diff … origin/HEAD…`, and `origin/HEAD` is a per-repo symref **not** created by a plain clone/fetch — absent it, the skill aborts with `fatal: ambiguous argument 'origin/HEAD...'` and can't determine its scope. Setting it first makes the skill's diff resolve against the configured base. The spawn prompt:

> You are running the security gate for `/land-ticket` on PR `<PR#>`, head branch `<head-branch>` in worktree `<head-worktree-path>`. Work from that worktree.
> 1. **Ensure `origin/HEAD` resolves first** — run `git remote set-head origin <baseBranch>` in the worktree (idempotent; the orchestrator already did this — this self-heal is belt-and-suspenders so the skill's internal `git diff … origin/HEAD…` cannot hit `fatal: ambiguous argument 'origin/HEAD...'` when `origin/HEAD` is unset). Then invoke the built-in `/security-review` skill (via the Skill tool) to analyze this branch's diff against `<baseBranch>`. Do NOT re-implement security checks — run the skill and read its findings.
> 2. Normalize every finding it reports into exactly one severity using this rubric (assign it yourself — `/security-review`'s own output format is not a guaranteed contract):
>    - **HIGH** — exploitable now: injection (SQL/command/path), authn/authz bypass, secret/credential exposure, broken crypto, unsafe deserialization of untrusted input, SSRF.
>    - **MEDIUM** — real weakness needing context to exploit: missing validation with limited blast radius, weak-but-not-broken crypto, permissive CORS, info leak.
>    - **LOW** — hardening/defense-in-depth: minor info disclosure, missing security header, style-level concern.
> 3. Return ONLY a findings list, one per line as `SEVERITY · file:line · one-line description`, each `file:line` real (from the skill's output / the actual diff — never invented). If the skill reports nothing, return the single line `NONE`. No narration, no raw diff.

**Route the returned findings:**
- **HIGH** → record as a **merge hard-block** carried to §6.7 (the merge cannot proceed while an unresolved HIGH stands — see §6.7). Also list them at §5 so the user sees them at the first gate.
- **MEDIUM / LOW** → record as **§5 acknowledgement items** — the user must explicitly acknowledge them at the §5 confirm gate to proceed (they do not hard-block the merge, but proceeding past them is a conscious yes, not silent).
- **NONE** → print `🔒 Security review: ran, no findings.` and carry nothing forward.
- Record the gate outcome (triggered/skipped, override applied if any, counts by severity) for the §9 report.

**Worked example — judgment escalates, never lowers the floor.** A diff adds a host-scheduler install line to a setup script. The scanner rates it `HIGH` (host-scheduler family); §4.6 records the structural HIGH that carries to §6.7 as a merge hard-block. Even if the gate's judgment finds a reasonable-sounding case that the line is benign, it does not matter — the rail is escalate-only, so the HIGH stands and the merge stays blocked until the finding is fixed or the human explicitly waives it at §6.7. Conversely, a diff that threads a new request parameter into an existing SQL builder trips no sensitive *filename* and may scan `non-sensitive` — judgment escalates it to a review on the "new external input/sink" criterion the keyword floor alone would have missed. The floor catches what tokens name; judgment catches what they don't; neither can talk the other's `HIGH` away. (This is the encoded proof of acceptance #3's invariant: the HIGH path is judgment-independent — see `bin/sensitive-diff-scan.test.sh`, which proves the floor rates all five sandbox/permission-surface families `HIGH`.)

## 4.7 Docs ripple scan (convention 6 — opt-in via `cfg.docs` block)
- If `cfg.docs` is absent → skip this phase entirely.
- If `cfg.docs.maintenance` is `"daily"` (V-284) → skip this phase entirely with the one-line note `docs maintenance deferred to daily /docs-refresh (cfg.docs.maintenance: daily)` — the daily pass reviews the day's merged changes and applies all doc updates in one consolidated PR; per-land doc work is off. (Default / absent `maintenance` = `"per-land"` — everything below runs unchanged.)
- Read `gh pr diff <n> --name-only` (already have from step 2).
- **Compute candidates (path overlap).** For each `cfg.docs.systemDocs[]` entry: if any PR-changed file path starts with any of its `covers` prefixes AND the entry's `doc` itself is NOT in the changed files → this doc is a *candidate*. Path overlap alone is a weak signal — it over-fires on docs that are perfectly fine to leave — so candidates are NOT findings yet; classify each in the next step.
- **Classify each candidate by reading it.** Read the candidate doc and the relevant PR diff, and judge whether the change makes a **specific claim in the doc now false** (a section describing the changed behavior, a code reference the change moved/renamed, an invariant the change broke):
  - **Genuinely stale** (the change contradicts a claim the doc makes) → emit `docs-stale: <doc>` carrying the now-false claim + what the change did. This is the **only** docs finding that gates (§5, as a single `update? y/n`).
  - **Low-value** (the covered path was touched but no claim in the doc is contradicted — the common case) → emit `docs-overlap-noted: <doc>`, **logged only** in the §9 report. It does **not** gate; the user is never asked to "acknowledge" a doc that's fine to leave.
- `docs-header-missing` (a `cfg.docs.systemDocs[].doc` whose line 1 doesn't match `> Last verified against code: YYYY-MM-DD (PR #...)`) → **logged only** in §9, never gated: a missing freshness header is never a false claim.
- **Routing:** only `docs-stale` carries into §5 (as one `update this stale doc? y/n` decision). `docs-overlap-noted` and `docs-header-missing` are logged in the §9 report — never gated, never an "acknowledge" prompt. A `docs-stale` doc the user declines to update now → logged to `cfg.docs.postponed` in §6.5.

## 4.7b Doc-accuracy scan (pipeline-docs accuracy guard)
A deterministic backstop for the factual-accuracy class that `/review-pr` and the rest of this gate don't assert: **miscited convention numbers, wrong on-disk counts, broken `~/.claude/…` or relative links** (the heavyweight `/review-produced` lens is the sibling concern). Independent of `cfg.docs` (the accuracy class isn't tied to the docs-lifecycle opt-in), and a no-op for any repo that isn't the `~/.claude` pipeline source (the checker scans `~/.claude`'s own doc tree, root-resolved from its `bin/` location).

- **Skip silently unless** `gh pr diff <n> --name-only` (in hand from §2) includes a file under `commands/`, `craft/`, `pipeline/`, or one of `workflow-conventions.md` / `workflow-chains.md`. No such file → this step does nothing.
- Otherwise run the checker scoped to the changed docs: `node ~/.claude/bin/check-doc-accuracy.mjs --paths <those changed doc paths>`. One allowlisted call (`Bash(node ~/.claude/bin/*.mjs)`); read its stdout in-context (do not pipe to `grep`/`jq`).
- **Exit 0** → nothing to surface. **Exit 1** → record each printed finding as a `doc-accuracy-risk` item carried to §5. **Exit 2** (bad args / no docs) → log the error in §9, don't gate.
- **Routing — advisory, never a hard-block** (sibling to `docs-overlap-noted`): `doc-accuracy-risk` items are listed at §5 for an explicit one-line acknowledgement. They do not block the merge — a deliberately-illustrative count or a link target not yet created is a legitimate reason to proceed — but proceeding is a conscious yes, not silent.

## 4.8 Acceptance↔artifact completeness gate (pre-merge)
The merge is irreversible; this is the last point to catch an Acceptance item whose real artifact never got built — a `migration` item ticked and merged with no migration, caught only after merge. Run this BEFORE the §5 confirm gate so its findings feed the gate. **This gate is diff-driven and route-invariant:** it fires off the actual PR diff regardless of the ticket's predicted route — a `route:fast` ticket that unexpectedly produces a migration hits the same hard STOP below.

- **Get each item's artifact-kind.** Read the scope plan's "Pre-build validation" section at `<worktree-root>/docs/plans/<ticket-id-lowercased>-build.md` (derive the path **lowercased** — same casing rule `/build` uses; `/scope` §6's both-casings guard keeps it unambiguous). No scope plan on the branch → classify each Acceptance item inline with the rule in `scope.md` §3 (`code` / `migration` / `dashboard-config` / `manual-verify` / `invariant`).
- **Get the merged diff's paths:** `gh pr diff <n> --name-only` (already in hand from §2).
- **Gate per kind:**
  - `migration` — diff has **no** `supabase/migrations/*` file → **hard STOP, do not proceed to §5**: `needs input: Acceptance item N is classified migration but the PR diff has no supabase/migrations/* file — the mechanism it promises was not built. Add the migration to the PR branch, or (if the item genuinely needs no migration) re-classify it in the scope plan before landing.`
  - `invariant` — the item asserts a negative/security property (keyword-gated per `scope.md` §3). **Diff-presence of enforcement code never satisfies it.** Verify the diff contains an **encoded regression test/probe that asserts the negative** (in a test-less repo, a committed probe script or worked example demonstrating it). No such proof in the diff → **carry into §5 as `needs-eyes`, never silently ticked** — the user explicitly decides at the gate: resolve (add the test) or **deliberately defer it to a follow-up**. A deliberate defer of a *genuinely-owed* verification is **routed into §4.9's DEFERRED VERIFICATION block** (the owed negative property + why it can't be proven at land), which §6.7 **auto-files as a `check-later:` deferred-verification ticket** into the deferred-checks bucket (V-319); §8 then names **that ticket ID**, not a bare prose note. (Does **not** hard-STOP like `migration` — an invariant item is a surfaced decision, not a build-incompleteness block.)
  - `dashboard-config` / `manual-verify` — no artifact can land in the diff, so these can't be auto-verified. Feed each into §4.9's verdict and collect its acknowledgment at the **§6.7 manual-test pause** (after db push, so the user tests against the real pushed state) — never auto-passed, and not acked at §5 (where db push hasn't run yet).
  - `code` — verify the file(s) the item implies are in the diff (the §8 acceptance walk, pulled ahead of the merge). Missing → carry into §5 as `needs-eyes`.
- Only the `migration`-with-no-migration case hard-STOPs here; every other mismatch (including an unproven `invariant`) surfaces at §5 for an explicit decision, and a `code` item already covered by the diff passes silently (no new gate for the common case).

## 4.9 Manual-test verdict — partition by who-can-do-it, then run the session's half
Compute an explicit verdict so the §6.7 pause states *exactly* what needs testing — never a silent "approve when ready," and never the whole check dumped on the human behind a blanket `p`. The user values this gate but wants the agent to do everything it can itself, so the verdict is **split by who can run it**, and the session runs its own half *before* the pause.

**Step 1 — gather the manual-test needs.** Two sources:
- **Scope plan:** the `manual-verify` and `dashboard-config` items already read in §4.8 (`<worktree-root>/docs/plans/<ticket-id-lowercased>-build.md`, lowercased) — each is a manual-test need. **Plus any `invariant` item §4.8 carried as `needs-eyes` whose owed property the §5 gate deliberately defers** (V-319): its owed verification is a *check-later* need (routed to the DEFERRED VERIFICATION block below), not a check-now test — so invariant-kind owed verifications reach the deferred class, not only `manual-verify`/`dashboard-config`.
- **Diff surface area:** `gh pr diff <n> --name-only` → flag *added* user-facing entry points (a new `page.tsx`/route/screen, a new top-level form or auth flow). Be conservative — clear new entry points only, not every changed file, or the verdict loses signal.

**Step 2 — partition each need by who-can-do-it.** The boundary is **observability**, anchored to concrete predicates so the line can't slide into auto-passing genuine human residue:
- **Session-runnable** — anything observable *now* from the diff/tools/repo, needing no live human judgment or real-world state: a note/string/symbol is present in a file, a referenced path exists, a function/route/config key is wired, a committed example or probe behaves as claimed, a doc's covered claim matches the code. The session can check these itself.
- **Human-only residue** — what the session genuinely cannot observe. Partition it one level further by **when** it becomes checkable — this sub-split is what routes a post-deploy outcome to auto-monitoring instead of dead-blocking the gate on an un-verifiable-yet check (V-257):
  - **Check-now residue** — a human *can* observe it at the §6.7 pause, against the already-pushed DB / deploy-preview, *before* the merge: taste/wording/copy-fit judgment, and any *current* real-world/UI/runtime behavior (a redirect lands in a browser, a toast renders, an OAuth consent screen, a dashboard toggle's live effect). This is genuine pre-merge human testing — it **blocks** the merge (unchanged). **Sub-classify check-now residue by whether judging it needs a rendered surface (V-318):**
    - **visual-eyeball** — the item can only be judged by *looking at a rendered UI*: a component/chips/list renders, a toast shows, a redirect lands, a layout reads right. These route to **Step 3b**, which spins the surface up **by default** and — wherever a browser tool can assert or screenshot the thing — discharges them itself (turning would-be trust into observed proof, convention 8); only genuinely-subjective visual judgment a tool can't assert (taste / aesthetics / copy-fit) stays human residue, and even then the localhost is already running with its URL handed over at §6.7.
    - **non-visual check-now** — observable without rendering the app (a dashboard toggle's live effect, a value in an already-visible surface). Handled as before; no dev server needed.
  - **Check-later residue (deferred verification) — genuinely-owed, but not runnable at the pause and not passive monitoring (V-319).** An owed acceptance-item verification that a human **cannot** run at the §6.7 pause (unlike check-now) *and* that is **not** a passive post-deploy outcome (unlike monitoring): it needs an **active verification run or setup inappropriate to perform synchronously at the land gate** — an end-to-end runtime read-back that would have to create a throwaway resource, exercise a full flow, or stand up state the gate shouldn't. It does **not** block the merge (a human can't discharge it at the pause either, so blocking would only extract a rubber-stamp — the same logic as post-deploy-only); §6.7 **auto-files it as a `check-later:` deferred-verification spin-out** (convention 9; V-319). **Two feeders land here:** an owed `manual-verify`/`invariant` need partitioned at this step, and a §4.8 `invariant`-item deliberate-defer routed in from §5. **Owed vs nice-to-have is the gate (Acceptance #3):** a *genuinely-owed* verification (an acceptance criterion's own check) is **minted**; a *merely-nice-to-have* extra is **dropped as today**. Default an ambiguous *owed-vs-nice* call to **owed** — over-file, never silently drop an owed verification (the *opposite* direction from the check-now over-block default below: here losing owed work is the worse failure).
  - **Post-deploy-only residue** — an outcome observable **only after** the change is merged/deployed and time passes: *does the recurrence stop*, *does the fix hold in production over the next N runs*. Nobody — human or agent — can check it *before* the merge, so blocking the gate on it just asks the human to rubber-stamp an un-verifiable-yet outcome (the deficit V-257 removes). It does **not** block; §6.7 auto-files it as a verify-the-fix / monitoring spin-out (convention 9). **Default an ambiguous *observability* item to check-now** — never route to auto-monitoring (or check-later) something a human could actually verify pre-merge (the safe direction: over-block, never under-verify).
- When a need has **both** an observable and a live part, **split it** — the observable part is session-runnable, the live part is residue (then classified check-now / check-later / post-deploy-only per the sub-split above). Default an ambiguous item to **residue** (never auto-pass something that might need a human) and state why in one line.

**Step 3 — the session runs its half now.** Execute every session-runnable item (Read/Grep/Bash against the worktree + merged diff) and record the **observed result** per item — pass/fail + the evidence (`file:line`, the matched string, the command output), per convention 8 (observed state, never asserted). A session-runnable item that *fails* is a build/landing defect surfaced here, not deferred to the human.

**Step 3b — auto-discharge visual-eyeball residue via the running surface (V-318).** Any check-now item classified *visual-eyeball* in Step 2 gets the session's own attempt at observed proof *before* the pause — the human should never be the first line of verification for something a tool can render and assert:
- **Spin up the surface by default — on a port this worktree owns.** If the repo declares a dev server (a `dev`/`start` script in `package.json`, a documented run command, or a project `/run` skill), **start it now without waiting to be asked** — the whole point of this gate is that if there is something to eyeball, the localhost is already running. Inherit env via the convention-5 `.envrc` symlink the worktree already carries. Resolve `WT_ABS="$(git rev-parse --show-toplevel)"` and the run command's preferred port (`3000` when it declares none), then select the actual port with `port="$(bash ~/.claude/bin/dev-server-port.sh select "$WT_ABS" "$preferred_port")"`; a listener rooted in another worktree makes the helper choose a deterministic distinct port instead of silently reusing it. Start the repo's server on `$port`, wait for readiness, then run `serving_cwd="$(bash ~/.claude/bin/dev-server-port.sh verify "$WT_ABS" "$port")"`. **Only a successful listener-cwd verification licenses a `running` URL or browser navigation.** A failed verification is an observed startup defect: surface the helper's owner-cwd output, do not claim the change is live, and do not send the human to that URL. Capture both the verified URL and `$serving_cwd` for the verdict/handoff below.
- **Assert or capture each visual item.** Drive a browser tool — the claude-in-chrome MCP (`navigate` → `find` / `read_page` / `get_page_text` to assert the named element/chips/text rendered; `computer` / `gif_creator` for a screenshot) or the repo's Playwright harness (the V-231 network/UI-update sibling) — and record the **observed result** per item (assertion passed + the matched element, or the screenshot path), per convention 8. An item the tool **asserts positively → promote it to SESSION-VERIFIED** with that evidence: it is no longer human residue, because it is now *observed*, not trusted.
- **What stays residue.** Only what a tool genuinely cannot judge — subjective taste / aesthetics / copy-fit — stays in HUMAN TESTING REQUIRED as a **`(VISUAL)`** item, carrying the localhost URL + what-to-check + its per-item discharge token (§6.7). A tool assertion that **fails** is a landing defect surfaced here (like a failed session-runnable item in Step 3), not silently downgraded to "the human will look."
- **Degrade cleanly — never block on tooling.** No dev server in the repo (this `~/.claude` repo has none — no `package.json`) → no-op with the note `no dev server in this repo — visual residue stays human-eyeball`; a browser tool unavailable or undrivable in this session (e.g. an unattended `/go` with no attached browser) → note it and leave the visual items as human residue with their tokens. The auto-discharge is best-effort proof, never a new hard dependency.

**Step 4 — emit the verdict** (consumed at §6.7):
- **SESSION-VERIFIED:** the session-runnable items with their observed results — **including any visual-eyeball item Step 3b asserted / screenshotted**, each with its browser-assertion result or screenshot path as the evidence — or *none*.
- **LOCALHOST:** the dev-server URL Step 3b started **by default**, including both the selected port and verified serving cwd (e.g. `http://localhost:3472/<route> — running · served by <worktree>/web`), when there is visual residue to eyeball, so §6.7 hands it over without the human asking — or `ownership unverified — no localhost URL` when startup did not pass the listener-cwd check, or `no dev server in this repo` / `no visual residue` when Step 3b started none.
- **HUMAN TESTING REQUIRED:** the **check-now** human residue, each with what to check — **visual items marked `(VISUAL)` and carrying the localhost URL + the per-item discharge token** (§6.7) — or **`No human testing required.`** when it is empty. This is the block the §6.7 pause **blocks** on.
- **POST-DEPLOY MONITORING:** the **post-deploy-only** residue, each phrased as a concrete post-deploy check (`no recurrence of <X> over the next N runs`, `<metric> holds after deploy`) — or **`No post-deploy monitoring needed.`** when empty. This does **not** block the merge; §6.7 auto-files it as a verify-the-fix ticket **after** the merge (V-257).
- **DEFERRED VERIFICATION:** the **check-later** residue — each item a concrete owed verification phrased as *what to verify* + *why it couldn't run at land* + its origin (the specific acceptance item), or **`No deferred verification owed.`** when empty. This does **not** block the merge; §6.7 auto-files it as a `check-later:` deferred-verification ticket **after** the merge, into the deferred-checks bucket (V-319). (A merely-nice-to-have check is **not** listed here — it is dropped, per Step 2's owed-vs-nice gate.)

The **residue** (not the whole check) is presented at the §6.7 manual-test pause, after db push so the human tests against the real pushed state; that is where the residual `manual-verify`/`dashboard-config` acknowledgments are collected. Only the **HUMAN TESTING REQUIRED** (check-now) block gates there; the **POST-DEPLOY MONITORING** and **DEFERRED VERIFICATION** blocks are shown for transparency (the human can redirect them) but are discharged by the §6.7 auto-file, not by a human report.

## 4.10 Land-gate rendering — droid-details-on-top, decision-at-the-bottom (route-aware — V-233)
§5 + §6.7 are the two gates a human lives in every land, so they render **for the human's decision, not the agent's audit**. Two rules, both load-bearing; this changes gate **layout/content only** — never the convention-11 *mechanism* (the `needs input:` prose stop, the `<TICKET-ID>: <goal>` + `changed:` restatement, the §6.7 security-HIGH / manual-residue ack rules) — and it adds/deletes **no** gate.

**Resolve `<route>` once** — the read §4.5 already does (in-session hand-off when `/next-ticket`/`/go` ran here, else the `Route:` comment line via one scoped `list_comments`, else `build`). Reuse §4.5's value — don't re-call.

**The two-part gate body (used by §5 and §6.7, every route).** Order is the point: the human reads top-down and acts at the bottom.

*Top — Droid details (verified; skimmable).* The receipts this flow already computed — PR(s)/diff-shape/mergeable/checks (§2), the §4 DB-change summary, §4.5 review tier + verdict, §4.6 security-floor result, §3.5 thesis-check verdict (from the plan), §4.8 acceptance↔artifact, conflict report (§3). **Lead any genuinely-blocking item loudly** (a `⚠️ DB CHANGES` push, a `needs-eyes`, a security HIGH) — those still gate exactly as before; everything green is **one skimmable line, not a per-tier celebration**. The existing "every `needs-eyes` item needs specific acknowledgment" rule (below) lives here — surfaced in the Top, gated as ever.

*Bottom — What this is (the payload the human acts on; diff-reconciled, convention 8).* Four fields, stating what **landed**, not what was intended:
- **Built** — what this land actually ships, 1–2 sentences.
- **How & why** — the approach and *why this shape*, brief.
- **Reached past?** — the pre-land craft reflection (`craft/building.md`): *did we build the best version of the goal, or just satisfy the literal plan?* Answer it **with evidence, in one line** — name what the build reached past the literal Acceptance to do (an in-scope-cheap improvement folded in rather than spun out), or state `satisfied the plan; nothing more was in-goal-and-cheap`. This is a load-bearing reflection, not a banner: an unanswered question is Ceremony (`craft/judgment.md`), so it is *answered*, surfacing under-reach (a hollow-letter build) before the irreversible merge. **Route-gated to avoid bloat** (the §4.10 collapse rule): rendered on the **standard / foreground** Bottom; on a **collapsed** fast land it folds to a single trailing clause on the one-line Built (`…; nothing reached past`) or is omitted when trivially satisfied — never a separate line there.
- **To check** — **only** what a human must verify *before* the merge: the §4.9 **HUMAN TESTING REQUIRED** (check-now) residue (taste/wording, current real-world/UI/runtime). Everything diff-observable was already checked and is **not** relisted — **including any visual item Step 3b already asserted/screenshotted, now SESSION-VERIFIED, not here**; **post-deploy-only** outcomes are **not** here either — they ride the separate **Monitoring** line below, auto-filed rather than human-checked (V-257). A **`(VISUAL)`** item carries the **localhost URL Step 3b started by default** (already running — the human clicks it, no separate request) and its per-item discharge token (§6.7). Empty residue → `Nothing — all of it was diff-observable and checked.`
- **Monitoring** — the §4.9 **POST-DEPLOY MONITORING** block: post-deploy-only outcomes (no recurrence, fix-holds-in-prod) that §6.7 auto-files as a verify-the-fix ticket **after** the merge, so the gate is not blocked on an un-verifiable-yet check. Empty → omit this line entirely. This is shown so the human can redirect it (amend the checks, or reply to skip the auto-file); it never gates.
- **Deferred** — the §4.9 **DEFERRED VERIFICATION** block: genuinely-owed check-later verifications that §6.7 auto-files as a `check-later:` ticket **after** the merge (into the deferred-checks bucket), so the gate is not blocked on a verification that can't run at land. Empty → omit this line entirely. Shown so the human can redirect it (amend, or reply to skip the auto-file); it never gates. (Sibling of **Monitoring** — both are deferred-past-land checks: **Deferred** = an owed *active* verification, **Monitoring** = a passive *post-deploy* outcome.)
- **Deploy** — a heads-up (V-343) when the merged diff touches a `cfg.deployedServices[].covers` path: `<name> — merged fix goes live only after redeploy (<one-line recipe>)`, so the approver of the irreversible merge knows a live service will still be serving stale code until it is shipped. Empty (no deployed-service touch, or `cfg.deployedServices` unset) → omit this line entirely. It does **not** gate here — the actual gate is the post-merge §6.7b redeploy ack; this line is the pre-merge transparency for it (sibling of **Monitoring**/**Deferred**, which likewise surface pre-merge and discharge post-merge).

  Source the brief by this **fallback ladder**, then reconcile against the merged diff the §4.5/§4.8 walk already read (so it never claims an un-landed Built):
  1. **Plan exists** (build; and fast — its plan is materialised by `/build §3`) → `Built` from the plan `## Goal`, `How & why` from `## Implementation design` (Approach) + `## Deviations`.
  2. **`research`** (no plan) → the **finding doc** this land ships: `Built` = its subject, `How & why` = its investigation approach (the same doc the foreground shape reads below).
  3. **`fix-bug`** / any no-plan land → the **ticket Goal + Linear description + the fix commit subjects**.

**Shape (route sets the bottom's depth + whether §5 collapses):**
- **`research` → foreground.** The deliverable *is* a decision, so the Bottom **leads with** the finding's decision substance, read **straight from the finding doc's structured Decision block** — its `## Question` · `## Candidate approaches` · `## Relevant standards` · `## Recommendation` (per-approach pros-cons) sections, the block `go.md` §5's research path is **required** to write (V-301). The finding doc is the `code`-kind doc under `docs/` in `gh pr diff <n> --name-only` (the one the ticket names when several land). Render it **concisely — the Decision block's substance, not the whole doc**: the question in a line, each candidate approach with its key pros-cons, the standards named, the recommendation called out. A pre-V-301 finding lacking the block → foreground whatever decision substance it carries and note it predates the contract; **never silently fall back to leading with the merge receipts** (the recurring failure this closes). Then `Built`/`How & why`/`To check` follow; the receipts stay in the skimmable Top.
- **`fast` AND collapse-eligible → collapsed.** Collapse-eligible iff every diff-driven input is empty: **no migration** (§4 = `No DB/migration changes in this PR.`), **no findings** (no §4.5 critical/high, §4.6 finding, §4.7 `docs-stale`, §4.7b `doc-accuracy-risk`, §4.8 `needs-eyes`), **empty §4.9 check-now residue, and an empty §4.9 DEFERRED VERIFICATION block** (a genuinely-owed check-later verification is significant enough to surface at a full gate, so it disqualifies collapse even though it never *blocks*; a non-empty POST-DEPLOY MONITORING block, a purely passive future outcome, does **not** disqualify — V-319). Then §5 issues no separate stop and §6.7 is the single gate — same two-part body, trivial Top, one-line Built. **Diff-driven, not route-trusting**: a `fast` land with a surprise migration/finding is **not** collapse-eligible → `standard` (the §5 db-push/findings approval still fires — route-invariant, like §4.8/§4.6).
- **else → standard.** `build` / `fix-bug` / non-empty `fast` / research-with-no-finding-doc → the two-part body, full Top, normal Bottom; §5 and §6.7 are two gates.

Record the resolved shape (`foreground` / `collapsed` / `standard`) for the §9 report and `/go`'s gate-audit (a `collapsed` land is one gate, not two).

## 5. CONFIRM GATE — approve *proceeding*, NOT the merge
**Render per §4.10 (Top: Droid details → Bottom: What this is → the stop):**
- **`collapsed`** (fast + every gate input empty) → **issue no separate §5 stop.** Print one line — `§5 collapsed: fast-route land, nothing to separately approve (no db push, no findings, empty residue) — the single merge confirm is at §6.7.` — and proceed to §6. The one human gate is §6.7.
- **`foreground`** / **`standard`** → present the gate as the §4.10 two-part body: **the receipts spec below populates the Top — Droid details** (blocking / `needs-eyes` items led loudly and gated exactly as written — the relayout removes no gating); then the **Bottom — What this is** brief (Built / How & why / Reached past? / To check, ladder-sourced + diff-reconciled per §4.10 — the `Reached past?` reflection answered with evidence so under-reach surfaces before the merge), with `foreground` leading the Bottom with the finding's Decision substance; then the convention-11 goal restatement + the `needs input:` proceed stop.

This gate authorizes running `db push` (§6) + all pre-merge work. **It is NOT merge approval** — the merge is a *second, separate* yes at the §6.7 manual-test pause, after the user has tested against the pushed DB. Never treat a §5 yes as authorization to merge; there is no auto-flow from `db push` to merge.

Present + wait for explicit approval: the **§4 DB-change summary** (loud if a migration will be pushed — this push hits prod); PR(s)/order/strategy; conflict report (highlight `needs-eyes`); **cross-sibling `git merge-tree` findings (§3, V-330) — each `sibling-conflict-risk` is listed as an advisory heads-up (a conflict with an *unmerged* sibling PR; the sibling may land first — not a hard block)**; migrations + when; **review findings — any `critical`/`high` is a `needs-eyes` requiring explicit ack**; **docs ripple findings (§4.7) — each genuinely-stale doc (`docs-stale`, a change-contradicted claim) is one `update this stale doc? y/n` decision; low-value path-overlap (`docs-overlap-noted`) and missing-header findings are logged, not gated — never an ask to acknowledge a doc that's fine to leave**; **doc-accuracy findings (§4.7b) — each `doc-accuracy-risk` (a miscited convention number / wrong count / broken doc link the checker flagged) is listed for one explicit acknowledgement to proceed; advisory, never a hard-block**; **acceptance↔artifact findings (§4.8) — any `code` item missing from the diff requires explicit ack; an `invariant` item with no encoded proof of its negative property in the diff is a `needs-eyes` requiring an explicit decision here (resolve by adding the test, or deliberately defer to a follow-up — a *genuinely-owed* deferral routes its verification into §4.9's **DEFERRED VERIFICATION** block, which §6.7 auto-files as a `check-later:` ticket named by ID in §8, V-319, never closed clean); a `migration` item missing its migration already hard-STOPped before this gate**; **security-review findings (§4.6) — list every finding by severity; each MEDIUM/LOW requires explicit acknowledgement to proceed, and any HIGH is surfaced here as a `needs-eyes` that will additionally hard-block the merge at §6.7 until resolved or waived**; cleanup + Linear plan. No step 6 without a clear yes; every `needs-eyes` item (from review OR conflicts OR §4.6 OR §4.8) needs specific acknowledgment, not a blanket approval. The `dashboard-config`/`manual-verify` acknowledgments are **deferred to the §6.7 pause** (db push hasn't run at §5, so they can't be honestly satisfied yet). **Render the gate per convention 11** — immediately preceded by a one-line restatement of the ticket's goal (`<TICKET-ID>: <goal>`, so the approver isn't deciding blind), then a single `needs input:` prose stop (the irreversible action + what proceeding does + how to redirect), never an `AskUserQuestion` modal: the bg-session state classifier reads message text not tool calls, and `/go` drives this same stop with its one-token `p`.

## 6. Execute (per PR, in order) — pre-merge work only
- Apply agreed resolutions with Edit (never delete a side wholesale).
- Verify merged tree: `npx tsc --noEmit` must pass (the `<requiredCheck>` gate); run `npm test` if warranted. Fail → STOP.
- Commit (never `--no-verify`); `git push origin <head-branch>` (never force).
- **Migration (if approved) — reconcile remote-ahead migrations first, then `sb-push --apply`.** §3's `git merge origin/<baseBranch>` ran at conflict-analysis time; the §5 gate dwell + a **parallel land** can push a *newer* migration to `<baseBranch>` in the window since. `sb-push --apply`'s dry-run compares the head worktree's local `supabase/migrations/` against the **remote** migration history, so a migration that landed on `<baseBranch>` after §3 but isn't in this branch's tree dead-ends the apply with `Remote migration versions not found in local migrations directory` (the apply-before-merge parallel-land window — `migrations/CLAUDE.md`). Close that window immediately before the apply, in the **head worktree**:
  - `git -C "<head worktree>" fetch origin <baseBranch>`. If `git -C "<head worktree>" rev-list --count <head-branch>..origin/<baseBranch>` is `0`, `<baseBranch>` hasn't advanced since §3 — nothing to reconcile, go straight to `sb-push --apply`.
  - Otherwise `<baseBranch>` is ahead → `git -C "<head worktree>" merge origin/<baseBranch>` to pull its migration file(s) into the local dir. **Clean** (fast-forward or auto-merged) → commit if the merge produced a merge commit, `git push origin <head-branch>` (never force), re-verify (`npx tsc --noEmit`), re-poll `<requiredCheck>` (a re-push re-triggers CI — §6.7's `wait-for-check.mjs` pattern). **Conflict** (a parallel land touched the same files) → resolve per §3 (read both sides, integrate or flag `needs-eyes`; freshness-header-only auto-resolves), commit, push, re-verify — never blind-`-X`/`reset --hard` a side.
  - Then `sb-push --apply` (wraps `supabase db push --linked`), re-verify. The dry-run now sees the remote's migration versions present locally and no longer dead-ends.
- **Do NOT merge yet.** Docs ripple work (§6.5) lands on the same PR head branch first so the merge commit carries it atomically. Merge happens in §6.7.

## 6.5 Pre-merge docs ripple (convention 6 — opt-in via `cfg.docs` block)
Runs **on the PR head branch, before merge**, in the PR's worktree. The session is isolated to this PR — writing to `<baseBranch>` directly is not available and would bypass review. Skip entirely if `cfg.docs` absent. **Skip entirely too if `cfg.docs.maintenance` is `"daily"` (V-284)** — print `docs maintenance deferred to daily /docs-refresh (cfg.docs.maintenance: daily)` and touch no docs: no changelog fragment, no freshness bump, no postponed prompt — the daily `/docs-refresh` pass owns all of it (one consolidated PR/day). §6.8's "ripple intended" recompute short-circuits in lockstep (see §6.8), so teardown never expects a ripple this mode doesn't write. (Default / absent `maintenance` = `"per-land"` — everything below runs unchanged.)

- **Changelog fragment** (if `cfg.docs.changelog` configured):
  - Write a **per-PR fragment file** — `<dir of cfg.docs.changelog>/changelog.d/<N>.md` (N = the PR number) — containing the single entry line: `- <PR title> ([#N](<PR url>)) — <ticket: ID or title>`. Use the **`Write` tool**: the path is unique to this PR (a brand-new file every land), so there is no `## Recent` anchor to Read first and no `old_string` to match. Create `changelog.d/` if absent (`Write` makes the parent dir).
  - **Do NOT append to `cfg.docs.changelog` (`## Recent`) directly.** A per-PR filename is unique, so two concurrent lands never touch the same file and GitHub's 3-way merge resolves them with zero conflict. The curation pass folds `changelog.d/*` into the changelog's `## Recent`/sections later and deletes the fragments — exactly the manual curation that already happened on `## Recent`, just sourced from fragments. This is the V-194 fix.
  - Mechanical, no prompt.
  - **Why a fragment file and not a shared-file append + `merge=union`:** the V-9 `<changelog-path> merge=union` git attribute resolves concurrent appends only in **local** git. GitHub computes PR mergeability with a plain 3-way merge that **ignores** `.gitattributes` merge drivers, so a shared-file append — top-of-`## Recent` *or* EOF — shows `mergeable: CONFLICTING` / `mergeStateStatus: DIRTY` server-side even when the local merge is clean, forcing the multi-minute manual re-sync race under parallel `/go` (V-194). A unique fragment file sidesteps PR mergeability entirely. `docs/v-194-changelog-conflict-repro.sh` proves it: shared append (prepend *and* EOF) → GitHub `CONFLICT`, fragment file → `CLEAN`. The `merge=union` attribute (ticket-flow-init §6) is retained only for the single-threaded curation merge; it is **not** the concurrency mechanism (it never covered GitHub-side mergeability).

- **Freshness-header update** (if any `cfg.docs.systemDocs[].doc` was in the PR diff):
  - For each such doc, **re-read line 1 of that doc in `<worktree>` first** (Read-before-edit Hard Rule below), then build the replace `old_string` from that fresh Read — never from memory.
  - Replace that line with: `> Last verified against code: <today YYYY-MM-DD> (PR #N)`.

- **Postponed prompt** (if PR body or Linear ticket description has `Deferred:` lines, OR §5 had a `docs-stale` doc the user declined to `update`):
  - Parse `Deferred: <what> — <why>` lines from PR body / Linear description; collect any §5 `docs-stale` doc the user chose not to update now. (Low-value `docs-overlap-noted` / `docs-header-missing` findings are report-only — they are never written here.)
  - Confirm the combined list with the user. On yes, append to `cfg.docs.postponed` under `## <today YYYY-MM-DD> — from PR #N`. Each entry: `- **<what>** — <why>`.

- **Gate before committing — distinguish a no-op ripple from a *failed* ripple. A total Edit failure must never read as "nothing to ripple."** First decide whether a ripple was **intended**: yes if ANY of — `cfg.docs.changelog` is configured, OR any `cfg.docs.systemDocs[].doc` was in the PR diff (freshness bump owed), OR there are `Deferred:`/declined-`docs-stale` items to log. (This `intended` definition is the §6.5-local superset of §6.8's — they must agree on the systemDocs-in-diff and docs-stale cases so the two gates never disagree.)
  - **Ripple intended** → after the edits, assert the working tree is actually dirty for the intended paths: `git -C "$WT_ABS" status --porcelain -- <changelog fragment path: <dir>/changelog.d/<N>.md> <each intended systemDoc> <postponed path>` must be non-empty for the paths that were supposed to change. (Read the porcelain output in-context — do not pipe to `grep`/`jq`.) Non-empty as expected → proceed to **Commit + push** below. **Empty when a change was owed → FAIL LOUD, STOP before §6.7** (the irreversible merge), do NOT commit, do NOT merge: `needs input: a docs ripple was intended (<which: changelog fragment | freshness bump for <doc> | postponed entry>) but the §6.5 edits produced no diff — the writes likely failed (the swallowed-Edit class). The PR worktree is live at <worktree>; Read the target file there, fix it, and re-run §6.5 before merging.` Do not skip-and-proceed on an intended-but-empty ripple — that is the exact silent-docs-less-merge this ticket closes.
  - **No ripple intended** (no changelog configured, no system-doc touched, no deferred items) → genuinely nothing to ripple → skip the commit, proceed to §6.7. (§6.8 remains the post-merge backstop for anything that still slips through.)
- **Commit + push** (reached only when the gate above confirmed a dirty intended path). All §6.5 edits ride in one commit on the PR head branch: `docs: ripple updates for #N`. Push with `git push origin <head-branch>` (never force). This re-triggers CI on the PR.

## 6.7 Final merge (the actual landing)
**Render the manual-test pause per §4.10 (Top: Droid details → Bottom: What this is → the merge stop):**
- **`standard`** → Top = the §4 DB-summary + §4.9 **SESSION-VERIFIED** block (what the agent already ran); Bottom = **Built / How & why / To check / Monitoring / Deferred** (`To check` = the §4.9 **HUMAN TESTING REQUIRED** check-now residue, the only thing the human must run; `Monitoring` = the §4.9 **POST-DEPLOY MONITORING** block auto-filed after merge — omitted when empty, V-257; `Deferred` = the §4.9 **DEFERRED VERIFICATION** block auto-filed as a `check-later:` ticket after merge — omitted when empty, V-319; `Deploy` = the V-343 deployed-service heads-up — omitted when the diff touches no `cfg.deployedServices[].covers` path, and the post-merge §6.7b ack is where it actually gates); then the convention-11 restatement (`<TICKET-ID>: <goal>` + `changed:`) + the `needs input:` merge stop. The security-HIGH + manual-residue ack rules below are unchanged.
- **`foreground`** (research) → the **Bottom leads with the finding's Decision substance**, read from the finding doc's `## Question` · `## Candidate approaches` · `## Relevant standards` · `## Recommendation` block (§4.10), **concisely — the decision, not the whole doc**; then Built / How & why / To check; the §4 DB-summary + §4.9 SESSION-VERIFIED sit in the **Top**. Then the restatement + merge stop.
- **`collapsed`** (fast + empty — §5 was collapsed) → **this pause IS the single land gate**, same two-part body: trivial Top (`No DB/migration changes in this PR.` + the §4.9 SESSION-VERIFIED block), one-line Built, `To check` = `Nothing — all of it was diff-observable and checked.`; then the restatement + the one-line merge stop. (`p` is the normal one-token approval — no migration, no HIGH, no residue stands.)

- **Wait for `<requiredCheck>` green** via the helper:
  ```bash
  node ~/.claude/bin/wait-for-check.mjs --pr <n> --check <requiredCheck>
  ```
  Exit codes: `0` green → proceed; `1` failure → classify per §2 (STOP); `2` timeout (default 30 min) → STOP and surface; `3` bad args / gh error → STOP. **One Bash call** — never wrap in `until`/`while` or pipe to `python3`/`jq`. See top-of-file Bash discipline.
- **MANUAL-TEST PAUSE — mandatory hard stop between `db push` and merge (the second, merge-specific gate).** By now `sb-push --apply` (§6) has already pushed any migration to **prod** and the PR head + docs ripple (under `"per-land"`; none exists under `maintenance: "daily"` — deferred to the daily `/docs-refresh` PR) are pushed (a deploy-preview, if the repo has one, reflects the change). The merge is irreversible. Before merging, present:
  - the **§4 DB-change summary** (what was just pushed to prod),
  - the **§4.9 SESSION-VERIFIED block** — the items the session already ran itself, each with its observed result, so the human sees what is *already* covered and doesn't redo it,
  - the **§4.9 HUMAN TESTING REQUIRED block** — only the genuine *check-now* human residue, each with what to check, or `No human testing required.`, and
  - the **§4.9 POST-DEPLOY MONITORING block** — the post-deploy-only outcomes that will be **auto-filed as a verify-the-fix ticket after the merge** (V-257), shown so the human can redirect it, or `No post-deploy monitoring needed.`, and
  - the **§4.9 DEFERRED VERIFICATION block** — the genuinely-owed check-later verifications that will be **auto-filed as a `check-later:` ticket after the merge** into the deferred-checks bucket (V-319), shown so the human can redirect it, or `No deferred verification owed.`

  Then **STOP and wait for an explicit, testing-aware approval.** The human runs the residual *check-now* tests against the pushed DB / deploy-preview and reports each check-now item's result before approving the merge. A §5 yes does **not** satisfy this gate — there is **no auto-flow from `db push` to merge**. Only on explicit approval here do you proceed. Render this pause per convention 11 — immediately preceded by the one-line `<TICKET-ID>: <goal>` restatement, then a `needs input:` prose stop, never an `AskUserQuestion` modal.
- **No blanket proceed past genuinely-required manual testing — the discharge token is distinct from the merge `p` (V-318); post-deploy-only outcomes do not gate (V-257).** When the **HUMAN TESTING REQUIRED** (check-now) block is non-empty, a bare proceed — including `/go`'s one-token `p` — does **not** satisfy it and must **not** be suggested or accepted as a way to skip the testing. **The overload this closes:** one `p` cannot mean both "merge the residue-free default" *and* "I looked and it's fine" — self-certifying an eyeball item on the merge keystroke carries no proof and is exactly the honor-system trust V-318 retires. So a standing check-now item is cleared **only** by a token **distinct from `p`**, mirroring the §4.6 `waive HIGH: <finding>` precedent:
  - **`checked <item>: <observed result>`** — the human reports what they actually saw for that specific item (`checked chips: all three render above the calendar`). Discharge-by-report.
  - **`waive check <item>`** — the human explicitly waives that specific item without checking it (a deliberate, per-item skip — never a blanket yes).

  Both name the item; neither is `p`. A bare `p` with a check-now item still standing authorizes **only** the residue-free merge default and **re-surfaces** the standing item as its own decision — it never discharges it (the encoded proof below). **For a `(VISUAL)` item the session already ran its half first (§4.9 Step 3b):** anything a browser tool could assert/screenshot was promoted to SESSION-VERIFIED and is no longer here; the localhost is **already running** and its URL is presented **by default** with the item, so the human eyeballs it in one click and replies `checked <item>: …` — the residual human step is only the genuinely-subjective judgment a tool can't make. This is the sibling of the §4.6 security-HIGH hard-block below: `p` advances the merge only when no required human-only *check-now* testing stands; the residue is the human's act, cleared by its own token, never the gate's default. **The POST-DEPLOY MONITORING and DEFERRED VERIFICATION blocks are different by design:** neither is human-checkable before the merge — monitoring because it is post-deploy-only, deferred verification because it needs an active run that can't happen at the gate (V-319) — so **neither blocks** and `p` is correct to pass both — each is discharged by the auto-file below, not by a human report. This is exactly why an owed-but-unverifiable-at-land item is captured as a `check-later:` ticket rather than waived on `p` into Done-comment prose: `p` never carries it, and it is never lost. (When the check-now residue is empty — `No human testing required.` — the merge is the residue-free default: still take the one-line irreversible-merge confirmation, but `p` / an immediate yes is fine, even with a non-empty POST-DEPLOY MONITORING or DEFERRED VERIFICATION block.)
- **Security hard-block (§4.6) — a HIGH is cleared only by a fix or a *distinct, finding-specific waive token*, never the routine merge keystroke.** Any unresolved HIGH-severity security-review finding from §4.6 **blocks the merge** — do NOT merge while one stands. It is cleared in exactly two ways: **(a) fix it** — push a fix to the head branch, then re-run §4.6 against the new diff; or **(b) the human waives the specific finding with a distinct, deliberate token** — typing **`waive HIGH: <finding>`** that names the finding (e.g. `waive HIGH: launchctl persistence install`). The routine merge-approval keystroke — the one-line "merge?" yes, and `/go`'s one-token `p` — authorizes **only the merge default** and **never** waives a HIGH: a bare approval with a HIGH still standing leaves the merge **blocked** and re-surfaces the HIGH as its own decision. This is what keeps the merge-blocking gate from collapsing into the happy-path yes — waiving a structural HIGH demands a stronger, finding-naming barrier than routine proceed, not the same keystroke. A §5 acknowledgement does not satisfy this either — a HIGH is surfaced at §5 but only cleared here, by a fix or the `waive HIGH: <finding>` token. MEDIUM/LOW were already acknowledged at §5 and do not block.
- On approval + green + no unresolved HIGH (or a distinct per-finding `waive HIGH: <finding>` token — never the routine merge keystroke / `p` alone): `gh pr merge <n> --squash` (the default per Load config — collapses the PR to one commit so merge commits don't accumulate; use `--merge` only when explicitly asked) (**no `--delete-branch`** — it tries to switch the local checkout toward `main`, already checked out at the main worktree, so local cleanup aborts mid-land; branch teardown — local **and** remote — is done explicitly in §7 instead). Confirm merged via `gh pr view <n> --json state`.
- The merge commit now carries both the feature work and the docs ripple atomically — no follow-up commit on `<baseBranch>` needed. (Under `maintenance: "daily"` there is no per-land ripple; the daily `/docs-refresh` PR carries doc updates separately — V-284.)
- **Auto-file the deferred-check spin-outs (post-merge) — both file into the deferred-checks bucket (V-257 monitoring; V-319 check-later + unified home).** *After* the merge is confirmed (never before — a declined merge must not orphan a follow-up), file each of the two non-blocking blocks the human did not redirect at the pause, per convention 9's shape — so a deferred check is *tracked*, not lost to a waived residue (the ad-hoc V-247/248/249 pattern, now mechanized). **Both file into `cfg.deferredChecksProject` ("deferred checks") — the unified home** (V-319 option B; falls back to `cfg.standaloneProject` if `deferredChecksProject` is unset, keeping older repos working). They stay distinct by dedupe key + title:
  - **Monitoring spin-out** — if the §4.9 **POST-DEPLOY MONITORING** block is non-empty: **dedupe** `list_issues` on `cfg.deferredChecksProject` across non-terminal states (`Todo`/`In Progress`/`Backlog`, `label: <scopeLabel>`) — a live ticket already carrying `monitor-key: <this-ticket-ID>` → **skip** (re-merge, monitor exists). Else **file** `mcp__linear save_issue` (create): `team: <linearTeam>`, `project: <cfg.deferredChecksProject>`, `title: "verify-the-fix: <TICKET-ID> — <one-line what to confirm post-deploy>"`, `labels: [<scopeLabel>]`, `description` = a `monitor-key: <TICKET-ID>` line + a `## Acceptance` checklist (convention 3) of the POST-DEPLOY MONITORING checks verbatim + a pointer back to this PR/ticket.
  - **Check-later deferred-verification spin-out (V-319)** — if the §4.9 **DEFERRED VERIFICATION** block is non-empty: **dedupe** the same way on `defer-key: <this-ticket-ID>` (distinct from `monitor-key`, so a ticket owing *both* files two independently-deduped tickets) → **skip** if a live one exists. Else **file** `mcp__linear save_issue` (create): `team: <linearTeam>`, `project: <cfg.deferredChecksProject>`, `title: "check-later: <TICKET-ID> — <one-line owed verification>"`, `labels: [<scopeLabel>]`, `description` = a `defer-key: <TICKET-ID>` line + the **origin** (`parent: <TICKET-ID>` · the specific waived acceptance item · why it couldn't run at land) + a `## Acceptance` checklist (convention 3) of the owed verification steps verbatim + a pointer back to this PR/ticket. Filed into Todo/Backlog so it surfaces for later discharge (Acceptance #2).
  - **Read back before naming either (convention 8, §4.6).** Use the `identifier`/`url` each `save_issue` response actually returns — never a fabricated ID. If a `save_issue` fails, surface it (do not claim a follow-up was filed); the residue was non-blocking, so a failed file does not block the completed merge, but it must be reported honestly, not papered over.
  - Carry each returned ID to §8's Done comment (named there per the deferred-item rule): `post-deploy monitoring: <monitor-ID>` and/or `deferred verification: <defer-ID>`.

**Worked example — `p` alone never waives a HIGH.** At the §6.7 pause a structural HIGH stands — say the by-design `launchctl`/`launchd` persistence install that §4.6 rated HIGH. The human replies `p` (or the bare one-line "merge?" yes). That is the routine merge-approval token: it authorizes the merge default, but the HIGH is still unresolved, so **the merge stays blocked** and the HIGH is re-surfaced as its own decision — `p` did *not* clear it. The merge proceeds only when the human either pushes a fix and re-runs §4.6, or types the distinct finding-naming token **`waive HIGH: launchctl persistence install`**. The routine proceed keystroke and the HIGH-waiver token are deliberately different barriers: this is the encoded proof of the negative for this gate — a routine `p` cannot, by construction, waive a HIGH (mirrors §4.6's escalate-only floor, and `/go` drives the same distinction at `go.md` §2's "`p` never carries the §4.6 waiver").

## 6.7b Post-merge redeploy — deployed-service code isn't live until it's shipped (V-343)
A merge lands code into `<baseBranch>`, but a **deployed service keeps serving its last built/shipped artifact** until it is redeployed — so a merged fix to a deployed service's source is **not live** the moment the PR merges. Nothing else in this flow ships it (the review + gates inspect only the diff), so a deployed-service fix silently sits un-deployed and any **verify-the-fix monitor §6.7 filed for it can never be honestly discharged** — its "fix is live" precondition never happened (the exact V-239→V-331 strand: the `query` filter merged into `services/linear-mcp/src` but the Hetzner box kept serving the Jun-21 `dist`). This step closes the gap for **any** land whose diff touched a declared deployed service — bug-fix or feature, every route (diff-driven, route-invariant like §4.8).

**Config-driven — the mechanism is here, the per-service commands live in `cfg.deployedServices`.** Declared in **each deploying service's own repo's** `.claude/ticket-flow.json` (absent/empty → this whole step is a **silent no-op**; a repo that deploys no runtime service — including this pipeline-source repo itself — leaves it empty, don't announce a skip):
```json
"deployedServices": [
  {
    "name": "myapp",
    "covers": ["src/", "dashboard/", "package.json"],
    "redeploy": "<the per-service redeploy recipe — e.g. build → ship to the box → restart the service unit(s); full runbook path>",
    "verify": "<the observed read-back that proves it went live — e.g. systemctl is-active + curl the live endpoint carries the change (convention 8)>"
  }
]
```
- `covers` — path prefixes (same shape as `cfg.docs.systemDocs[].covers`), **repo-relative to the deploying service's own repo**: a merged-diff touch of any of them means that service's running artifact is now stale.
- `redeploy` — the human-readable redeploy recipe for **that** service (per-service; each deployed repo fills in its own — a product app's deploy path in its own repo, the extracted linear-mcp wrapper's VPS steps in `linear-mcp-lean` per LEAN-9).
- `verify` — the observed check that the redeploy went live, read back per convention 8 (never asserted from intent).

**Fire when the merged diff touched a declared service's `covers` paths.** For each `cfg.deployedServices[]` entry, reuse the `gh pr diff <n> --name-only` read (§4.9 Step 1 already ran it — don't re-call): any changed path prefixed by an entry's `covers` → that service **needs redeploy**. No entry matches → nothing to ship, proceed to §6.8.

**Surface it as a post-merge redeploy ack-gate (convention 11's third kind) — the redeploy is a live remote-infra action, the human's, never auto-run by the land flow.** A redeploy is a secret-gated live-system mutation (SSH/`rsync`/`systemctl restart` against prod) — exactly the class the Hard-rules "no silent live-system mutation" clause + `guard-secret-access.py` govern (the hook `ask`s on the `systemctl` write by design). So `/land-ticket` does **not** run it silently; it presents the redeploy the service needs and holds an **ack gate a bare `p` must not pass** — because letting a deployed fix sit un-shipped is the precise silent-stale-prod failure this step exists to stop. Render per convention 11, immediately preceded by the goal restatement:
  `<TICKET-ID>: <goal>`
  `needs input: [HARD STOP] PR #<n> touched deployed-service <name> — the merged fix is NOT live until it is redeployed: <redeploy recipe>. Redeploy it (a live remote-infra action — yours) and confirm it live (<verify>), or reply 'defer deploy: <name>' to track it as a check-later follow-up. A bare 'p' does not pass this gate.`
- Clear it exactly one of two ways:
  - **Redeploy + read back (convention 8).** Run the per-service `redeploy` recipe (the secret-gated commands the human authorizes — the `guard-secret-access.py` hook `ask`s on the `systemctl`/settings writes as designed; a bg worker that cannot perform the remote action surfaces that and hands it to the human), then observe `verify` (GET the live endpoint / `systemctl is-active`, assert the change is actually live — **never** claim it from intent). Record the observed result; carry a one-line `redeployed <name>: <observed proof>` to §8's Done comment.
  - **Defer it — `defer deploy: <name>`.** The redeploy is genuinely owed but the human chooses to ship it later. File it as a **`check-later:` deferred-verification spin-out** exactly like §6.7's DEFERRED VERIFICATION auto-file: **dedupe** `list_issues` on `cfg.deferredChecksProject` (non-terminal, `label: <scopeLabel>`) for a live `defer-key: <TICKET-ID>-deploy-<name>` → skip if present; else `save_issue` (create) `team: <linearTeam>`, `project: <cfg.deferredChecksProject>`, `title: "check-later: <TICKET-ID> — redeploy <name> so the merged fix goes live"`, `labels: [<scopeLabel>]`, `description` = a `defer-key: <TICKET-ID>-deploy-<name>` line + origin (`parent: <TICKET-ID>` · the touched `covers` paths · "merged but not yet redeployed") + a `## Acceptance` checklist (convention 3) of the `redeploy` recipe steps + the `verify` read-back, and a pointer back to this PR. **Read the returned ID back** (convention 8 — never a fabricated ID); carry `deferred redeploy: <defer-ID>` to §8's Done comment. (The `-deploy-<name>` key suffix keeps this distinct from §6.7's `monitor-key`/`defer-key`, so a ticket owing a monitor *and* a redeploy files independently-deduped tickets.)
- **Under `/go`** this is an ack gate driven per `go.md` §2's ack-gate protocol: a bare `p` re-states it; only an explicit redeploy-report or `defer deploy: <name>` clears it (logged `ack'd`/`intervened`).
- **Ties the verify-the-fix monitor to reality (Acceptance #2).** When §6.7 auto-filed a verify-the-fix monitor for this ticket, its "fix is live" precondition **is** this redeploy. Completing the redeploy here means the monitor's precondition is actually met (so it can later be discharged); deferring records the redeploy as a tracked `check-later:` blocker — so the monitor is never left un-closeable sitting on a silently-un-deployed merge (the exact V-331 strand).

**Worked example — deployed-service touch → redeploy ack-gate, never a silent stale prod (V-343; the encoded proof for this markdown repo).** In a repo that declares a deployed service (e.g. `myapp` in its own repo, `covers`-ing its service source), a ticket lands a change into a covered path (say `src/worker.ts`). §6.7b reads `gh pr diff --name-only`, matches it against the `myapp` entry's `covers` → the service needs redeploy. After the merge confirms, the gate renders `needs input: [HARD STOP] PR #<n> touched deployed-service myapp — the merged fix is NOT live until it is redeployed: <myapp's redeploy recipe>. … A bare 'p' does not pass this gate.` The human either redeploys (and `/land` reads back the service active + the live endpoint carrying the change, convention 8) or replies `defer deploy: myapp` → a `check-later: <ID> — redeploy myapp so the merged fix goes live` ticket is filed into "deferred checks" (`defer-key: <TICKET-ID>-deploy-myapp`, acceptance = the redeploy steps + the read-back). **The negative this proves:** the exact motivating incident — the V-239 wrapper fix that merged but was never redeployed, leaving prod serving the stale build while its V-331 verify-the-fix monitor sat un-closeable (back when the wrapper lived in this repo; it has since been extracted to `linear-mcp-lean` per V-351, which owns its own redeploy gating as LEAN-9/LEAN-8) — can no longer pass silently: the merge surfaces the owed redeploy as an ack gate, either shipped-and-verified or tracked as a ticket, never lost. (Contrast: a land whose diff touches no `cfg.deployedServices[].covers` path — the common case, including every land in this pipeline-source repo, which deploys no service of its own — is a silent no-op here.)

## 6.8 Pre-teardown deliverable gate (gate the irreversible step on deliverables, not just "merged")
§7's teardown is irreversible — it removes the worktree and the branch. The moment it runs, the only place left to fix a missed deliverable is the **shared background checkout**, which trips the bg-isolation + read-state guards (a silently-missed §6.5 ripple becomes unrecoverable on a live worktree). So before §7, re-confirm any *intended* docs ripple actually landed in the merge. **Skip this gate entirely — silent no-op — if `cfg.docs` is absent, OR `cfg.docs.maintenance` is `"daily"` (V-284 — no per-land ripple is ever intended under the daily model; this skip is deliberately in lockstep with §6.5's `"daily"` skip so the two gates' "intended" definitions keep agreeing), OR no ripple was intended** (the common case; do not announce a skip).

- **Was a ripple intended?** Yes if EITHER: §4.7 raised `docs-stale-risk` item(s) the user chose **update** at §5, OR any `cfg.docs.systemDocs[].doc` was itself in the PR diff (so §6.5 was required to bump that doc's freshness header). Neither holds → no ripple intended → skip to §7.
- **Assert the §6.5 ripple commit landed in the merge.** The ripple rides in the `docs: ripple updates for #N` commit (§6.5). Confirm it's present in the merged history and touched the intended doc(s): `git -C <worktree> fetch origin <baseBranch>`, then `git -C <worktree> log origin/<baseBranch> --grep "docs: ripple updates for #<n>" --oneline` is non-empty, AND `git -C <worktree> show --name-only <that commit>` lists each intended doc path. (Reading paths in-context — do not pipe to `grep`/`jq`.)
- **Missing → STOP, do NOT tear down.** Emit `needs input: a docs ripple was intended (<docs-stale-risk update on <doc> | freshness bump for <doc>>) but no §6.5 ripple commit touched it in the merge. The PR worktree is still live at <worktree> — fixing it there now is a small edit; tearing down first would strand the fix in the shared checkout. Read the doc in <worktree> first, then edit + commit + push the ripple, before §7 removes the worktree.` Teardown stays blocked until the intended ripple is present in the merged history. (Recovery edits obey the "isolated-worktree, Read-first" hard rule below.)
- Present (or gate skipped) → proceed to §7.

## 7. Post-merge cleanup (mode-aware — convention 5 + parallel-eligibility)
**§6.8's deliverable gate must have passed (or been a no-op) before any teardown below runs.** Read `<worktree>/.claude/active-project.json` and branch on `mode`:

> **Teardown never `cd`s into or checks out the main branch.** Every op that touches the main checkout uses `git -C "<main worktree>"` — where `<main worktree>` is `$MAIN_WT`, the resolver from "Load config" (`git worktree list --porcelain` first entry), **never** a slug-derived `~/<repo>` guess; never `git switch main` / `git checkout main` — `main` is already live in the main worktree, so checking it out from anywhere trips *"'main' is already checked out at <main>"*. To refresh main, **use the `refresh-main-ff.sh` helper below** (fetch + fast-forward in place, drift-preserving — never a blind `pull`, never `reset --hard`/force, which would discard the main checkout's local drift); to ready a *reused* project worktree for the next ticket, **detach** (`git switch --detach origin/<baseBranch>`, feature branch below) — never name the `main` branch in a checkout/switch.

**`mode: "standalone"`** (per-ticket worktree — covers both the `standaloneProject` bucket and any feature project carrying the Linear `parallel` project label; binding doesn't distinguish them, cleanup is identical):
- Tear it down immediately on merge. **Order matters:** git refuses to `branch -D` a branch that is still checked out in a worktree — and with `--delete-branch` dropped from §6.7, nothing switches the standalone worktree off `<branch>` first — so the worktree must be **removed before** the branch is deleted. Run the steps below as **discrete** Bash calls (never `&&`-chained or padded — see the warning in the remove step):
  - **Refresh main — `bash ~/.claude/bin/refresh-main-ff.sh "<main worktree>" <baseBranch>`** (the canonical refresh the other modes reference). ONE allowlisted call (matches the `Bash(bash ~/.claude/bin/*.sh)` prefix — no prompt, no padding; conv. 7). The helper always `fetch`es (so `origin/<baseBranch>` / `git log origin/<baseBranch>` are current even if the local ref can't move), then fast-forwards the local `<baseBranch>` ref **in place**. Crucially it handles the **perpetual-drift collision** that used to abort every land in this repo's own main checkout (V-334): `settings.json` (harness-written `model`/`effortLevel` machine-local drift) and `pipeline/audit/gate-audit.md` (`/go`'s append log) carry permanent uncommitted drift, and when an incoming commit touches a drifted file `--ff-only` aborts **file-level** (`Your local changes … settings.json … Aborting`) — so pre-V-334 every land ended with the same manual "commit/stash then ff-only" recovery note. The helper advances the ref while **preserving the drift verbatim** (stash → ff → `git checkout <stash> -- <drifted files>`, an overwrite-restore, **not** a `stash pop` — a pop's 3-way merge leaves `<<<<<<<` conflict markers in the live `settings.json`; empirically both `stash pop` and `assume-unchanged`/`skip-worktree` fail, so the helper uses neither). It **never forces, resets hard, or discards drift**: a clean checkout just fast-forwards; a genuine collision advances the ref and leaves the drift as the same uncommitted `M` it was; a non-drift blocker (staged change) restores state and surfaces the recovery note, continuing teardown either way (branch + worktree teardown still complete).
  - **`ExitWorktree(action: "keep")`** (returns the session to the dir it was in before `/next-ticket`'s `EnterWorktree(path)` call — i.e., main; no-op if the session was never inside the worktree). MUST come before the remove — deleting the dir while the session's cwd is inside it leaves the session in a dead-cwd state. **If `ExitWorktree` is refused** (`already isolated / cwd override` — a harness tool, refuses non-deterministically on resumed/isolated sessions; we can't fix it, only report upstream): do NOT self-remove from inside (dead-cwd) and do NOT reach in via `git -C`/absolute paths (that defeats the worktree protection `EnterWorktree` enforces). Leave the worktree on disk, skip the remaining steps, still do §8 (Linear close), and surface that `git worktree remove "<path>"` (plus the `branch -D` / `push origin --delete` below) must be run by a principal **outside** the sandbox — the main session or the user — never this worker.
  - **Remove the worktree** — **`bash ~/.claude/bin/worktree-remove.sh "<standalone worktree>"`** — the helper: ONE call that matches the already-allow-listed `Bash(bash ~/.claude/bin/*.sh)` prefix, runs `git worktree remove` and reports git's own exit code itself (refuses on changes → STOP, ask; never force). **Do NOT run a bare `git worktree remove` here and do NOT pad it (or the `branch -D` / `push origin --delete`) with `2>&1 | tail` / `; echo "exit: $?"` — a pipe or chain matches no allowlist prefix and prompts every land (conv. 7).** (The main-refresh `fetch` + `merge --ff-only` are now inside `refresh-main-ff.sh` — one allowlisted call, nothing to pad.)
  - **Delete the local branch** (now free — the worktree that held it is gone): `git -C "<main worktree>" branch -D <branch>` (force — a **squash-merge** leaves the local tip a non-ancestor of `main`, so plain `branch -d`'s ancestor check fails *"not fully merged"*; `-D` is the correct test here *because* §6.7 verified merge-on-remote upstream).
  - **Delete the remote branch** (this is what `--delete-branch` used to do): `git -C "<main worktree>" push origin --delete <branch>`. Run from the main worktree (outside any worktree on the branch) — it only deletes the remote ref and never switches a local checkout, so it cannot trip the "main already checked out" condition.
- Do NOT close the Linear project. The bucket is perpetual by design; a parallel-labeled feature project is finite but the user closes it manually in Linear when the matter is done (deliberate signal, not automated).

**Feature binding** (no `mode` field; has `linearProject`) — the worktree is **bound to the project, reused across its tickets**, do NOT remove per-ticket:
- **Detach the worktree off the merged branch first.** In the project worktree: `git -C "<project worktree>" fetch origin <baseBranch>`, then `git -C "<project worktree>" switch --detach origin/<baseBranch>` so it's clean, off `<branch>`, and ready for the next ticket. Working tree must be clean — if not, STOP and ask. This MUST precede the branch deletion: git refuses to `branch -D` a branch still checked out in a worktree, and with `--delete-branch` dropped from §6.7 nothing else switches the worktree off it.
- Then delete the merged ticket branch — local **and** remote (discrete calls; never `&&`-chained or padded): `git -C "<project worktree>" branch -D <branch>` (force — §6.7 confirmed the merge on the remote, and a squash-merge leaves the local tip a non-ancestor of `main`, so `branch -d` would fail *"not fully merged"*; `-D` is safe because merge-on-remote was verified upstream), then `git -C "<project worktree>" push origin --delete <branch>` (this is what `--delete-branch` used to do; it deletes only the remote ref and never switches a checkout).
- **Completion check:** existence-only — `list_issues` `project: <bound>`, `label: <scopeLabel>`, **`limit: 1`**, run once per non-done state in order `["Todo", "In Progress", "In Review", "Backlog"]`, **short-circuit on first hit** (any hit → project continues, keep the worktree; stop checking remaining states). All four empty → last ticket: refresh main (the canonical refresh from the standalone branch above — `bash ~/.claude/bin/refresh-main-ff.sh "<main worktree>" <baseBranch>`; advances the ref through drift, preserving it verbatim, never forcing/discarding — V-334), then `ExitWorktree(action: "keep")` (returns the session to main; no-op if the session was never inside the worktree — same rationale as the standalone branch above), then **`bash ~/.claude/bin/worktree-remove.sh "<project worktree>"`** (the helper — one allowlist-matching call reporting git's exit code; refuses on changes → STOP, ask; never force; **never a bare/padded `git worktree remove … | tail` or `; echo "exit: $?"` — see the standalone branch above**), and `mcp__linear save_project` → completed, with a summary. The `ExitWorktree` MUST come before the remove (same refusal-fallback as the standalone branch above — on refusal, leave the worktree and delegate the remove to an outside principal).

**`mode: "milestone"`** (per-milestone worktree — `{linearProject, linearMilestone}`; convention 5's milestone mode — reused across the milestone's tickets, like feature mode, but N milestones run in parallel) — the worktree is **bound to the milestone, reused across its tickets**, do NOT remove until the milestone's *last* ticket lands:
- **Detect last-in-milestone (the keep-vs-teardown decision).** The issue isn't Done until §8, so detect by **excluding the current landing ticket**: `list_issues` `project: <linearProject>`, `label: <scopeLabel>` (paginate `hasNextPage`/`cursor`), **filter client-side** to `projectMilestone.id === <linearMilestone>` (`list_issues` has no milestone filter), drop the current ticket, and check whether **every remaining milestone issue is Done** (`statusType === "completed"`). All-others-Done → this is the milestone's last ticket (`isLastInMilestone = true`). **Carry this verdict to §8's milestone close-out** (don't re-detect there — by §8 the current issue is also Done, which yields the same verdict). No other milestone issue exists at all → also last (a single-ticket milestone). **This `isLastInMilestone` is computed only on the milestone-mode path** (it drives the teardown decision); for a standalone/feature ticket that *carries* a milestone, §7 never runs this detection, so §8's milestone close-out computes last-in-milestone **inline** instead.
- **Detach off the merged branch first**, same as feature mode: `git -C "<milestone worktree>" fetch origin <baseBranch>`, then `git -C "<milestone worktree>" switch --detach origin/<baseBranch>`. Working tree must be clean — if not, STOP and ask. Then delete the ticket branch — local **and** remote (discrete calls; never `&&`-chained or padded): `git -C "<milestone worktree>" branch -D <branch>` then `git -C "<milestone worktree>" push origin --delete <branch>`.
- **Not last (`isLastInMilestone = false`)** → **keep the milestone worktree** (it carries over to the milestone's next ticket via `/next-ticket` §1.A's milestone-cwd-binding reuse). Done with §7.
- **Last (`isLastInMilestone = true`)** → tear the milestone worktree down: refresh main (the canonical refresh from the standalone branch — `bash ~/.claude/bin/refresh-main-ff.sh "<main worktree>" <baseBranch>`; advances the ref through drift, preserving it verbatim, never forcing/discarding — V-334), `ExitWorktree(action: "keep")`, then **`bash ~/.claude/bin/worktree-remove.sh "<milestone worktree>"`** (same helper + refusal-fallback as the other branches — never a bare/padded `git worktree remove`). This leaves **no orphan** — the deterministic per-milestone path is freed and `orphan-detect.sh` finds nothing for it. Do NOT `save_project → completed` here — milestone-mode project completion is decided in §8's close-out (only when *all* milestones have landed), since other milestones may still be in flight in their own worktrees.

## 8. Close Linear (umbrella-aware)
- **Acceptance-checklist verification first:** read the issue's `## Acceptance` checklist (convention 3). For each item, verify it against the merged diff/tests. List met vs unmet. If any item is unmet, surface it at the step-5 confirm gate — do NOT auto-close the issue with unmet acceptance unless the user explicitly waives the specific items. (The artifact-kind completeness portion of this already ran **pre-merge** in §4.8 — so a missing migration blocks before the irreversible merge, not after; §8 re-confirms against the now-merged state and closes.)
- **An `invariant` item is "met" only with its encoded proof (§4.8) — and an unmet/deferred one must be NAMED in the Done comment, never papered over.** When an `invariant` item was deliberately deferred at the §5 gate (its property genuinely belongs to a follow-up), the Done comment states it explicitly, **naming the auto-filed `check-later:` ticket §6.7 minted** — **"criterion X (invariant) deferred to <check-later-ID>: <property> not yet proven"** (V-319 — the owed verification is a tracked ticket in the deferred-checks bucket, not a bare prose note) — rather than reporting "all acceptance met." The clean "all acceptance met" close is reserved for when every item, invariants included, is **actually** true (each invariant has its encoded proof in the merged diff). Honest Done > clean Done.
- **Merging a PR does NOT transition Linear** — by design, PR bodies carry no closing keyword (§0), so this step is the *only* close path. Don't wait for an auto-move; close explicitly here.
- Umbrella with PRs still open → leave In Review, comment what landed/remains.
- Last linked PR merged + acceptance met (or waived) → `save_issue` state Done + summary comment (PRs, migrations, checklist status, **docs-ripple status** — and any deferred `invariant` criterion named per the rule above, **and the auto-filed spin-out IDs if §6.7 filed any** — `post-deploy monitoring: <monitor-ID> — <what it verifies>` (V-257) and/or `deferred verification: <defer-ID> — <what it verifies>` (V-319), so the post-deploy-only residue *and* any owed check-later verification are tracked, not silently dropped). This explicit, acceptance-gated transition is the sole way the ticket reaches Done.
- **Closing a ticket as a duplicate → set `Canceled`, never the `Duplicate` state (V-239).** The `Duplicate` workflow state requires a formal duplicate issue-relation, which `save_issue` cannot create (`save_issue state:"Duplicate"` fails `missing duplicate relation`). So when a ticket is a duplicate, `save_issue` → `Canceled` with a comment naming the canonical ticket (`duplicate of <ID>`), and — if a true duplicate link is wanted — add the relation by hand in Linear's UI. **Do not retry the `Duplicate` state** on the `missing duplicate relation` error; it will never succeed via the wrapper. (Finding the canonical existing ticket to point at uses `list_issues({query: "<title text>"})` — the V-239 text search — not raw `linear_graphql`.)
- **Docs-ripple line — sourced from the committed diff, never from intent.** The summary comment's docs-ripple status comes from the **actual `docs: ripple updates for #N` commit** — reuse the SHA + touched-paths that §6.8 already resolved (its `git log origin/<baseBranch> --grep "docs: ripple updates for #<n>"` + `git show --name-only`). §8 runs *after* §7 teardown, so the per-ticket worktree is gone — do **not** re-run the lookup with `git -C <worktree>`; carry §6.8's captured result forward, or re-run the same `git log`/`git show` from a live checkout of the repo (the shared main checkout), never the torn-down worktree path. Report `docs ripple: <K> file(s), commit <short-sha>` listing the touched docs, or `docs ripple: none` when no such commit exists — except under `cfg.docs.maintenance: "daily"`, where the accurate line is `docs ripple: deferred to daily /docs-refresh` (no per-land ripple exists by design; V-284). **Never write "docs updated" / "changelog appended" from the *intended* edit** — only from a commit that is provably in merged history.

**Milestone close-out — runs right after the Done transition above.** The landing issue is now Done. A Linear milestone has no state field of its own — "closed" is purely issue-derived (a milestone is closed precisely when its last issue reaches Done), so there is **no separate milestone-close action to fire here**. (The subsystem-altitude close-out question is covered on-demand by `/validate` §2, which computes each milestone's all-issues-Done + verification-criterion probe inline, and by `/verify-tests` at the diff level. The keep-vs-teardown worktree decision still uses §7's `isLastInMilestone`; project auto-completion is below.)
- **Project completion (milestone mode):** existence check — `list_issues` `project: <linearProject>`, `label: <scopeLabel>`, **`limit: 1`**, once per non-done state `["Todo", "In Progress", "In Review", "Backlog"]`, **short-circuit on first hit**. All empty → **every** milestone has landed → `mcp__linear save_project` → completed, with a summary. Any hit → another milestone is still in flight (in its own parallel worktree) → leave the project open. (This mirrors feature mode's auto-complete-on-last; a `milestone-parallel` project is a finite feature project, so feature-mode completion is the right behavior — unlike the perpetual standalone bucket, which never auto-completes.)

## Hard rules
- **No silent live-system mutation / activation step.** The land flow's review + gates inspect only the PR diff; the most security-relevant action is often the post-merge "run the installer / activate" step that rides *outside* it (e.g. `--dangerously-skip-permissions` added to a crontab via `crontab -`, seen by no gate). So any `crontab`/`launchctl`/`systemctl` **write**, any `settings.json` `permissions`/`allow`/`deny` edit, or any command carrying a **permission-bypass flag** is never a free post-merge action: the `guard-secret-access.py` PreToolUse hook hard-DENIES a bypass flag (anywhere in the command — incl. buried in a `crontab -` heredoc) and raises an explicit `ask` on the crontab/launchd/systemd write + the settings-perms edit. Treat "activate" as a gated step; never reach for a bypass flag to get past it. (Enforcement is the guard, not this prose — `bin/guard-secret-access.test.sh` is its proof. The hook sees tool-level commands only, so a mutation made *inside* a shell script you run is invisible to it — invoke such commands directly so the gate can see them.)
- **Never edit the shared background checkout.** Every `/land-ticket` edit — pre-merge work (§6/§6.5), a §6.8 gate-tripped recovery, or any post-merge fix — happens inside the live isolated PR worktree (`<worktree>`), and you MUST `Read` a target file **in that worktree path** before editing it (the harness read-state guard is keyed to the absolute path; a guessed `old_string` from a sibling path fails). If the worktree was already torn down, `EnterWorktree` a fresh isolated worktree and `Read` there first — never reach into the shared checkout via `git -C`/absolute paths (that trips the bg-isolation guard cascade). §6.8 exists to keep the worktree alive so this rule is satisfiable.
- **A merged fix to a deployed service isn't live until it's redeployed (§6.7b, V-343).** When the merged diff touches a `cfg.deployedServices[].covers` path, the running service keeps serving its old artifact — so §6.7b holds a post-merge redeploy ack-gate (a bare `p` never passes it) that is cleared only by shipping + reading the change back live (convention 8) or an explicit `defer deploy: <name>` that files a tracked `check-later:` follow-up. The redeploy itself is a secret-gated live-system mutation (the "no silent live-system mutation" rule above governs it) — the land flow never runs it silently; it surfaces it as the human's gated act. Never let a deployed-service land finish with the fix silently un-shipped.
- Classify failures (code/staleness/infra) before reacting; never "land red"; never blind-retry infra.
- `<requiredCheck>` is the only required gate — never merge unless it's green; other jobs non-blocking by design.
- No migration (`db push`) without the §5 confirm; no **merge** without the *separate* §6.7 manual-test pause approval. §5 approves proceeding, not merging — never auto-flow `db push → merge`.
- The session runs every manual-test item it can observe from the diff/tools/repo itself (§4.9) and hands the human only the genuine *check-now* residue; a blanket proceed (including `/go`'s `p`) never skips genuinely-required human-only *check-now* manual testing, and is never *suggested* as a way to — that residue is cleared only by the human's per-item result or an explicit, per-item waive (§6.7). A **post-deploy-only** outcome (no recurrence after deploy, fix-holds-in-prod) is *not* human-checkable pre-merge, so it never blocks the gate: §4.9 partitions it into the POST-DEPLOY MONITORING block and §6.7 auto-files it as a verify-the-fix / monitoring spin-out **after** the merge (convention 9; V-257) — tracking the outcome instead of dead-blocking on an un-verifiable-yet check. A **genuinely-owed acceptance-item verification that cannot run at land** (needs an active run / a throwaway resource) is likewise not human-checkable at the pause, so it too never blocks: §4.9 partitions it into the DEFERRED VERIFICATION block and §6.7 auto-files it as a `check-later:` deferred-verification spin-out into the deferred-checks bucket (convention 9; V-319) — a `p` never waives it into Done-comment prose; a merely-nice-to-have check is dropped, an owed one is minted.
- Never modify an **applied** migration; sole exception is a never-applied migration on the unmerged PR branch (step 4).
- Never force-push / `reset --hard` / discard a side without the user agreeing. Pushing a feature branch to re-trigger CI (steps 0/2.5) is fine.
- One ticket per invocation (an umbrella's multiple PRs = that one ticket).
