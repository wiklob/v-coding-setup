# /next-ticket — ingest one ticket (project-scoped)

Set up the build environment for the next ticket **of this worktree's bound project**. Does NOT write the feature code — building is the interactive phase after. Read `~/.claude/workflow-conventions.md` first (esp. conventions 4, 5) and follow it. (**Under `/go`**, `workflow-conventions.md` + the active profile are already in context from the chain's top-of-run read — skip the re-read per the `read-discipline` knob (V-293); a standalone invocation reads them here.)

**Known-ISSUE-ID fast path (read-discipline, `pipeline/profiles/opus-4-8.md`).** When `$ARGUMENTS` resolves a specific ISSUE-ID, the resolution is mechanical — one `get_issue`, optionally one label-check `get_project`, no `list_*` calls, a deterministic worktree path — so skip re-reading the conventions 4/5 *excerpts* upfront and proceed on the conventions already in context. This scopes only the redundant header re-read: convention 5 still governs §1.C's worktree naming/sanitizer and dirty-stop posture on the ISSUE-ID path, so honor it there. Keep the upfront excerpt re-read for an **auto-pick or ambiguous** target (no args, or a project-name picker), where ticket ranking and worktree-mode eligibility lean on the convention text.

## Linear MCP call discipline

`list_issues` defaults to `limit: 50` × ~1.2KB per issue, all of which stays in context. Always pass `state` + a tight `limit`. Use `get_issue` for known IDs, never `list_issues`.

**Schema-valid args — never pass `query` to these:** `get_issue` and `get_project` take `id` (the issue/project identifier), **not** a `query` key — a `query` key is rejected with `MCP error -32602 … unrecognized_keys ["query"]`. To inspect a project's labels use `get_project` and read `labels[*].name`; `list_project_labels` takes no `query` either (filter via `name`; args are `name` / `limit` / `cursor` / `orderBy`).

## Load config
- `root="$(git rev-parse --show-toplevel)"`; this exact command-launch checkout is the explicit `sourceRoot` for any worktree resolution in this invocation. Never substitute `git rev-parse --git-common-dir` or the first `git worktree list` row. Read `$root/.claude/ticket-flow.json` (missing → run `/ticket-flow-init`, STOP). Use `linearTeam`, `scopeLabel`, `baseBranch`.

## 1. Resolve target + mode (convention 5 + standalone exception)
Config may include `standaloneProject` (optional bucket — see convention 5).

### A. Cwd binding (when the launch checkout is a linked worktree)
- Determine whether `$root` is linked by comparing `git -C "$root" rev-parse --path-format=absolute --git-dir` with `git -C "$root" rev-parse --path-format=absolute --git-common-dir`. The primary checkout is unbound; do not call binding commands on it (the helper intentionally refuses primary-worktree binding state).
- For a linked worktree, run `node ~/.claude/bin/ticket-worktree.mjs migrate-binding --worktree "$root"` **before any presence check or read**. This atomically moves an old checkout marker into private Git metadata when safe; malformed JSON or a private/legacy mismatch is a hard STOP. Then run `node ~/.claude/bin/ticket-worktree.mjs binding-status --worktree "$root"`. `none` → continue to §1.B as unbound; `private` → run `node ~/.claude/bin/ticket-worktree.mjs read-binding --worktree "$root"` and use the returned JSON. No command reads the legacy checkout marker directly.
- If this linked checkout is an already-entered legacy sibling, **defer path migration**: keep using `$root` for this invocation and never move the session's current cwd. A later `/next-ticket` launched outside that legacy worktree performs the outside-in move in §1.C.
- **`mode: "standalone"`** → worktree hosts exactly one ticket (`linearIssue`). `/next-ticket` here is a no-op (one ticket per standalone worktree). STOP and soft-redirect: `/resume-ticket` to pick up the bound ticket, `/land-ticket` to finish it, or run `/next-ticket <ISSUE-ID>` **from the main worktree** to start another standalone in parallel.
- **`mode: "milestone"`** (has `linearMilestone`) → the bound milestone IS the target; this worktree is reused across that milestone's tickets (the way a feature worktree is reused across a project's). Ignore any arg naming a different ticket/project (warn). Skip to step 2 in **milestone mode** — §2's per-milestone In-Progress guard + bound-milestone auto-pick choose the milestone's next ready ticket.
- **Feature binding** (no `mode` field; has `linearProject`) → that project IS the target. Ignore any arg naming a different project (warn). Skip to step 2 in feature mode.

### B. Establishing run (no cwd binding)
Resolve **target + mode** from `$ARGUMENTS` + `standaloneProject` + parallel-label check (convention 5).

**Worktree-mode eligibility helper.** A project's worktree mode is read from its Linear project labels (one `mcp__linear get_project` on the candidate → inspect `labels[*].name`) plus config, in this precedence:
- **`milestone-parallel` label → milestone mode** (one worktree per milestone — see convention 5). **Wins over `parallel`:** a project carrying *both* labels takes milestone mode, and the skill **warns** (`project carries both 'parallel' and 'milestone-parallel' — using milestone mode; a project is per-ticket OR per-milestone, not both`). A project is either per-ticket-parallel or per-milestone-parallel, never both.
- **`standaloneProject` (config) OR the `parallel` label → standalone mode** (one worktree per ticket). The config case is known without an MCP call; the label case needs the `get_project`.
- **neither → feature mode** (one worktree per project).

When milestone- or standalone-eligible, the skill drops into **picker behavior** (present the ready tickets, ask which) — never the feature-mode auto-pick. (Milestone mode binds the worktree to the *picked ticket's* milestone — `issue.projectMilestone` — so the pick comes first, the milestone follows from it.)

Resolution:
- `$ARGUMENTS` = ISSUE-ID → no list calls. One `get_issue`; verify it carries `<scopeLabel>`; look up its parent project. Parent = `standaloneProject` → **standalone mode**, target = the issue (no extra call). Else → one `get_project` on the parent (read `labels[*].name`, precedence per the helper): `milestone-parallel` → **milestone mode**, target = the issue, bound milestone = `issue.projectMilestone` (no extra call — the `get_issue` already carried it; if the issue has *no* milestone → STOP and surface, a milestone-mode project's tickets must each belong to a milestone); else `parallel` → **standalone mode**, target = the issue. Neither label → **feature mode**, target = the parent project.
- `$ARGUMENTS` = project name/id → no `list_projects`. If = `standaloneProject` → **standalone-bucket picker**: `list_issues` `project: <bucket>`, `label: <scopeLabel>`, `state: "Todo"`, `limit: 10` → present, ask which. (If user names an ID not in the list, take it directly via `get_issue`.) On pick: standalone mode, target = chosen issue. Else → one `get_project` on the target (precedence per the helper): `milestone-parallel` → **milestone-mode picker** (`list_issues` `project: <target>`, `label: <scopeLabel>`, `state: "Todo"`, `limit: 10`, ask which; on pick → milestone mode, target = chosen issue, bound milestone = its `projectMilestone`); else `parallel` → **same picker behavior as the bucket** (Todo limit 10, ask which, standalone mode on pick), target = chosen issue. Neither → **feature mode**, target = that project.
- No args → one `list_projects` `state: started`, `label: <scopeLabel>` (+ the bucket as a row if `standaloneProject` set; the bucket itself isn't `started`). **No per-project preview — just project names.** User picks → bucket → standalone-bucket picker (lists individual tickets); other pick → one `get_project` on it (precedence per the helper). `milestone-parallel`-labeled → ticket picker (same as bucket), **milestone mode** on pick (bound milestone = the picked ticket's `projectMilestone`); `parallel`-labeled → ticket picker, standalone mode on pick. Otherwise → feature mode (step 2 auto-picks the top ticket).
- Never proceed without a target.

### C. Resolve the worktree (create new OR reuse existing)
This section runs only for an establishing, unbound launch checkout, so `sourceRoot="$root"` is the explicit checkout root captured in **Load config**. Derive the unchanged deterministic basename and both layouts through the installed helper — never reimplement the sanitizer:
- **Feature mode:** `node ~/.claude/bin/ticket-worktree.mjs resolve --root "$sourceRoot" --mode feature --project "<project-slug>"`.
- **Standalone mode:** `node ~/.claude/bin/ticket-worktree.mjs resolve --root "$sourceRoot" --mode standalone --issue "<ISSUE-ID>"`.
- **Milestone mode:** `node ~/.claude/bin/ticket-worktree.mjs resolve --root "$sourceRoot" --mode milestone --project "<project-slug>" --milestone "<milestone-name>"`.

Read the returned JSON in-context (no `jq`/pipe): capture `name` as `wt_dir`, plus absolute `managedPath`, `legacyPath`, `preferredPath`, and `layout`. The helper preserves the existing feature / per-ticket / per-milestone basename and the independently-capped project + milestone semantics. Ordinary repositories report `layout: "managed"` and prefer `<sourceRoot>/.claude/worktrees/<wt_dir>`; only an exact installed `~/.claude` source checkout reports `layout: "legacy-source"` and prefers the sibling path until V-376.

Classify **both returned paths** against `git -C "$sourceRoot" worktree list --porcelain` and the filesystem before creating or moving anything:
- Both paths registered, one path registered while the other is also occupied on disk, or any other conflicting dual presence → hard STOP and surface both paths. Never merge, nest, delete, or guess ownership.
- **Ordinary repo, managed registered only** → `WT_ABS=<managedPath>`; normal reuse.
- **Ordinary repo, legacy registered only** → migration is outside-in only. The managed path must be absent from registration **and** absent on disk; otherwise hard STOP as an occupied migration target (do not reclaim it). Then run `node ~/.claude/bin/ticket-worktree.mjs move-legacy --root "$sourceRoot" --legacy "<legacyPath>" --managed "<managedPath>"`. `status: "moved"` → `WT_ABS=<managedPath>`; `status: "deferred-current-worktree"` → keep `WT_ABS=<legacyPath>` and report migration deferred (the helper detected the caller inside the legacy cwd). Any helper refusal is a hard STOP.
- **Exact `~/.claude` legacy source** → never migrate to the nested managed path in this ticket. Use `<legacyPath>` as `WT_ABS`; unexpected managed registration/occupancy for the same basename is a conflict and STOPs.
- **Neither path registered** → a leftover at the non-preferred path is ambiguous and STOPs. Otherwise set `WT_ABS=<preferredPath>` and apply the create-target leftover handling below.

**Worktree not registered — reclaim only a truly empty create target, then create it.** A path can exist on disk while absent from Git registration. Guard the create without treating `.claude` as a disposable binding marker — active binding state no longer lives in the checkout:
- If `[ -e "$WT_ABS" ]`, inspect `find "$WT_ABS" -mindepth 1 -maxdepth 1` with **no `.claude` exclusion**.
  - **Disposable** — genuinely empty only: remove the empty directory, then create below.
  - **Any content, including `.claude/`** → it may hold work, nested managed worktrees, or local settings. **STOP-and-offer**: surface the unregistered leftover and offer (1) remove + recreate only after explicit confirmation that it is disposable; (2) abort. Never auto-delete. In `/go` this remains the existing confirm gate (`p` = remove-and-recreate default). (A dangling symlink remains the pre-existing `[ -e ]` edge.)
- Prepare the selected target's parent through the sanctioned helper: `node ~/.claude/bin/ticket-worktree.mjs prepare-parent --path "$WT_ABS"`. Then `git -C "$sourceRoot" fetch origin <baseBranch>` → `git -C "$sourceRoot" worktree add "$WT_ABS" origin/<baseBranch>`. Always quote. This preserves the existing configured base-branch semantics; the Linear branch is still created verbatim in §3, and no direct Bash write into `.claude/` is needed.
- **Switch the session into the worktree:** `EnterWorktree(path: "$WT_ABS")`. **Required before any worktree-local writes in a background session**; safe in foreground sessions too. If already in this worktree, `EnterWorktree` errors with "already in a worktree" — call `ExitWorktree(action: "keep")` first, then re-enter.
- **Write the binding immediately through the helper**, before envrc/dependency bootstrap, so no protected checkout marker is created and the orphan window stays minimal:
  - Feature: `node ~/.claude/bin/ticket-worktree.mjs write-binding --worktree "$WT_ABS" --json '{"linearProject":"<id>","planSlug":"<slug if known>"}'`.
  - Standalone: `node ~/.claude/bin/ticket-worktree.mjs write-binding --worktree "$WT_ABS" --json '{"mode":"standalone","linearIssue":"<ISSUE-ID>"}'`.
  - Milestone: `node ~/.claude/bin/ticket-worktree.mjs write-binding --worktree "$WT_ABS" --json '{"mode":"milestone","linearProject":"<project-id>","linearMilestone":"<milestone-id>"}'`.
  Parse/validation/conflict failures hard-STOP. Never read or write the legacy checkout marker directly.
- **Symlink `.envrc` from the source checkout** via `bash ~/.claude/bin/ensure-envrc.sh "$WT_ABS"`. Idempotent — silent no-op if `$WT_ABS/.envrc` already exists, if `$sourceRoot/.envrc` doesn't exist, or outside a git tree. The SessionStart hook covers sessions opened directly in an existing worktree; this explicit call covers creation mid-session.
- **Bootstrap deps** in `$WT_ABS` (`npm ci` if `package-lock.json`, else `npm install`, or the repo's documented equivalent). Without this a bare `npx <tool>` may install an unrelated package and false-green the required check.
- **Scoped per-worktree edit permissions are bootstrapped automatically** by the `SessionStart` hook in `~/.claude/settings.json` (script: `~/.claude/bin/bootstrap-worktree-perms.sh`). Do not write `<wt>/.claude/settings.local.json` from this skill — the hook is the only writer.

**Worktree exists — branch by mode.** First switch the session into `WT_ABS`: `EnterWorktree(path: "$WT_ABS")` (if already there, `ExitWorktree(action: "keep")` then re-enter). Then migrate/read binding state only through the helper:
- Run `node ~/.claude/bin/ticket-worktree.mjs migrate-binding --worktree "$WT_ABS"` before normal presence/read. This safely imports an old checkout marker; malformed/conflicting state hard-STOPs.
- Run `node ~/.claude/bin/ticket-worktree.mjs binding-status --worktree "$WT_ABS"`, then `node ~/.claude/bin/ticket-worktree.mjs read-binding --worktree "$WT_ABS"` only when status is `private`. No direct checkout-marker access.
- **Orphan check first.** Classify the registered worktree with `bash ~/.claude/bin/orphan-detect.sh "$WT_ABS" <baseBranch>` (the script uses the same helper-backed binding contract). Branch on its one-word verdict:
  - `has-binding` → the normal case; fall through to the mode bullets below using the helper-returned binding JSON.
  - `orphan-recoverable` → the recognizable **crashed-mid-setup orphan**: binding-less, clean, parked on a non-base ticket branch. The deterministic path proves ownership. **OFFER recovery — do NOT bare-STOP**:
    1. **Reconstruct binding** (recommended default) — run the matching `node ~/.claude/bin/ticket-worktree.mjs write-binding --worktree "$WT_ABS" --json '<payload>'` command from the create branch, then continue via the matching-binding bullet below. No work is lost; this only completes the interrupted private metadata write.
    2. **Clean + recreate** — `git -C "$sourceRoot" worktree remove --force "$WT_ABS"`, then recreate exactly as above. Use only when the partial worktree is suspect.
    3. **Abort** — keep the STOP and surface.
    In `/go` this remains the existing confirm gate (`p` = reconstruct-binding default).
  - `dirty-stop` / `foreign-stop` → keep the existing hard STOP (a dirty tree may hold real work; a base/detached/foreign branch is ambiguous).
- **Feature mode, binding `linearProject` matches the target** → **reuse it.** Verify `git -C "$WT_ABS" status --porcelain` is empty; dirty → STOP and surface. Do NOT infer "ticket in flight" from the current branch; step 2's Linear-state guard is authoritative.
- **Feature mode, binding mismatch** (different project, or `mode: "standalone"`) → STOP and surface — ask the user.
- **Standalone mode, binding `linearIssue` matches the target** → STOP and redirect to `/resume-ticket` (an existing per-ticket worktree means prior cleanup did not finish).
- **Standalone mode, binding mismatch or missing** → STOP and surface; ask before touching.
- **Milestone mode, binding `linearMilestone` matches the target's milestone** → **reuse it.** Verify clean; dirty → STOP and surface. The per-milestone Linear-state guard in step 2 is authoritative.
- **Milestone mode, binding mismatch** (different milestone/project, or standalone/feature binding) → STOP and surface — ask the user.

All subsequent git ops use `git -C "$WT_ABS"` regardless of cwd — `cd` doesn't persist across Claude bash calls anyway. The user-facing `cd` hint is at hand-off (step 5).

## 2. Pick the ticket
- **Standalone mode** (binding has `mode: "standalone"`): the ticket IS `binding.linearIssue`. Verify it's not Done/Canceled (one `get_issue`, no list). **No project-wide In Progress block** — multiple standalone tickets can be In Progress simultaneously, each in its own per-ticket worktree.
- **Milestone mode** (binding has `mode: "milestone"` + `linearMilestone`). Bound milestone = `binding.linearMilestone`; never widen beyond it.
  - **Per-milestone In-Progress guard**: `list_issues` `state: "In Progress"`, `assignee: "me"`, `project: <bound>`. `list_issues` has **no milestone filter**, so **paginate** (`hasNextPage`/`cursor`) and **filter client-side** on `projectMilestone.id === binding.linearMilestone` (the same client-side milestone filter `/land-ticket`'s last-in-milestone detection runs — `land-ticket.md` §7). Any In-Progress ticket in **this** milestone → STOP, suggest `/resume-ticket` (or `/resume-ticket <ID>`). The guard is **milestone-width** — a different milestone running In Progress in its own worktree never blocks (project-width would re-impose feature-mode sequentiality and defeat this mode's parallelism; per-ticket-width is standalone mode, not this).
  - **Establishing run** (entered via §1.B's milestone picker, or a specific ISSUE-ID) → the ticket is already chosen; verify it isn't Done/Canceled and that its `projectMilestone.id` matches `binding.linearMilestone`.
  - **Reuse run** (entered via §1.A cwd-binding — the milestone worktree carried over from the previous ticket) → **auto-pick within the milestone**: `list_issues` `state: "Todo"`, `project: <bound>`, `label: <scopeLabel>` (paginate; client-side-filter to `projectMilestone.id === binding.linearMilestone`). Rank as feature mode does (exclude `blockedBy`, priority Urgent→Low, then oldest); top = the pick. Empty → fall through to `state: "Backlog"`, same filter + ranking. None ready → the milestone may be complete (suggest `/land-ticket` on its last ticket — that tears the milestone worktree down); STOP.
- **Feature mode** (binding has `linearProject`). Never widen beyond the bound project.
  - **In-Progress guard**: `list_issues` `state: "In Progress"`, `assignee: "me"`, `project: <bound>`, `limit: 1`. Hit → STOP, suggest `/resume-ticket` (or `/resume-ticket <ID>`).
  - `$ARGUMENTS` was a specific ISSUE-ID in the bound project: take it (one `get_issue` for acceptance + branch name). Skip the preview gate below — user picked explicitly.
  - Auto-pick: `list_issues` `state: "Todo"`, `project: <bound>`, `label: <scopeLabel>`, `limit: 5`. Rank in-prompt: exclude `blockedBy`, priority Urgent→Low (0 lowest), then oldest. Top = the pick. Empty → fall through to `state: "Backlog"`, `limit: 5`, same ranking.
  - None ready → project may be complete (suggest finishing the last `/land-ticket`); else report blockers. STOP.
- **Preview gate (auto-pick only).** One `get_issue` on the picked ticket — needed downstream anyway (step 5's acceptance summary uses the full description, which `list_issues` truncates). Print one line: `Would pick <ID>: <title> — branch <gitBranchName> — acceptance: N items`. **STOP and wait for explicit go** before step 3-4 fire branch ops + Linear state changes. The user can name a different ID here to override the auto-pick.

## 3. Branch in the project worktree
- For a reused worktree, cleanliness was already checked in 1.C. Additionally: `git -C "$WT_ABS" branch --list "<gitBranchName>"` — if a local branch with the new ticket's name already exists, that's leftover state. STOP and ask the user; don't recreate or overwrite.
- **Re-evaluate the In-Progress guard immediately before checkout — TOCTOU close (V-229).** The §2 guard runs at *pick* time and the In-Progress stamp is at §4, with this checkout in between: a window in which a concurrent `/go` on a milestone-less ticket passes its own §2 guard and gets stamped into the same milestone before either run sees the other — the shared milestone worktree's HEAD then switches under one run, caught (in the original race) only downstream by `/scope` §2 branch-drift. Re-run the guard here against the ticket's **current** milestone (one `get_issue` — a mid-flight re-milestoning since §2 is the trigger, so re-read it rather than trusting the §2 value):
  - **Milestone mode**, or a ticket re-milestoned into a milestone since §2: re-run the §2 per-milestone guard (`list_issues` `state: "In Progress"`, `assignee: "me"`, `project: <bound>`; paginate; client-side-filter `projectMilestone.id === <current milestone>`). Another In-Progress ticket now in this milestone → STOP before checkout.
  - **Feature mode**: re-run the §2 project-wide guard (`list_issues` `state: "In Progress"`, `assignee: "me"`, `project: <bound>`, `limit: 1`). A hit absent at §2 → STOP before checkout.
  - **Standalone mode**: per-ticket worktree, no project/milestone block — nothing to re-check (the §2 Done/Canceled check stands).
  - **Observable outcome (Acceptance).** A trip STOPs with a message that names the race, so it is no longer silently absorbed by `/scope` branch-drift: `In-Progress guard re-check tripped before checkout: <other-ID> is In Progress in milestone <M> (concurrent /go race) — resume/land/cancel the other run, then re-run`. Under `/go` this is a **hard stop** (two runs genuinely collided), logged `forced` in the gate-audit. No checkout and no §4 In-Progress stamp on a trip.
  - **Worked example — the encoded proof of the negative (this test-less repo).** Two `/go` runs, A and B, on milestone-less tickets that both get re-milestoned into M3 mid-run. A's §2 guard sees no In-Progress ticket in M3 → passes; B's §2 guard likewise → passes (neither has stamped yet). *Before this fix* both proceed to §3 checkout + §4 stamp and collide — the shared M3 worktree's HEAD switches under A (the original race). *With the re-check:* whichever reaches §3 second re-reads its **current** milestone (now M3) and re-runs the guard, which now finds the first run In Progress in M3 → it STOPs before checkout with the named race message and never stamps. So the double-stamp the race produced **cannot occur undetected** — the second run is halted and surfaced, not silently collided.
- `git -C "$WT_ABS" fetch origin <baseBranch>`; the issue's Linear `gitBranchName` (verbatim → auto-link); `git -C "$WT_ABS" switch -c <gitBranchName> origin/<baseBranch>`.

## 4. Mark In Progress
- Stamp **immediately after §3's re-check** (no gate between) so the re-checked guard result is still fresh when the mutation lands — this is what keeps the TOCTOU window closed (V-229).
- `mcp__linear save_issue`: state → In Progress, assignee `me`. `save_comment`: branch + worktree path.

## 5. Hand off — STOP
- **Print a `## Goal` paragraph.** State CONCRETELY what we are building — not a paraphrase of the title. 2–4 sentences naming the behavior-level outcome and the rough shape of the change (e.g. "Adjust 3 assertions in `auth-flow.test.ts` so they match the post-login redirect and the toast copy specified by the linked tickets. No production code touched."). Both you and the user must finish step 5 with the same mental model of the deliverable. **If you cannot write a concrete Goal from the ticket description, STOP and ask the user before continuing** — a fuzzy goal here breeds a fuzzy build.
- **Print the ticket's `## Acceptance` checklist verbatim.**
- **Do NOT pre-open files the ticket references** — that triggers Claude Code's on-demand load of subdirectory `CLAUDE.md`s (ancestor CLAUDE.mds load at launch, subfolder ones load when files in those subfolders are read) and pollutes context with per-subsystem orientation you may not need yet. The build session opens them when implementing. **Don't render a file-list in the hand-off either** — you'll find them when implementing.
- **All subsequent bash in the build session uses `$WT_ABS` explicitly** (`git -C "$WT_ABS" ...`, absolute file paths). Claude's `cd` from Bash doesn't persist, so without explicit paths you silently run against the wrong tree. Do NOT ask the user to `cd` — they iterate via Claude.
- **Do NOT implement here.** Build is a separate phase: `/scope` validates the ticket against current code + writes a per-ticket plan, then `/build` runs the implementation autonomously (or iterate interactively). All paths feed into `/land-ticket`.
- **Chain-entry router — emit the route, record why.** Resolve the ticket's **route** — exactly one of **`build | fast | fix-bug | research`** — by this precedence (first match wins):
  1. **`route:` issue label** (Initiative I, stamped by `/spawn-tickets`; already on the `get_issue` in hand — no extra call): `route:build` → `build`; `route:fix-bug` → `fix-bug`; `route:research` → `research`.
  2. else **bugs bucket**: `ticket.projectId === cfg.bugBucket.id` → `fix-bug` (the bugs-bucket detection).
  3. else **small-rubric pass** — the "Decide `skip`" criteria below; the rubric doubles as the fast-lane test → `fast`.
  4. else → `build`.

  What each route runs:
  - **`build`** → the scope-necessity decision below fires exactly as before — `scope` or `skip` per the rubric, no regression (a `route:build`-labeled small ticket still earns `skip`).
  - **`fast`** → the **recorded delta over the profile-lean chain** (convention 12): `/scope` skip is guaranteed (the scope decision IS `skip`, recorded, never re-derived); `/build` §5 skips the `/verify-tests` invocation when the diff turns out docs/test-only; `/land-ticket` §4.5's review is guaranteed **Tier A**. The **diff-driven** gates are route-invariant and run regardless of this prediction — `/land-ticket` §4.8's migration/`invariant` hard-gate and §4.6's security floor fire off the actual diff, so a fast ticket that unexpectedly produces a migration still hits the §4.8 STOP.
  - **`fix-bug`** → the repro-first path: run **`/fix-bug`** (origin-convo-first via `/ingest-convo`, reproduction-as-oracle with 3 outcomes, repro-gated fix) **in place of** `/scope`+`/build`, ending in the full unchanged `/land-ticket`. The scope-necessity decision below does **not** apply — `/fix-bug` drops the scope plan and the `/build §3.5` thesis-check by design (`commands/fix-bug.md`; rationale in `docs/fix-bug-path-design.md`).
  - **`research`** → the knowledge-deliverable path: **investigate freeform inline** (run the investigation against the ticket's Acceptance directly — *not* `/research`, which is external-prior-art recon and the wrong tool for an internal analysis), land a **finding doc** in `docs/` (the deliverable is knowledge, per `craft/planning.md`'s route taxonomy), then the full unchanged `/land-ticket`. Skip `/scope`, the `/build §3.5` design thesis-check, `/build`'s implement phase, and `/verify-tests` — there is no code to design, build, or verify. The scope-necessity decision below does **not** apply (like `fix-bug`). `/go` drives this path **inline** (no separate command — it is the routed replacement for Phases 3–5); run the chain by hand, the investigation is freeform and the finding lands via `/land-ticket`. Design rationale: `docs/research-route-path-design.md`.

  **Record it twice (consumers read, never re-decide):** print the route in this hand-off, and append it to the step-4 Linear comment with the exact greppable shape `Route: <build|fast|fix-bug|research> — <one-line rationale>` (durable, ticket-scoped, survives worktree teardown). Downstream reads: `/go` captures the route from the hand-off; a cross-session `/build`/`/land-ticket` reads the comment's `Route:` line (one scoped `list_comments`); a ticket with no `Route:` record (pre-router) defaults to `build`. `/go` reads, never re-decides (`go.md` hard rule).

  *Worked examples (precedence + route-invariance, the encoded proof for this test-less repo):* a ticket carrying `route:research` → `research` (label wins; the knowledge-deliverable path, scope/thesis-check/build/verify all skipped); a bugs-bucket ticket carrying `route:build` → `build` (label beats bucket); an unlabeled bugs-bucket ticket → `fix-bug` (bucket beats rubric); an unlabeled 8-item cross-subsystem ticket → `build` (rubric fails); an unlabeled 3-item docs-only ticket → `fast` (rubric passes) — and if that fast ticket's PR later turns out to contain a `supabase/migrations/*` file, `/land-ticket` §4.8 still hard-STOPs it: the gate reads the diff, and the route never disarms it.

- **Scope-necessity decision (within route `build`) — decide `/scope` or skip, record why.** Evaluate how much `/scope` this ticket actually needs against the rubric below and emit **one** decision — `scope` or `skip` — with a one-line rationale, recorded in the hand-off and on the step-4 comment (`Scope-necessity decision: <scope|skip> — <rationale>`). On route `fast` this decision is fixed at `skip` by the router — record it, don't re-derive it. `/build` consumes a `/scope` plan when one exists and runs its inline path when one doesn't — it does **not** re-make this call, so the decision lives here. This rubric is the single source of the scope-vs-skip criteria (and, via its `skip` arm, of the router's small rubric); `/build`, `/scope`, and the router reference it rather than re-deriving.
  - **Decide `scope`** when ANY trips: Acceptance has > 5 items, or items are interdependent / ambiguous; external-service semantics (Supabase / DB / auth / network / migration) or a sensitive-path keyword — `\b(auth|authn|authz|middleware|migration|crypto|acl|rbac|security|csrf|cors|oauth|saml|jwt|session|cookie|password|secret|token)\b`; a migration or generated artifact; an unverified premise — referenced code / path whose existence is uncertain, **or a rationale that rests on a claimed current-system property, especially a security baseline ("X is already denied", "this is the only writer") — a security-baseline premise IS an unverified premise and routes to `/scope`'s empirical premise-check (§3)** — or no concrete file targets derivable from the ticket; ticket-provided code snippets needing compile-validation against the real import graph; large or cross-subsystem blast radius.
  - **Decide `skip`** when ALL hold: ≤ ~5 well-specified Acceptance items; no sensitive-path / external-service / migration signal; affected files are identifiable and few; no unverified premise — the change shape is obvious from the ticket. A skip is **not** a downgrade: `/scope <ID>` stays available, and if `/build` later trips a defect outside the ticket's scope it triages-and-spins-out per `/build` §4.7 (convention 9).
- **Derive the `depth-class` — one graded size signal, folded into the same rubric (V-323).** From the *same* signals the scope/skip decision above already read — no second pass, no parallel calibrator — emit `depth-class ∈ {light, standard, deep}`:
  - `skip` / `fast` arm ⇒ **light** (all skip-conditions hold; the change shape is obvious).
  - `scope` arm ⇒ **deep** when any *hard* trigger is present (large / cross-subsystem blast radius, interdependent / ambiguous items, a sensitive-path / external-service / migration signal, or an unverified / security-baseline premise), else **standard**.

  **Record it twice on the same rails as `Route:`** — print `Depth-class: <light|standard|deep> — <one-line rationale>` in the hand-off, and append it to the step-4 Linear comment (greppable, ticket-scoped, survives worktree teardown). Consumers read it exactly as they read `Route:` — `/go` captures it from the hand-off; a cross-session `/scope` / `/build` / `/thesis-check` reads the comment's `Depth-class:` line (one scoped `list_comments`). **No `Depth-class:` record ⇒ default `standard`** (byte-for-byte today's behavior — an additive signal, never a new control-flow branch on the default path).

  depth-class is a **raise-only conditioner** on the profile knobs: `deep` guarantees the fuller `scope-plan-depth` (full build-plan schema) and `design-check` (the adversarial `/thesis-check` subagent) **above** the active profile's floor; `light` permits the leaner setting. It can only **raise** rigor above that floor — it never lowers a hard / irreversible gate or the fresh-context-review core (`pipeline/profiles/README.md` model-independent core). It is model-**independent** (a ticket's size is the same across models), which is why its canonical home is this rubric, not the profile knob table.
  - *Worked examples:* a one-line typo-fix (1 well-specified item, one obvious file, no signals) ⇒ **light**; V-323 itself (4 interdependent items across 5 command files, cross-subsystem blast radius) ⇒ **deep**; an unlabeled mid-complexity ticket carrying no `Depth-class:` record ⇒ **standard**, routed exactly as today.
- **End — next step (branch on route + decision):**
  - Route **`fast`** → print exactly `/build` → `/land-ticket` (guaranteed `/scope` skip; `/build` runs its inline path — `/scope <ID>` remains available if a written plan is later wanted).
  - Route **`fix-bug`** → print exactly `/fix-bug` → `/land-ticket` (`/fix-bug` runs the repro-first path in place of `/scope`+`/build`; no scope-necessity decision applies).
  - Route **`research`** → print exactly: investigate inline (freeform — *not* `/research`) → write a finding doc in `docs/` → `/land-ticket` (no `/scope`/thesis-check/`/build`/`/verify-tests`; under `/go` this is driven inline as the Phase 3–5 replacement — `docs/research-route-path-design.md`). No scope-necessity decision applies.
  - Route **`build`**, decision **`scope`** → print exactly `/scope` (recommended — validates Acceptance + writes the build plan) → `/build` → `/land-ticket`.
  - Route **`build`**, decision **`skip`** → print exactly `/build` → `/land-ticket` (no `/scope` plan; `/build` runs its inline path and does not demand one — `/scope <ID>` remains available if a written plan is later wanted).
  - In all cases: after landing, re-run `/next-ticket` *in this same worktree* for the project's next ticket; the worktree persists until the project's last ticket.

## 6. Origin-transcript recovery — read the source before acting on a suspected-wrong premise (any bucket)

A spec the build flow **inherits** (`/scope` §3, `/build` §4/§5 apply it), not a step `/next-ticket` itself runs — its home is here because `/next-ticket` is what establishes the ticket-working context it keys on, for **every** bucket. The problem: a session can read a ticket whose framing is *wrong in either direction* — the premise looks **made-up** (a misdescribed harvested bug wrongly discarded), **or** the premise reads **true but the scope/context is off** (the ticket lost the originating conversation's nuance). The rule is **bucket-agnostic**: whenever the session suspects the ticket's scope/premise is wrong, recover and read the origin conversation before acting.

**When it fires (all must hold):**
- The session has already run **normal premise/scope verification** — the `verify-harvested-bug-premise` disproof for a bug (`git log -S` / `git grep` for the cited artifact, a code read, a repro attempt), or the equivalent scope-validation for any ticket — and come away with a **live suspicion that the ticket's scope or premise is wrong**, in **either** direction: it looks **made-up / stale / non-reproducible**, *or* it reads true but the **framing/scope/context feels off** (the session is unsure it's about to build the right thing). Any bucket — bugs, standalone, or feature; the `cfg.bugBucket.id` gate is **gone**.
- That suspicion would otherwise lead to **acting on a possibly-wrong understanding** — discarding / cancelling / force-halting the ticket, finalizing a `needs-eyes` verdict (in `/scope`), or building from the ticket's prose as-written despite the doubt.

**What it does — one last step before acting on the suspicion:** recover and read the **origin transcript** via whatever handle the ticket exposes — `/ingest-convo <conversation-or-session-id>` — and **re-decide from the actual source** (build it correctly described, close citing the transcript, or re-scope with the recovered nuance). Resolve the handle from, in order:
- an embedded **`conversations:` / `sessions:` line** on the ticket (harvested bugs tickets carry this always — `harvest-pipeline-bugs.md` §5); else
- a **`/capture` `Source:`** value that points at a conversation; else
- a **stated provenance pointer** the ticket carries to a log that records the origin — e.g. a *"filed via `/report-bug`, see `errors.jsonl`, `activeCommand: report-bug`"* note: the matching `errors.jsonl` / `feedback.jsonl` entry carries both `session` and `conversation` (the resume-chain root) per the logger, so the origin is recoverable from **provenance** even when the ticket body has no inline handle.

**Explicitly last-resort, not routine.** It fires only on a *suspected-wrong* premise/scope after normal verification has already raised the doubt — never on a normal ticket the session is confident about. Cost is ~0 on the common case (one extra `/ingest-convo`, only when a session was about to act on a doubted understanding); broadening the bucket must **not** turn this into a per-ticket origin read — the gate is the active post-verification suspicion, not the ticket's mere existence. This is the build-time counterpart to the harvest-time locality check, and the thing that consumes the harvested origin handle.

*Example: a non-bugs-bucket ticket whose premise verifies TRUE but whose scope reads off fires the rule; its stated provenance (`errors.jsonl` entry → `session`/`conversation`) resolves the transcript and recovers decisive context the bare ticket had lost.*

**Degrade safely — never block, loop, or error:**
- **No handle on the ticket** (no embedded line, no `Source:`, no stated provenance pointer — nothing resolvable) → nothing to open; fall back to the prior decision **and say so** (`no recoverable origin handle on this ticket — proceeding with the conclusion from premise/scope verification`).
- **Handle present but transcript unrecoverable** (GC'd / `/ingest-convo` resolves nothing) → same: fall back and say so (`origin transcript <id> unrecoverable (GC'd) — proceeding with the conclusion from premise/scope verification`).
- Never retry-loop on a missing transcript and never error out — the recovery is best-effort; its absence simply leaves the pre-existing decision standing.
