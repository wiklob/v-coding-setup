# Craft A/B harness — blind before/after test of the craft hypothesis (V-125)

> Decision doc. The harness this describes is the field-test the project's research (`plans/skill-craft-layer-research.md`, open Q1) had to exclude — measuring whether the craft layer actually shifts a command's *behavior*, not just its instructions. It owns the project's top-level Validation criterion (`plans/skill-craft-layer.md` → "Behavioral (blind)").

## What it measures

The hypothesis: wiring a command to the craft register changes how it works, observably — its output, run after the change, carries craft markers its pre-craft output did not. The criterion: the **same fixed input** run through a command's craft-equipped form and its **genuine pre-craft** form yields **≥2 of 3 craft markers present-after / absent-before**.

The three markers are a **sharpened, command-shape-aware operationalization** of `craft/judgment.md` §"The stance" and `craft/building.md` — *not a verbatim copy* (V-158, revised V-186). §"The stance" states the behavioral *target* the craft layer aims to produce; this rubric is the *measurement* of it. **V-186 revised the instrument** after runs 1–5 exposed three defects (`docs/craft-ab-result.md` Aggregate + the V-160 Reconcile): marker (c) scored **0/5** across 3 commands, the markers were `/review-pr`-prose-shaped so they **under-detected craft in structured `/scope`/`/build` output** (run 4 scored `/scope` 0/3 *despite* the craft wiring being present in `scope.md`), and marker (b) **confounded** with a capable base model's own observed-over-asserted reasoning (run 1 / run 3). The revision keeps the rubric **3 markers wide** — so the project's ≥2/3 criterion phrasing is untouched — and makes each marker **command-shape-aware** (scored against the evidence a command's *output form* actually emits), with a sharpened (b) separator and a re-targeted (c). The divergence from §"The stance" is deliberate and documented here so a reader doesn't mistake it for drift:

**Marker (c) — retire→re-target decision (V-186, Acceptance item 1).** The V-158 definition of (c) — "**visibly resists** a checklist" (names the step-ticking instinct and declines it) — scored **0/5 clean across all 5 runs / 3 commands** (`docs/craft-ab-result.md` Aggregate, marker (c) row: `⊘` at run 1 then `❌` at runs 2–5). It isolates nothing as defined: it is not a behavior `/review-pr`, `/scope`, or `/build` produces (each runs structured sections; none stages a *visible* checklist-refusal), and the diagnostic reasoning it aimed at is base-rate high — present in *both* forms at run 1. It is **re-targeted, not retired**: retiring it to a 2-marker rubric would force the ≥2/3 criterion to become ≥2/2 (strictly harder) or be restated — a change to the *project criterion*, not the instrument. Instead (c) is re-aimed at a real craft behavior that *can* surface in structured output — **reaching past the literal letter to the goal** (`craft/building.md`'s goal-over-letter stance), plausibly absent in a letter-satisfying pre-craft form since the `a96db7f^` baseline predates `craft/building.md`. See the rubric below.

The three markers, scored against the command's **output shape** (Acceptance item 2 — so a structured-output command isn't measured against `/review-pr`-tuned prose evidence):

| marker | prose-form evidence (`/review-pr`) | structured-form evidence (`/scope`, `/build`) |
|---|---|---|
| **(a) names + resists a default instinct** | the first-person "the pull is to X; I resist by Y" move in the narrative | an explicit *named-instinct-resisted* statement anywhere in the rationale / Risks / design fields (e.g. "the instinct is to trust the ticket's prose; I probed the premise instead") — **not** required to be first-person prose |
| **(b) cites a named craft rail** *(sharpened separator, V-186)* | cites a `craft/` file or a named Constraint/Anti-Pattern *by name* in the review reasoning | a by-name `craft/` register reference in the validation/build reasoning — a Risks/premises line, a §4.5 acceptance reconciliation (run 5's clean (b)) |
| **(c) reaches past the literal letter to the goal** *(re-targeted, V-186)* | flags where the literal diff/ask would underserve the goal and reviews to the goal, not just the letter | flags in the plan/validation where a literal Acceptance item would ship a hollow version of the goal and builds/validates to the goal instead (`craft/building.md`) |

**Marker (b) — sharpened separator (V-186, Acceptance item 3).** Score (b) `y` **only** on a *by-name reference to the craft register* — a `craft/` file, or a named Constraint/Anti-Pattern from it (e.g. *"the accepted-premise anti-pattern"*, *"judgment.md ## Constraints"*, *"every claim of done cites an observation"*). A base-model **observed-over-asserted self-check that names no craft rail scores `n`**, even when it paraphrases a rail's *concept* — this **reverses** V-158's "a genuine craft self-critique that paraphrases a rail still scores." Rationale: the run 1 / run 3 confound was precisely that a capable base model independently produces observed-over-asserted reasoning (`"green is asserted, not observed"`) that a rater can read as a rail paraphrase; only a *by-name register citation* is unambiguously craft-attributable, because the pre-craft form has **no register to name**. This is a deliberate **precision-over-recall** trade — a genuine craft critique that paraphrases without naming now loses the point — justified because the confound *is* base-model paraphrase, so precision is exactly what the separator was missing. (Neither checking the command's own procedural contract nor citing a `workflow-conventions.md` convention scores (b) — a base model does both unaided; this is unchanged from V-158.)

**Validity caveat on (b) — it is a weak behavioral signal by construction.** Because the pre-craft baseline (`a96db7f^`) predates the craft register, the before-form has *no register to name*, so (b) can essentially only ever score `y` on the after-form — it detects "this command was wired to read `craft/`" nearly as much as "the command *behaves* differently." That is a near-deterministic consequence of the wiring, not a hard-won behavioral delta, and it sits in tension with the harness's "measures behavior, not text" claim. So (b) alone should never be read as confirming the criterion. The genuinely-behavioral confirmation comes from **(a)/(c)** — markers the base model *can* produce unaided (and does: (a) on both `/scope` forms, (c) on both `/build` forms) but that the craft form produces where the pre-craft form doesn't. A run's ≥2/3 is only trustworthy when it includes **at least one clean non-(b) marker** — which runs 6 (clean (c)) and 7 (clean (a)) both do. Read (b) as the *floor* the wiring guarantees; read the clean non-(b) marker as the *evidence*.

It measures **behavior, not text** — we don't diff the two `.md` files and read markers off the instructions (that would assert a change, not observe one — convention 8). Each form is *actually executed* by a subagent on a fixed input, and the markers are scored on the produced output.

## The two halves

**Deterministic mechanics — `bin/craft-ab-harness.mjs`.** Given `--command <name> --baseline <ref>`, it:
1. Extracts both forms — before via `git show <baseline>:commands/<cmd>.md`, after from `HEAD`.
2. **Guards the baseline.** The before form must be *genuinely pre-craft*: the script greps the extracted before-form for craft markers and refuses (exit 1) if any are present. This catches the confounded-baseline trap — for `/review-pr`, `a96db7f^` still carries the V-117 stance paragraph, so only `f35ad76^` (the parent of the *first* craft commit on that command) is valid. The genuine pre-craft baseline is **per-command**:

   | command | first craft commit | valid baseline |
   |---|---|---|
   | `/review-pr` | `f35ad76` (V-117 header) | `f35ad76^` |
   | `/scope` | `a96db7f` (V-119/P3) | `a96db7f^` |
   | `/build` | `a96db7f` (V-119/P3) | `a96db7f^` |

3. **Blind-labels** the forms — randomly (or `--seed N` for reproducibility) maps before/after to forms A/B, writes `tmp/<cmd>-form-A.md` / `-form-B.md` and the unblinding key `tmp/<cmd>-ab-map.json`. The scorer sees only A and B.
4. Scaffolds the results-doc skeleton to stdout.

**Agent-driven run + score (executed by `/build`, or a human re-running the harness).** A bin script can't run a command-as-agent and can't score behavior, so these legs are subagent-driven:
1. **Pin a fixture** — one input identical across both forms. For `/review-pr` that is a single merged PR number (it resolves a PR via `gh pr view/diff <n>`, so the same `<n>` gives byte-identical input to both forms).
2. **Run each form** — dispatch one subagent per form, each told to execute the command's instructions (`tmp/<cmd>-form-A.md` / `-form-B.md`) literally on the fixture, and to return its produced output verbatim. The before-form subagent is given the pre-craft instructions; neither is told which is which. **Capture each form's output verbatim** to `tmp/<cmd>-output-A.md` / `-output-B.md` — this is the exact text the scorers receive.
3. **Blind-score with ≥2 independent raters, fed verbatim (V-158).** Dispatch **at least two** *neutral* blind scorer subagents, **independently** (each in its own context — no rater sees another's scores). Each is given the **two captured outputs verbatim** (`tmp/<cmd>-output-A.md` / `-output-B.md`, untouched — *no orchestrator summarization or condensation of the outputs*), the 3-marker rubric, and **nothing else** (not the craft register, no before/after hint). Each returns y/n + a verbatim-quoted evidence span per marker per form. *Why both rules are load-bearing:* run 1 used a single rater on a *condensed* scorer hand-off (`docs/craft-ab-result.md` limitation #2) — a summarized input lets the orchestrator's reading leak into the score, and one rater gives no agreement signal. Passing the raw outputs to ≥2 independent raters removes both.
4. **Unblind + record, with inter-rater agreement.** Read `tmp/<cmd>-ab-map.json`, map A/B back to before/after, and for **each rater** compute markers present-after-and-absent-before and the ≥2/3 verdict. Record **per-rater scores** and the **inter-rater agreement** — percent agreement over a **stated denominator** (the 6 marker×form cells, i.e. 3 markers × {form A, form B}; state it explicitly so a reader can recompute). Cohen's κ is *not* used — at n=3 binary markers it is unstable and misleading; percent agreement plus the raw per-rater table is the honest statistic at this N. The criterion is met only when **≥2/3 holds across the raters** (note any rater split rather than averaging it away). Write all of this to `docs/craft-ab-result.md`.

## Discipline (why the design is shaped this way)

- **Observed over asserted (convention 8) — load-bearing.** Outputs must be real subagent runs captured verbatim. A narrated or fabricated result is the worst failure class here: this ticket owns the project's *top-level* validation, so a fake "criterion met" would certify the whole project on a lie.
- **Blind integrity.** The scorer can't know before from after, or it will find what it expects. Hence the randomized A/B labels and a scorer prompt with no temporal hint and no craft register.
- **No self-reference.** The craft layer grading its own markers skews toward finding them. The primary scorer is a neutral subagent given only the rubric. `/review-skill` is **not** a valid behavioral scorer — it grades a command *file*, not run output — so it serves at most as a secondary cross-check on the instructions, never as the primary rater.
- **≥2 independent blind raters, required (V-158).** Three markers from *one* scorer is both a small sample and a single subjective reading. Every run is scored by **at least two independent blind rater subagents** (§"The two halves" step 3), each fed the two outputs **verbatim**, and the **inter-rater agreement is recorded** (percent agreement over the 6 marker×form cells). This supersedes the original "≥2 where cheap, else single-rater limitation" allowance: a single-rater run is no longer a valid run of this harness — run 1's single condensed rater is exactly the instrument limit this fixes. The criterion (≥2/3 present-after/absent-before) must hold **across** the raters, not on an average that hides a split.

## Recording the result — including a negative one

`docs/craft-ab-result.md` records: the forms + baseline, the pinned fixture, the verbatim outputs (or `tmp/` references), the per-marker scores, the unblinded verdict, and the **feedback action**. A negative or partial result is recorded too, never dropped (Acceptance item 3):

- A single negative A/B result → the **documented-limitation** route, not a `Status` flip. `craft/governance.md`'s honest-scope convention covers this: one A/B run is below the N≥3 evidence floor that flips a craft file's `Status`, so a negative result is logged as a limitation of *this run* and a prompt to re-run with more raters / another command — it does not by itself retire `craft/judgment.md`.
- A sustained negative signal (a craft file repeatedly failing to produce its markers across runs) is what `craft/governance.md`'s trigger (c) routes to a `/review-skill` review of the rail, which can flip the file to `retired` on evidence.

## Re-running

```
node bin/craft-ab-harness.mjs --command review-pr --baseline f35ad76^ --seed 7
# then run each tmp/review-pr-form-*.md on a pinned PR via subagents (capture each output verbatim to
# tmp/review-pr-output-*.md), blind-score with >=2 independent raters fed those verbatim outputs,
# unblind, compute per-rater scores + inter-rater agreement, append to docs/craft-ab-result.md
```

To test another command, pass its row's baseline from the table above; the guard rejects a confounded one.

## Fixtures & captured output for non-`/review-pr` commands (V-159)

`/review-pr`'s fixture is trivially pinnable — a PR number gives byte-identical input to both forms via `gh`, and the scored output is the review text. `/scope` and `/build` are *stateful/mutating* commands whose input is the ambient active ticket (current branch + worktree) and whose output isn't a clean read-only artifact, so they need an explicit protocol (the missing-seam V-159's thesis-check flagged). The rule, so a future runner doesn't re-derive it:

- **Pin one concrete real ticket as the fixture.** Capture its `get_issue` payload + a fixed base commit once, and hand **both** form-runner subagents the *same* payload + base ref, instructing each to treat it as the active ticket. This restores the identical-input invariant `/review-pr` gets for free. Read affected files at the base ref (`git show <base>:<path>`) — the worktree HEAD may already contain the ticket's change.
- **Bypass the mechanical preflight, deliberately.** `/scope`/`/build` preflight (In-Progress state, branch match, clean tree) is execution-gating, not where the craft markers live (the markers are in the validation/build *reasoning*). Handing the form the ticket directly skips the ambient resolution and the In-Progress STOP a Done fixture would trip, without altering the scored reasoning. Record the bypass per run (observed, not asserted).
- **`/scope` scored output = the produced validation report + build plan** (read-only). Write each form's plan to a *disjoint scratch path* (`tmp/scope-plan-{A,B}.md`), never the fixture's canonical `docs/plans/<id>-build.md`, so the forms don't collide.
- **`/build` scored output = the form's *actually-executed* reasoning trace through §3–§4.5** (the phases carrying the craft self-critique rail). Run each form for real (genuine reads + a real produced edit) **pre-commit** — write the produced result to a `tmp/` scratch file (or an isolated scratch worktree), **no commit/push/Linear**, and capture the reasoning verbatim. It must be an executed run, **not** a hand-narration of what the command *would* do — a narration reintroduces the assert-not-observe failure (convention 8) the harness exists to exclude. To bound cost, nested delegation (the §3.5 `/thesis-check`, §5 `/verify-tests`) may be performed inline rather than dispatched; note it.
- **Caveat (V-159 finding) — resolved by the V-186 command-shape-aware rubric.** The original three markers were authored against `/review-pr` narrative prose and **under-detected** craft in structured-output commands (run 4 scored `/scope` 0/3 despite the craft wiring being present in `scope.md`). V-186 fixed this: the §"What it measures" rubric is now **command-shape-aware** — each marker carries a distinct structured-form evidence column, so a `/scope`/`/build` run is scored on evidence its plan/table output actually emits (a named-instinct-resisted line, a by-name craft-rail citation in Risks/§4.5, a goal-over-letter flag), not on the first-person prose move only `/review-pr` produces. Runs 6/7 (`docs/craft-ab-result.md`) are the re-run against this revised instrument.
