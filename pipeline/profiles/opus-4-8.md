# Profile: Opus 4.8 (and Opus 4.x)

Source basis: Anthropic's Opus 4.8 migration guide + this pipeline's own Opus-era findings (the 2026-06-04 observations: chained commands lose per-step diligence, in-session reviews miss what fresh contexts catch, plans read instrumental). Posture: **strong builder that under-reaches for its own capabilities and under-questions its own designs** — so this profile adds back triggering rails and keeps the adversarial design rung. This is also the **conservative default profile** when the model family is unknown.

## Knobs

- **`design-check`** — always fire. When the plan lacks a `## Thesis-check` block, invoke `/thesis-check` (fresh-context adversarial subagent) before any code is written. Opus-era evidence: with the human out of the loop, the chain executed wrong theses without questioning them; the rung pays for itself here.
- **`scope-plan-depth`** — full. Write the complete build-plan schema (`scope.md` §6) including `## Implementation design`, `## Implementation steps`, and `## Verification strategy`. A low-context later session (or `/resume-ticket`) executes better from a rich plan than from a thin one.
- **`research-depth`** — recommend-then-gate. Assess the work (point count, net-new vs pattern-following, cross-cutting coupling), recommend **full** or **small** with a one-line rationale citing those signals, and stop at a convention-11 gate (`needs input:`); execute the human's chosen depth. At full depth, dispatch a parallel agent per plan-point (prescribed — this generation under-reaches for fan-out on its own) and produce the `/plan` §3 outputs: `file:line` findings, a spike per genuine unknown.
- **`re-grounding`** — re-verify before building. Planning-time findings are context, not truth: re-check load-bearing findings against the current code/command files before implementing (the V-137 lesson). Chained runs especially: execute each chained command with the diligence of a fresh invocation.
- **`read-discipline`** — read narrow, not whole. `Read` is the pipeline's #1 context cost (`/scorecard --aggregate` 2026-06-19: 20.4MB of tool_result re-read across 171 sessions; CB-284's build burned 438.7k Read tokens — 44% of its context — re-reading whole component/CSS/lib files). Prefer a targeted `offset`/`limit` range over a whole-file pull when you already know the section you need; route bulk or exploratory reads through a subagent digest (the finding comes back, not the file dump); re-read only the section that changed, never the whole file to re-confirm one part. This model over-pulls whole files by default — name that pull and resist it. (This curbs *redundant* reads; it never licenses skipping a read you genuinely need — `re-grounding` still governs re-verifying load-bearing findings.) **Mechanically backed, not advice alone (V-267):** a PostToolUse `Read` hook (`bin/nudge-read-discipline.mjs`) injects a just-in-time, model-visible reminder whenever a large file is read whole (no `offset`/`limit`), and `bin/read-footprint.mjs` rolls each build's Read-token footprint up from `usage-stats.mjs` and flags a regression against the recent-build baseline — so the discipline bites at the read and is measured per build, not left to prose.
- **`autonomy`** —
  - Put the full task specification up front; don't drip-feed yourself scope mid-run.
  - For minor choices (naming, formatting, defaults, equivalent approaches), pick a reasonable option and note it rather than asking — keep asking for scope changes and destructive actions. (This model over-asks by default.)
  - **Reach for your capabilities explicitly**: when a task fans out across independent items, delegate to subagents rather than iterating serially; check the memory/lessons layer before long tasks and write findings back; when the answer depends on information not in context, search before answering from memory.
  - Default to silence between tool calls — one sentence when you find something load-bearing, change direction, or hit a blocker. No routine narration.
- **`review`** — fresh-context verifier subagents (core rule), **plus** the report-everything shape: report every finding including uncertain/low-severity ones with confidence + severity attached; filtering happens downstream, never at the finding stage (this model follows severity filters so literally that measured recall drops).

## Reading the commands under this profile

Follow command bodies as written — their rails were calibrated against this model generation's behavior and earn their keep here. Where a command and this profile conflict, the command's explicit gate/evidence rule wins; the profile fills judgment gaps, it doesn't relax procedure.
